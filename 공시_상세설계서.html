에이전트 공시(Disclosure) 분석

SEC 공시(8-K·10-Q/K·Form 4)를 Form 종류마다 코드와 LLM 비중을 다르게 쓰는 라우팅 엔진으로 구조화 신호로 바꾼다. 실제 매수 확정은 별도 Strategist 몫.

SEC EDGAR (키 불필요) 코드=사실 · LLM=해석 이벤트 온톨로지 11종 매매권한 6단계 출력 DisclosureBundle (0~1)

01 왜 공시인가 — "도장 찍힌 사실"

공시는 주가를 직접 움직이는 가장 신뢰도 높은 정보다. 뉴스는 추측·과장이 섞이지만, 공시는 법적으로 의무 제출하고 거짓이면 처벌받는 "도장 찍힌 사실"이라 호재/악재 판단의 기준점이 된다.

🍳 쉽게 말하면 — 공시 = 기업이 정부(SEC)에 내는 공식 신고서. "우리 이런 일 있었습니다"를 법적으로 보고하는 것. 거짓 보고하면 처벌받으니 소문·뉴스와 신뢰도 차원이 다르다. SEC=미국 증권 감독기관, EDGAR=그 공시를 누구나 무료로 보는 공개 시스템.
핵심 원칙 — 코드=사실, LLM=해석만 — 숫자·표·정형 데이터는 코드가 확정적으로 뽑고, LLM은 뉘앙스 해석만 맡는다. 텍스트에서 숫자를 뽑게 하면 환각(hallucination) 위험이 커지므로 금지. 이 원칙이 공시 파이프라인 전체를 관통한다.

02 수집 — SEC EDGAR

워치리스트 종목의 새 공시가 뜨면 가져온다(이벤트 트리거). 흐름은 ticker → CIK → submissions → 문서/Item 청킹.

1) 티커 → CIK — SEC company_tickers.json으로 티커를 기업 고유번호(CIK)에 매핑.
2) submissions — CIK로 최근 제출 목록을 받아 Form 종류·accession(공시 고유 ID)·제출일(filed_at)을 확인.
3) 문서 수집·청킹 — 8-K는 Item 단위로 잘라내고, 10-Q/K는 XBRL 수치 API를 쓴다.
⚡ 메타레벨 증분 (비용의 핵심) — 스케줄러는 peek_recent_filings로 submissions만 1회 보고 form별 최신 accession을 확인한다. 이미 처리한 accession이면 문서 다운로드·LLM·저장을 전부 스킵. "변화가 있을 때만" 무거운 일을 한다.
🍳 쉽게 말하면 — 매번 서류를 다 내려받아 비교하는 게 아니라, "새 서류 번호가 떴나?"만 먼저 훑고 새 것일 때만 펼쳐 읽는다.

03 Form별 코드/LLM 비중 차등 라우팅

핵심은 "공시 = LLM 1개"가 아니라, Form 종류마다 코드와 LLM 비중을 다르게 쓰는 라우팅 엔진이라는 점이다.

Form내용코드가 하는 일LLM이 하는 일
8-K
수시공시		 인수·실적·임원변경·상폐 등 돌발 사건 코드 Item 번호로 기본 event_type·중요도·권한 부여, 위험 Item 하드룰 LLM 해당 Item 본문만 청킹해 뉘앙스 해석
10-Q / 10-K
실적보고		 분기 / 연간 재무제표 코드 XBRL API로 매출·EPS 확정수치 직접 수집 + YoY 변화율 계산 LLM 가이던스·뉘앙스만. 숫자 추출 금지(환각)
Form 4
내부자거래		 임원·이사 매수/매도 신고 코드 100% XML 파싱 → 직급·매수/매도·수량 규칙 스코어링 — (LLM 미사용)
🍳 쉽게 말하면 — 같은 "공시"라도 서류 종류에 따라 손이 다르다. 숫자표(실적·내부자)는 코드가 정확히 읽고, 사람 말로 쓰인 사건(8-K 본문)만 LLM이 해석한다.

04 8-K Item 라우터 — 위험 Item은 하드룰 우선

8-K는 Item 번호에 따라 의미가 완전히 달라진다. 본문을 LLM에 넘기기 전에 SEC 메타데이터의 구조화된 Item 번호로 기본 성격을 먼저 부여한다(정규식은 fallback).

Item의미기본 권한하드룰
1.01중요 계약 체결WATCH_ONLY—
2.02실적 발표WATCH_ONLY—
2.06자산 손상차손WATCH_ONLY✔ 표시(권한 미반영)
3.01상장폐지 통보BLOCK_BUY✔ 위험
4.01감사인 변경BLOCK_BUY✔ 위험
5.02임원·이사 변경WATCH_ONLY—
🛡️ 3.01 · 4.01은 기본 권한 자체가 BLOCK_BUY라 LLM이 "중립"이라 판단해도 매수를 막는다("지능이 아니라 안전장치"). 2.06은 위험 Item으로 hard_risk 표시되지만 기본 권한은 WATCH_ONLY — 실제 차단은 본문 하드리스크 키워드(§4-3)가 잡을 때만.
⚠️ 알려진 갭 — item_event_map.yaml의 hard_risk:true 플래그는 route_item이 반환하지만 decide_final_permission이 아직 소비하지 않는다(권한은 default_permission만 후보). 그래서 2.06은 현재 자동 차단되지 않음 — 플래그를 권한에 반영할지는 팀 결정 사항(Phase 2).

05 10-Q / 10-K — XBRL 수치 경로 + 본문 리스크 스캔

수치는 코드 — companyconcept XBRL API로 매출·순이익 등 확정 수치를 직접 받아 전년동기(YoY) 변화율을 코드로 계산한다. LLM에게 숫자를 세게 하지 않는다.
해석만 LLM — 계산된 수치 요약을 주고 "좋아졌나/나빠졌나, 가이던스 뉘앙스"만 해석하게 한다.
⚠️ 본문 리스크 문구는 별도 스캔 — XBRL 경로는 수치 요약만 보므로 going concern(계속기업 불확실성)·material weakness 같은 문구를 놓칠 수 있다. 그래서 원문을 별도로 내려받아 하드리스크 룰(07장)로 다시 스캔한다.
💡 백테스트 기준일(day0) = filed_at — 8-K/Form4는 submissions의 filingDate, 10-Q/K는 companyconcept의 filed를 저장해 수익률 측정 기준일을 정확히 잡는다. (컨센서스 대비 Surprise는 데이터가 유료라 보류)

06 Form 4 — 100% 코드 스코어링

정형 XML 표 데이터라 LLM이 비효율·불필요. form4_scorer.py가 규칙으로 점수화한다.

거래 유형해석가중
공개시장 매수자신감 신호 호재직급(CEO/CFO 등 senior) 가중 ↑
공개시장 매도약한 악재—
베스팅·세금·옵션행사·10b5-1 예정매도중립 (일상적)—
🍳 쉽게 말하면 — "CEO가 자기 돈으로 자사주를 샀다" = 자신감. "예정된 세금용 매도" = 그냥 일상. 이 구분을 코드가 규칙으로 점수 매기고, 높은 직급일수록 무게를 더 준다.

07 하드리스크 키워드 룰

Form과 무관하게 본문을 스캔해 위험 키워드가 있으면 매수를 강제 차단한다. 첫 매치를 즉시 반환한다.

# hard_risk_policy — 위험 키워드 → 권한 강등
going concern        → BLOCK_BUY    # 계속기업 불확실성
auditor resignation  → BLOCK_BUY    # 감사인 사임
material weakness     → BLOCK_BUY    # 내부통제 중대 결함
secondary offering   → BLOCK_BUY    # 유상증자(희석)
SEC investigation    → BLOCK_BUY    # SEC 조사
🛡️ 하드룰은 LLM보다 항상 우선. LLM이 "중립"이라 해도 코드 키워드 규칙이 즉시 매수를 막는다.

08 DisclosureSignal — LLM 출력 스키마 (추론 강제)

LLM은 자유 텍스트가 아니라 검증된 구조화 객체만 낸다. 결론 전에 reasoning(CoT)을 먼저 서술하게 하고, certainty_level로 모호성을 스스로 분류하게 해 신뢰도를 방향과 분리한다.

class DisclosureSignal:
    reasoning: str            # [CoT] 결론 전 단계적 추론 강제
    certainty_level: str      # High / Medium / Low (모호성)
    event_type: str           # 통일 11종 온톨로지
    sentiment: str            # positive / negative / neutral / mixed
    importance: int           # 0~10 (거래정지=10, 대형 M&A=8~9, 일상=1~2)
    risk_score: float         # 0~10 하방 위험
    hard_risk_flag: bool      # 보통 코드 하드룰이 설정
    trade_permission: str     # 1차 제안(보수적) — 최종은 코드가 재결정
    reason: str               # 한 줄 근거(수치 포함 권장)
    evidence_quotes: list     # 원문 인용(환각 방지)
    summary: str              # ★ 공시 전문 3~4줄 요약
    keywords: list            # ★ 핵심 키워드 5개
    verdict: str              # ★ 검증 결과 한 문장(맨 앞 호재/악재/중립 명시)
💡 요약·키워드·판정 (신규) — 구조화 점수와 별개로, 사람이 바로 읽는 전문 3~4줄 요약·핵심 키워드 5개·호재/악재 판정 한 문장을 함께 낸다. LLM이 원문 근거로만 채우며(환각 방지), 하류(Strategist·리뷰·대시보드)가 근거를 빠르게 파악하게 한다.
💡 event_type 온톨로지 11종 (공시·뉴스 공유, ontology.py 단일 정의): earnings · guidance_change · ma · capital_raise · management_change · insider_trade · product_deal · analyst_rating · regulation_legal · delisting_halt · other. 과세분화 금지(표본 부족).

09 매매권한 종합 — 6단계, 가장 보수적 채택

마지막에 공통 decision 레이어를 통과한다. 하드룰·LLM 신호·Item 기본값 중 가장 제한적인 권한을 최종값으로 뽑는다.

권한의미제한도
BLOCK_ALL매수·보유 모두 위험6 (최고)
BLOCK_BUY신규 매수 차단5
RISK_REDUCE보유 축소 검토4
WATCH_ONLY관찰만, 매수 금지3
TRADE_ELIGIBLE_SMALL_SIZE소액 진입만2
TRADE_ELIGIBLE매수 후보1 (최저)
⚠️ "공시 떴으니 바로 산다" 금지 — 시장반응 미확인 시 매수계열(TRADE_ELIGIBLE*)은 WATCH_ONLY로 강등된다. 이 파트는 매수 신호를 스스로 확정하지 않는다. 실제 매수 확정은 Strategist 몫.

10 산출물 — DisclosureBundle (Strategist 입력)

공시 최종 산출물. 뉴스와 완전 별개 객체로 넘어가고(합치기는 Strategist가), 모든 점수는 0~1 소수점으로 통일한다. 방향은 sentiment_score(0=악재·0.5=중립·1=호재)로 점수화 + sentiment 라벨 병기.

class DisclosureBundle:
    ticker; company_name; category    # 종목 식별(company_name=SEC 자동, category=스크리닝 전달)
    as_of; created_at; has_signal      # ★ created_at=레코드 생성시각(초, DB생성일)
    filing_title; filing_no; filed_at  # ★ 공시 제목·번호(accession)·업로드일시(초 포함)
    event_type                        # 통일 11종
    sentiment; sentiment_score: 0~1   # 방향 라벨 + 점수(0=악재·0.5=중립·1=호재)
    importance: 0~1                   # 강도(방향 뺀 별도 축)
    risk_score: 0~1
    confidence: 0~1                   # certainty_level 수치화
    confirmed_score: 0~1              # 공시=1.0 (법적 의무 제출=확정)
    fact_check                        # ★ 팩트체크 한 문장(코드 판정)
    hard_block: 0/1 + hard_block_reason   # 상폐·going concern → 매수 override
    verdict; summary; keywords         # ★ 호재/악재 판정·전문요약·키워드5
    reason; ref                       # 한 줄 근거 · accession/링크
    def to_prompt() -> str            # LLM이 읽는 압축 텍스트
💡 원문 식별 (신규) — 판단 근거를 추적·인용하도록 filing_title(공시 제목)·filing_no(공시 번호=accession)·filed_at(업로드 일시, 초까지 — SEC acceptanceDateTime 우선, 없으면 제출일)을 함께 전달한다.
🍳 sentiment_score 산출 & 근거 — 0.5 + 0.5 × (부호 × importance)로 0~1에 매핑하되, 호재(+)는 confidence로 중립쪽 감쇠·악재(−)는 유지하는 비대칭 규칙(뉴스와 동일). 공시는 대개 certainty=High(0.9)라 거의 안 눌리지만, 모호(Low)한 호재 공시는 방향을 덜 단정한다. importance(순수 강도)와 방향/강도 축을 분리 유지.
💡 hard_block은 sentiment에 안 녹이고 별도 플래그. 상폐·going concern 같은 안전장치는 이 단계에서 0/1로 못 박아 뒤 단계가 뒤집지 못하게 한다. 정식 전달은 JSON(model_dump_json()), to_prompt()는 보조.

10-1. to_prompt() 실출력 예시

[NVDA · NVIDIA CORP · Semiconductors] Disclosure as_of 2026-07-03
 Disclosure: ma / positive(0.86) imp0.8 risk0.3 conf0.9 confirmed1.0
       Filing: "NVDA 8-K — Item 1.01 Entry into Material Agreement" · No 0001045810-26-000123 · filed 2026-07-03T16:30:12.000Z
       Fact-check: Verified — official SEC filing (legally binding fact).
       Verdict: Bullish — large all-cash acquisition, near-term upside likely (high trust).
       Summary: Acquirer to buy Target for 35% cash premium. Deal size $10B. Regulatory approval pending.
       Keywords: acquisition, 35% premium, all-cash, regulatory approval, large M&A
       Reason: "Large acquisition announced (30% premium)"
 Overall: hard_block=0 · generated 2026-07-03T16:31:05+00:00
💡 출력 언어 = 영어 — 자유서술 필드(summary·verdict·reason·evidence_quotes)와 to_prompt() 라벨은 전부 영어다. 전략가(Strategist) 에이전트가 영어로 읽기 때문(사람 검수용 아님). enum·점수 필드는 공통.

11 역할 경계 — 하는 것 / 안 하는 것

✔ 이 파트가 한다

  • SEC EDGAR 실수집(키 불필요, User-Agent만) + 메타레벨 증분
  • Form별 코드/LLM 차등 구조화 분석(8-K·10-Q/K·Form4)
  • 하드리스크·Item 안전 필터 + 보수적 1차 매매권한(6단계)
  • DisclosureBundle 산출·전달 + DB 저장 + 백테스트 수익률(측정만)

✘ 이 파트가 안 한다 (Strategist·이후 몫)

  • 실제 매수 확정 — 매수계열을 WATCH로 강등만
  • 시장반응 검증(가격·거래량) · 비중·수량·EV 계산
  • 가중치 자동 튜닝 — 과적합 방지로 측정·리포트까지만
🍳 한 줄 요약 — 공시 파트는 "도장 찍힌 사실을, 검증된 구조로, 보수적으로 요약해서" Strategist에게 넘긴다. 판단의 지능은 넘기되, 안전장치(하드블록)는 이 단계에서 확정해 못 박는다.
