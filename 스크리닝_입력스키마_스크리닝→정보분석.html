입력 스키마 명세 v1
스크리닝(스케일러) → 정보분석(공시·뉴스)

스크리닝 에이전트가 넘길 50개 종목 리스트의 형식·전달 방식. 정보분석 서비스 코드를 고치지 않고 그대로 돌릴 수 있게 우리 내부 소비 지점에 1:1로 맞춘 계약이다.

스크리닝(스케일러) → 정보분석(공시·뉴스) → Strategist
작성 정창욱 대상 스크리닝(스케일러) 담당 형식 JSON 배열 필수 필드 ticker 하나
TL;DR 1. 우리가 쓰는 값 2. 입력 스키마 3. 규약 4. 전달·실행 5. 열린 질문

핵심 4가지

① JSON 배열 하나50개 종목을 배열로. 각 원소가 종목 1개. 봉투(as_of·count)는 선택.
② 필수는 ticker뿐회사명·CIK는 우리가 SEC에서 자동으로 채운다. 티커만 있으면 공시·뉴스를 스스로 수집.
③ 필드명 = 우리 내부명매핑·변환 코드 없이 upsert_company()에 바로 흘려 유니버스 적재 → 기존 엔트리 그대로 실행.
④ 점수·센티먼트 금지그건 우리가 만든다. 스크리닝은 선별 + 분류(category)까지만.
스크리닝이 주는 것

종목 티커 · 순위(rank) · 분류(category) — 이미 아는 것만

우리가 알아서 하는 것

회사명 · CIK 매핑 · 공시/뉴스 수집 · 분석·점수화

1 우리가 실제로 쓰는 값

정보분석 파이프라인은 전부 ticker 기준으로 동작한다. 티커만 있으면 공시(SEC)·뉴스(RSS)를 스스로 수집한다.

우리 코드가 받는 것어디서 쓰나없으면?
ticker공시/뉴스 수집 키 · companies PK필수 — 없으면 아무것도 못 함
category번들에 그대로 통과 (우리에게 다른 소스 없음)결과에 카테고리가 빔
company_name표시용SEC에서 자동
cikSEC 조회 키티커로 자동 매핑
sectorcompanies.sector(선택)비워둠
rankcompanies.priority = 처리 순서기본값(순위 없음)
설계 원칙
스크리닝이 이미 아는 것(순위·카테고리)만 주면 되고, 우리가 알아낼 수 있는 것(회사명·CIK)은 안 줘도 된다.

2 입력 스키마 (권장 형식)

2-1. 봉투(envelope) + 종목 배열

{
  "schema": "quantinue.screening_input.v1",
  "as_of": "2026-07-03",
  "count": 50,
  "companies": [
    { "ticker": "NVDA", "company_name": "NVIDIA CORP", "category": "반도체", "rank": 1 },
    { "ticker": "AAPL", "company_name": "Apple Inc.",  "category": "빅테크", "rank": 2 },
    { "ticker": "LLY",  "company_name": "Eli Lilly",   "category": "제약/바이오", "rank": 3 }
  ]
}
봉투 없이 배열만 줘도 OK
[ {"ticker":"NVDA", ...}, ... ] 형태도 받는다. 봉투는 as_of·count로 검증·감사에 편해서 권장.

2-2. 종목 원소 필드

필드타입필수우리 쪽 매핑설명
tickerstr필수수집 키 · companies.ticker미국 티커, 대문자. 예 NVDA. 클래스주는 SEC 표기(BRK-B)
categorystr권장번들 category 통과종목 분류(섹터/테마). 우리가 못 만드는 값 → 줄 수 있으면 꼭
rankint권장companies.priority스크리닝 순위(1=최상위). 우리 처리 순서가 됨. 없으면 배열 순서
company_namestr선택companies.name없으면 SEC에서 자동
cikstr선택companies.cik10자리(0001045810) 또는 정수. 없으면 자동 매핑
sectorstr선택companies.sectorGICS 섹터 등 별도 저장(선택)
note/reasonstr선택저장만(로직 미사용)스크리닝 근거. 감사·디버깅용 보관

2-3. 보내지 말아야 할 것

분석 결과성 값은 금지 — 우리가 만드는 값
sentiment · sentiment_score · importance · risk_score · event_type · confidence · hard_block · score 등. 보내도 무시되며, 혼선 방지를 위해 아예 넣지 말 것. 스크리닝의 역할은 선별 + 분류까지다.

3 규약

인코딩UTF-8. 한글 카테고리 그대로 OK
티커 정규화대문자, 공백·따옴표 없이. 같은 티커 중복 금지(마지막 것으로 upsert)
개수50개 전후 자유. count를 주면 배열 길이와 대조 검증
rank 유일성가능하면 1..N 유니크. 겹치면 우선순위 동률(티커 알파벳순 tie-break)
category 라벨값 체계는 합의 대상(§5). 무엇이든 문자열 하나면 우리는 통과시킴

4 전달 방식 — 우리가 그대로 돌리는 경로

전달 = 파일 하나

screening_input.json (UTF-8, §2 형식). 전용 로더 run_screening_input.py 가 받는다:

  1. 각 원소를 upsert_company()로 적재 후 priority=rank·active=1 확정 (cik/name 없으면 SEC 자동, --exclusive 면 목록 외 비활성)
  2. category는 파일→메모리→번들로 그대로 통과(companies 스키마 변경 불필요)
  3. --run 이후 또는 별도로 기존 엔트리를 코드 수정 없이 실행

그대로 실행

# 로더 — 적재(+이 50개만 active)
python run_screening_input.py \
  --file screening_input.json --exclusive

# 로더 — 적재 + 번들 생성(category 통과)
python run_screening_input.py \
  --file screening_input.json --run --llm --json

# 유니버스 순회(공시+뉴스, rank 순)
python run_scheduler.py --once --limit 50 \
  --forms 8-K,10-Q --news --llm --save

스크리닝 rank → companies.priority 이므로 --limit N이 스크리닝 순위 순 상위 N개를 돈다.

검증됨
견본 screening_input.example.json로 적재(priority=rank 정확)·--exclusive working-set·--run 번들 생성(category "반도체" 양쪽 통과)까지 실측 확인. 기존 서비스 코드는 무수정.

5 스크리닝 담당과 합의할 것

  1. category 값 체계 — GICS 11섹터 영문? 자체 한글 테마 라벨? (우리는 문자열이면 통과하지만, 백테스트 집계·Strategist 결합엔 일관 라벨이 좋음)
  2. rank 의미 — 시총 순위? 스크리닝 점수 순위? 우리는 순서로만 쓴다
  3. 갱신 주기 — 50개 리스트를 얼마나 자주 새로 주는가(매일? 장중?). 우리 스케줄러 주기와 맞춤
  4. CIK 제공 여부 — 주면 SEC 매핑 1스텝 절약. 안 줘도 됨
