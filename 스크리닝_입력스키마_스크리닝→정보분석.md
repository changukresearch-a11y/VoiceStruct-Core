# 입력 스키마 명세 — 스크리닝(스케일러) → 정보분석(공시·뉴스)

> 작성: 정보분석 담당(정창욱) · 대상: 스크리닝(스케일러) 에이전트 담당
> 목적: **정보분석 서비스 코드를 고치지 않고 그대로 돌릴 수 있는** 입력 형식·전달 방식 정의
> 방향: 스크리닝 → 정보분석 (이 문서). 반대 방향(정보분석 → Strategist)은 `데이터스키마_인터페이스명세_v2.md`.

---

## ⚡ TL;DR

1. **JSON 배열 하나**로 50개 종목을 넘겨준다. 각 원소는 종목 1개.
2. **필수는 `ticker` 하나뿐.** 나머지는 우리가 SEC/시장데이터로 자동으로 채우거나(회사명·CIK), 그대로 통과시킨다(카테고리).
3. **필드명을 우리 내부명과 똑같이** 맞춰 준다 → 매핑·변환 코드 없이 `upsert_company()`에 바로 흘려 유니버스에 적재하고 기존 엔트리(`run_scheduler` / `run_bundle` / `run_universe --run`)를 그대로 돌린다.
4. **점수·센티먼트·이벤트유형은 보내지 말 것.** 그건 우리(정보분석)가 만든다. 스크리닝은 "어떤 종목을 볼지 + 그 종목의 분류(category)"까지만.

---

## 1. 우리가 실제로 쓰는 값 (왜 이 스키마인가)

정보분석 파이프라인은 **전부 `ticker` 기준으로 동작**한다. 종목 티커만 있으면 공시(SEC)·뉴스(RSS)를 스스로 수집한다.

| 우리 코드가 받는 것 | 어디서 쓰나 | 없으면? |
|---|---|---|
| `ticker` | 공시/뉴스 수집 키, `companies` 테이블 PK | **필수** — 이게 없으면 아무것도 못 함 |
| `category` | 번들(`DisclosureBundle`/`NewsBundle`)에 **그대로 통과** | 우리에게 다른 소스가 없음 → 결과에 카테고리가 비게 됨 |
| `company_name` | 표시용. 없으면 SEC `company_tickers.json`에서 자동 | 자동으로 채움(문제 없음) |
| `cik` | SEC 조회 키. 없으면 티커로 자동 매핑 | 자동으로 매핑(문제 없음) |
| `sector` | `companies.sector` 컬럼(선택) | 비워둠 |
| `rank` | `companies.priority` = **처리 순서**(작을수록 먼저) | 100 기본값(순위 없음) |

> 핵심: **스크리닝이 이미 아는 것(순위·카테고리)만 주면 되고, 우리가 알아낼 수 있는 것(회사명·CIK)은 안 줘도 된다.**

## 2. 입력 스키마 (권장 형식)

### 2-1. 봉투(envelope) + 종목 배열

```json
{
  "schema": "quantinue.screening_input.v1",
  "as_of": "2026-07-03",
  "count": 50,
  "companies": [
    { "ticker": "NVDA", "company_name": "NVIDIA CORP", "category": "반도체", "rank": 1 },
    { "ticker": "AAPL", "company_name": "Apple Inc.",  "category": "빅테크", "rank": 2 },
    { "ticker": "LLY",  "company_name": "Eli Lilly",   "category": "제약/바이오", "rank": 3 }
  ]
}
```

> **봉투 없이 배열만** 줘도 된다: `[ {"ticker":"NVDA", ...}, ... ]`. 봉투는 `as_of`·`count`로 검증·감사에 편해서 권장.

### 2-2. 종목 원소 필드

| 필드 | 타입 | 필수 | 우리 쪽 매핑 | 설명 |
|---|---|:---:|---|---|
| `ticker` | `str` | ✅ | 수집 키 · `companies.ticker` | 미국 종목 티커, **대문자**. 예 `NVDA`. 클래스주는 SEC 표기대로(예 `BRK-B`) |
| `category` | `str` | ▲권장 | 번들 `category` 통과 | 종목 분류(섹터/테마). 우리가 못 만드는 값 → **줄 수 있으면 꼭** |
| `rank` | `int` | ▲권장 | `companies.priority` | 스크리닝 순위(1=최상위). 우리 처리 순서가 됨. 없으면 배열 순서 |
| `company_name` | `str` | ○선택 | `companies.name` | 없으면 SEC에서 자동 |
| `cik` | `str` | ○선택 | `companies.cik` | 10자리(예 `0001045810`) 또는 정수. 없으면 티커로 자동 매핑 |
| `sector` | `str` | ○선택 | `companies.sector` | GICS 섹터 등 별도 저장용(선택) |
| `note` / `reason` | `str` | ○선택 | 저장만(우리 로직 미사용) | 스크리닝 근거. 감사·디버깅용으로 보관 |

### 2-3. 보내지 말아야 할 것 (우리가 만드는 값)

`sentiment` · `sentiment_score` · `importance` · `risk_score` · `event_type` · `confidence` · `hard_block` · `score` 등 **분석 결과성 값**. 보내도 **무시**되며, 혼선을 줄이려 아예 넣지 않기를 권장한다. 스크리닝의 역할은 **선별 + 분류**까지다.

## 3. 규약

- **인코딩**: UTF-8. (한글 카테고리 그대로 OK)
- **티커 정규화**: 대문자. 공백·따옴표 없이. 중복 티커는 마지막 것으로 upsert(같은 티커 2번 주지 말 것).
- **개수**: 50개 전후 자유. `count`를 주면 배열 길이와 대조해 검증한다.
- **rank 유일성**: 가능하면 1..N 유니크. 겹치면 그대로 우선순위 동률 처리(티커 알파벳순 tie-break).
- **category 라벨 표준**: 값 체계(GICS 섹터 코드? 자체 한글 분류?)는 합의 대상 → §5. 무엇이든 **문자열 하나**면 우리는 통과시킨다.

## 4. 전달 방식 (우리가 그대로 돌리는 경로)

### 4-1. 파일로 전달

`screening_input.json` (UTF-8) 한 파일. 위 §2 형식. 전용 로더 **`run_screening_input.py`** 가 이걸 받는다:

```bash
# 적재만 — companies 유니버스에 넣고(priority=rank) 스케줄러/유니버스로 돌릴 준비
python run_screening_input.py --file screening_input.json

# 적재 + 이 50개만 작업대상(active)으로 고정 → 스케줄러 working-set = 스크리닝 목록
python run_screening_input.py --file screening_input.json --exclusive

# 적재 + 종목별 Strategist 입력 번들(공시+뉴스) 생성 — category 그대로 통과
python run_screening_input.py --file screening_input.json --run --llm --json --limit 5
```

로더가 하는 일:

1. 각 원소를 `upsert_company()`로 `companies` 테이블에 적재 후 **`priority=rank`·`active=1`** 확정
   (cik/name 없으면 SEC 자동 매핑, rank 없으면 기본값). `--exclusive` 면 목록 외 종목은 비활성.
2. `category`는 파일→메모리→번들로 **그대로 통과**(companies 스키마 변경 불필요)
3. `--run` 이후 또는 별도로 **기존 엔트리포인트를 코드 수정 없이 실행**:

```bash
# (a) 유니버스 순회 — 공시+뉴스 증분 수집·분석 (priority=rank 순)
python run_scheduler.py --once --limit 50 --forms 8-K,10-Q --news --llm --save

# (b) 단일 종목 번들 (수동 확인용)
python run_bundle.py --ticker NVDA --category "반도체" --llm --json
```

> 스크리닝 `rank` → `companies.priority` 이므로 `--limit N` 이 **스크리닝 순위 순** 상위 N개를 돈다.
> 견본 입력: `screening_input.example.json`.

### 4-2. (대안) 인메모리 리스트

같은 스키마의 파이썬 `list[dict]` 객체로 넘겨도 동일하게 적재 가능. 파일이 표준, 인메모리는 같은 프로세스에서 직접 호출할 때 편의용.

## 5. 스크리닝 담당과 합의할 것

1. **`category` 값 체계** — GICS 11개 섹터 영문? 자체 한글 테마 라벨? (우리는 문자열이면 뭐든 통과하지만, 백테스트 집계·Strategist 결합을 위해 **일관된 라벨**이면 좋음)
2. **`rank` 의미** — 시총 순위인지, 스크리닝 점수 순위인지. 우리는 순서로만 쓴다.
3. **갱신 주기** — 50개 리스트를 얼마나 자주 새로 주는지(매일? 장중?). 우리 스케줄러 주기와 맞춤.
4. **CIK 제공 여부** — 주면 SEC 매핑 1스텝 절약. 안 줘도 됨.
