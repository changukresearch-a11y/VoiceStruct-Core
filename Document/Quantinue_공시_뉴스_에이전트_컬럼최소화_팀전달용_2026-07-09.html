컬럼 최소화 및 Strategist 입력값 정리

05 공시 분석과 06 뉴스 분석의 테이블을 분리 유지하면서, 중복·과잉 컬럼을 줄이고 Strategist가 실제로 읽는 압축 신호값을 명확히 정리한 팀원 전달용 문서입니다.

0. 최종 결정 요약

핵심: 공시와 뉴스는 통합하지 않습니다. tb_disclosure와 tb_news를 별도로 유지하되, 점수 체계와 Strategist 입력 구조는 최대한 맞춥니다.
공시공식 이벤트 신호. 1시간마다 50종목을 처리하고, 종목별 1행을 tb_disclosure에 append합니다.
뉴스대표기사 1개 기준이 아니라 기사별 점수 산정 후 종목 단위 가중 집계값을 tb_news에 저장합니다.
점수importance_score, sentiment_score, risk_score, trust_score는 0.00~1.00, 소수점 2자리입니다.
근거대표 기사·대표 공시의 제목, 링크, 시각 등은 개별 컬럼이 아니라 evidence_json에 넣습니다.

1. 공통 점수 규칙

컬럼범위의미
importance_score0.00~1.00전략 판단에 얼마나 중요한 이벤트인가
sentiment_score0.00~1.00악재↔호재 방향성. 0.50이 중립
risk_score0.00~1.00매수 시 위험도가 얼마나 높은가
trust_score0.00~1.00뉴스 출처 신뢰도 집계값. 뉴스 전용
 
score = round(max(0, min(1, score)), 2)
중요: has_signal=false일 때 점수 컬럼에 0.00을 넣지 않습니다. 점수 컬럼은 null로 두고, 카운트는 0, hard_block=false로 둡니다.

sentiment_score 해석

값해석
0.00강한 악재
0.25약한 악재
0.50중립
0.75약한 호재
1.00강한 호재

2. 공시 테이블 최소안 — tb_disclosure

tb_disclosure는 1시간마다 오늘의 50개 종목에 대해 공식 공시 신호를 분석하고, 종목별 최신 공시 상태를 한 행으로 저장합니다.

최종 컬럼 16개

No컬럼타입 예시설명
1disclosure_idUUID/TEXT공시 분석 행 ID
2cycle_idTEXTStrategist 판단 사이클 연결용 ID
3tickerTEXT종목 코드
4collected_atTIMESTAMP공시 수집·분석 시각
5has_signalBOOLEAN분석할 공시가 있는지
6filing_countINT해당 구간 또는 당일 집계된 공시 수
7main_filing_noTEXT대표 공시 번호
8main_filing_typeTEXT8-K, 10-Q, 10-K 등
9main_filed_atTIMESTAMP대표 공시 제출 시각
10event_typeTEXT대표 이벤트 유형
11importance_scoreNUMERIC(3,2)공시 중요도
12sentiment_scoreNUMERIC(3,2)공시 방향성
13risk_scoreNUMERIC(3,2)공시 기반 위험도
14hard_blockBOOLEAN즉시 매수 차단 여부
15summaryTEXTStrategist가 읽을 공시 요약
16evidence_jsonJSONB근거 공시 목록

공시 점수 산정

importance_score =
0.50 * event_severity
+ 0.20 * recency_score
+ 0.20 * market_relevance
+ 0.10 * novelty_score

sentiment_score =
0.70 * event_direction
+ 0.30 * text_direction

risk_score = max(event_risk_score, special_risk_score)

hard_block = true
if event_type in HARD_BLOCK_EVENTS
or risk_score >= 0.90
공시는 공식문서이므로 별도 trust_score 컬럼을 두지 않습니다. Strategist 입력에서 필요하면 내부적으로 1.00으로 간주하면 됩니다.

3. 뉴스 테이블 최소안 — tb_news

tb_news는 5분마다 오늘의 50개 종목에 대한 최근 뉴스를 수집하고, 기사별 점수를 가중 집계하여 종목별 1행으로 저장합니다.

최종 컬럼 17개

No컬럼타입 예시설명
1news_idUUID/TEXT뉴스 집계 행 ID
2cycle_idTEXTStrategist 판단 사이클 연결용 ID
3tickerTEXT종목 코드
4collected_atTIMESTAMP뉴스 수집·분석 시각
5window_startTIMESTAMP뉴스 집계 시작 시각
6window_endTIMESTAMP뉴스 집계 종료 시각
7has_signalBOOLEAN분석할 뉴스가 있는지
8article_countINT집계에 사용된 기사 수
9source_countINT서로 다른 출처 수
10event_typeTEXT대표 이벤트 유형
11importance_scoreNUMERIC(3,2)기사별 중요도 가중 집계값
12sentiment_scoreNUMERIC(3,2)기사별 방향성 가중 집계값
13risk_scoreNUMERIC(3,2)평균 위험도와 최고 위험도 반영
14trust_scoreNUMERIC(3,2)전체 출처 신뢰도 집계
15hard_blockBOOLEAN즉시 매수 차단 여부
16summaryTEXT뉴스 전체 흐름 요약
17evidence_jsonJSONB상위 근거 기사 목록

뉴스 집계 공식

article_weight =
article_trust_score
* article_recency_score
* article_novelty_score
* article_importance_score

sentiment_score =
Σ(article_sentiment_score_i * article_weight_i)
/
Σ(article_weight_i)

importance_score =
0.70 * weighted_avg(article_importance_score)
+ 0.30 * max(article_importance_score)

risk_score =
max(
  weighted_avg(article_risk_score),
  max(article_risk_score)
)

trust_score =
weighted_avg(article_trust_score)

hard_block = true
if any(article_hard_block == true and article_trust_score >= 0.60)
or risk_score >= 0.90
금지: 뉴스 점수는 대표기사 1개로 산정하지 않습니다. 대표 기사는 설명용 근거일 뿐이고, 최종 점수는 기사 묶음의 가중 집계값입니다.

4. Strategist 직접 전달값

Strategist는 DB 전체 컬럼이 아니라 압축된 신호값만 읽습니다.

공시 입력

{
  "disclosure_signal": {
    "has_signal": true,
    "event_type": "earnings",
    "importance_score": 0.82,
    "sentiment_score": 0.76,
    "risk_score": 0.18,
    "hard_block": false,
    "summary": "공식 실적 공시에서 매출과 이익 개선이 확인됨."
  }
}

뉴스 입력

{
  "news_signal": {
    "has_signal": true,
    "article_count": 8,
    "source_count": 5,
    "event_type": "earnings_news",
    "importance_score": 0.71,
    "sentiment_score": 0.68,
    "risk_score": 0.29,
    "trust_score": 0.83,
    "hard_block": false,
    "summary": "실적 전망 상향과 긍정적 애널리스트 코멘트가 다수 확인됨."
  }
}
hard_block=true는 점수보다 우선합니다. 공시나 뉴스에서 차단 신호가 나오면 Strategist의 GPT 판단 전에 즉시 보류 처리할 수 있습니다.

5. SQL 예시

tb_disclosure

CREATE TABLE IF NOT EXISTS tb_disclosure (
    disclosure_id TEXT PRIMARY KEY,
    cycle_id TEXT NOT NULL,
    ticker TEXT NOT NULL,
    collected_at TIMESTAMPTZ NOT NULL,

    has_signal BOOLEAN NOT NULL DEFAULT FALSE,
    filing_count INT NOT NULL DEFAULT 0 CHECK (filing_count >= 0),

    main_filing_no TEXT,
    main_filing_type TEXT,
    main_filed_at TIMESTAMPTZ,
    event_type TEXT,

    importance_score NUMERIC(3,2)
        CHECK (importance_score IS NULL OR importance_score BETWEEN 0.00 AND 1.00),
    sentiment_score NUMERIC(3,2)
        CHECK (sentiment_score IS NULL OR sentiment_score BETWEEN 0.00 AND 1.00),
    risk_score NUMERIC(3,2)
        CHECK (risk_score IS NULL OR risk_score BETWEEN 0.00 AND 1.00),

    hard_block BOOLEAN NOT NULL DEFAULT FALSE,
    summary TEXT,
    evidence_json JSONB
);

CREATE INDEX IF NOT EXISTS idx_tb_disclosure_cycle_ticker
ON tb_disclosure (cycle_id, ticker);

CREATE INDEX IF NOT EXISTS idx_tb_disclosure_ticker_collected
ON tb_disclosure (ticker, collected_at DESC);

tb_news

CREATE TABLE IF NOT EXISTS tb_news (
    news_id TEXT PRIMARY KEY,
    cycle_id TEXT NOT NULL,
    ticker TEXT NOT NULL,
    collected_at TIMESTAMPTZ NOT NULL,

    window_start TIMESTAMPTZ NOT NULL,
    window_end TIMESTAMPTZ NOT NULL,

    has_signal BOOLEAN NOT NULL DEFAULT FALSE,
    article_count INT NOT NULL DEFAULT 0 CHECK (article_count >= 0),
    source_count INT NOT NULL DEFAULT 0 CHECK (source_count >= 0),

    event_type TEXT,

    importance_score NUMERIC(3,2)
        CHECK (importance_score IS NULL OR importance_score BETWEEN 0.00 AND 1.00),
    sentiment_score NUMERIC(3,2)
        CHECK (sentiment_score IS NULL OR sentiment_score BETWEEN 0.00 AND 1.00),
    risk_score NUMERIC(3,2)
        CHECK (risk_score IS NULL OR risk_score BETWEEN 0.00 AND 1.00),
    trust_score NUMERIC(3,2)
        CHECK (trust_score IS NULL OR trust_score BETWEEN 0.00 AND 1.00),

    hard_block BOOLEAN NOT NULL DEFAULT FALSE,
    summary TEXT,
    evidence_json JSONB
);

CREATE INDEX IF NOT EXISTS idx_tb_news_cycle_ticker
ON tb_news (cycle_id, ticker);

CREATE INDEX IF NOT EXISTS idx_tb_news_ticker_collected
ON tb_news (ticker, collected_at DESC);

6. 구현 체크리스트

영역체크 항목
DB공시·뉴스 테이블 분리 유지, 점수는 NUMERIC(3,2), 0~1 체크 제약, evidence_json JSONB
공시1시간마다 50종목 처리, 공시 없는 종목도 1행 생성, hard_block 룰 기반 처리
뉴스5분마다 50종목 처리, 대표기사 1개 평가 금지, 기사별 점수 산정 후 가중 집계
Strategist전체 컬럼이 아니라 압축 신호만 입력, hard_block 우선, trust_score 낮은 뉴스는 약한 참고 신호로 처리
최종 구조: tb_disclosure 16개 컬럼, tb_news 17개 컬럼. 공시·뉴스는 분리하지만, Strategist가 읽는 핵심 신호는 동일한 체계로 정리합니다.
