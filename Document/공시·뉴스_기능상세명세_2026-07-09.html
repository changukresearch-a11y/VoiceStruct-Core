공시·뉴스 기능 상세 명세
저장 스키마 · 전달 계약 · 산출 로직

공시·뉴스 정보분석 파트의 현재 확정 상태를 실제 코드 그대로 정리한 명세. 핵심은 저장(tb_disclosure·tb_news, 상세 컬럼) ≠ 전달(Strategist 압축 신호)의 분리. 2026-07-09 리뷰 반영(단일 _id PK · is_positive · temperature=0 · KST 표시 · disclosure_ref 유지 · 압축 계약)까지 포함.

저장 tb_disclosure 19 · tb_news 26 (+_id) 전달 공시 7 · 뉴스 10 필드 압축 LLM temperature=0 · Pydantic AI

00 개요 — 저장과 전달의 분리

📥 스크리닝 50 티커🗄️ tb_disclosure · tb_news📤 Strategist 압축 신호담당: 정창욱

정보분석 파트는 50개 종목의 공시·뉴스를 분석해 두 층으로 내보낸다.

  • 저장층 — tb_disclosure(19)·tb_news(26)에 상세 컬럼을 시계열 append. 감사·백테스트·디버깅·근거 추적용.
  • 전달층 — Strategist에겐 판단에 필요한 압축 신호(공시 7·뉴스 10 필드)만 JSON으로. 인터페이스 안정성·프롬프트 토큰 절감.
50 티커 │ ├─ 공시(1h) ─▶ 수집·분석·집계 ─▶ tb_disclosure(상세19) ─▶ latest_disclosure_signal() ─▶ 공시 압축 7필드 └─ 뉴스(5m) ─▶ 수집·분석·집계 ─▶ tb_news(상세26) ─▶ latest_news_signal() ─▶ 뉴스 압축 10필드 │ 공시·뉴스는 통합하지 않고 각자 주기로 별개 전달 → 07 Strategist
🛠 스택 — Python · Pydantic · Pydantic AI(구조화 출력 강제, temperature=0 결정론). LLM 코드기본 anthropic:claude-haiku-4-5 / 운영 .env openai:gpt-4o-mini. SEC·RSS·Yahoo 전부 키 불필요. DB=SQLite. 시각은 UTC 저장 + KST 표시.

01 파이프라인 흐름

공시 (1시간)

수집(8-K Item·10-Q/K XBRL·Form4 XML) ─▶ Form/Item 라우팅 ─▶ 하드리스크 스캔 ─▶ LLM 분석(DisclosureSignal 0~10) ─▶ 매매권한 종합(가장 보수적) ─▶ 집계(종목당 1행) ─▶ tb_disclosure append (신호 없어도 has_signal=0 빈 행)

뉴스 (5분)

수집(Google News RSS) ─▶ 출처 3등급·키워드 사전필터(LLM 전 drop) ─▶ LLM 분석(NewsSignal) ─▶ 기사별 신뢰가중 집계 + 대표기사 선정 ─▶ disclosure_ref 매칭(오늘 공시와 교차확인) ─▶ tb_news append
🕐 이중주기 — 틱은 뉴스 5분으로 돌고 공시는 1시간 배수마다. 매 사이클 50종목 각각 1행 append(덮어쓰기 ❌·시계열 누적). Strategist는 종목별 최신 행을 읽는다. 메타레벨 증분(accession 기준)으로 새 공시/기사만 LLM 호출.

02 tb_disclosure — 저장 스키마 (19 + _id)

PK: _id (AUTOINCREMENT)UNIQUE: (ticker, collected_at)점수 0~1
#컬럼타입설명
—_id 🔑INT AUTO단일 기본키(리뷰 반영)NEW
1tickerTEXT종목 심볼 · UNIQUE 키
2collected_atTS스냅샷 생성 시각(UTC) · UNIQUE 키
3trade_dateDATE분석 기준일(FK→tb_daily_pick)
4has_signal0/1공시 있음(1)/없음(0, 뒤 컬럼 null)
5filing_titleTEXT대표 공시 제목
6filing_noTEXTaccession(원문 역추적 ID)
7filed_atTS제출 시각(look-ahead 방지)
8event_typeTEXT(12종)사건 종류(통일 온톨로지)
9sentiment라벨positive/negative/neutral/mixed
10sentiment_score0~10=강악재·0.5=중립·1=강호재
11is_positive0/1sentiment_score>0.5 불리언 파생NEW
12reasonTEXT점수 근거 한 줄
13importance0~1강도(방향 뺀 축)
14risk_score0~1위험도(오늘 공시 최댓값 보존)
15confidence0~1LLM 확신도(certainty 매핑)
16hard_block 🛡️0/1매수 즉시 차단
17hard_block_reasonTEXT차단 사유
18summaryTEXT공시 3~4줄 요약
19keywordsTEXT[]키워드 5개
🧩 종목당 오늘 공시를 1행 집계. 방향·메타는 대표 1건에서, risk_score는 최댓값·hard_block은 OR로 보존(치명 위험이 묻히지 않게). 공시는 공식 문서라 별도 출처신뢰 컬럼 없음.

04 Strategist 압축 신호 — 전달 계약 NEW

저장 상세(19/26)에서 판단에 필요한 핵심만 뽑아 JSON으로 전달. 공시·뉴스는 통합하지 않고 각자(1h/5m 주기) 전달한다. 명칭은 _score 접미로 통일(전달 계약에서만 개명, 내부 컬럼은 유지).

공시 압축 (7필드) — latest_disclosure_signal(ticker)

{
  "has_signal": true,
  "event_type": "earnings",
  "importance_score": 0.82,   // ← importance 개명
  "sentiment_score": 0.76,
  "risk_score": 0.18,
  "hard_block": false,
  "summary": "공식 실적 공시에서 매출과 이익 개선이 확인됨."
}

뉴스 압축 (10필드) — latest_news_signal(ticker)

{
  "has_signal": true,
  "article_count": 8,           // news_count
  "event_type": "earnings",
  "importance_score": 0.71,     // 신뢰가중 평균
  "peak_importance": 0.75,      // ← 추가: 강신호 희석 방지
  "sentiment_score": 0.68,
  "risk_score": 0.29,
  "trust_score": 0.83,         // ← source_trust 개명
  "hard_block": false,
  "summary": "실적 전망 상향과 긍정적 애널리스트 코멘트가 다수 확인됨."
}

매핑 · 규칙

압축 필드← 저장 컬럼규칙
importance_scoreimportance개명(전달 계약만)
trust_score(뉴스)source_trust개명
peak_importance(뉴스)peak_importance강신호 보존용 포함
article_count(뉴스)news_count확산도
summarysummary(+hard_block_reason)hard_block이면 사유를 앞에 삽입
🛡️ hard_block=true면 summary 앞에 [차단: {사유}]가 자동으로 붙는다. 예: "[차단: going concern] 상폐 위험 공시". Strategist가 즉시 보류하면서 왜 막혔는지를 바로 본다.
💤 신호 없음 → {"has_signal": false, "hard_block": false}만 반환(점수 필드는 null 대신 생략). 0.00은 '강한 악재'로 오인될 수 있어 넣지 않는다.

05 점수 산출 · 집계 로직

정규화 · 방향점수

importance/risk = _n10(v) = round(clip(v,0,10)/10, 2) # LLM 0~10 → DB 0~1 confidence = {High:0.9, Medium:0.6, Low:0.3}[certainty] sentiment_score = 0.5 + 0.5 × signed (호재만 confidence로 감쇠, 악재 유지) is_positive = sentiment_score > 0.5

뉴스 신뢰가중 집계

weight = 확신(certainty) × 확정(1.0/0.5) × source_trust # 믿을수록 무겁게 importance = Σ(강도 × weight) / Σweight # 신뢰가중 평균 peak_importance= max(importance, 신뢰검증 기사 강도들) # 강신호 보존 risk_score = Σ(위험 × weight) / Σweight (+ 최댓값 성격 보존) sentiment_score= 강악재(≤−0.7) 1건이면 최솟값, 아니면 가중평균
📐 왜 신뢰가중? 단순 평균이면 저신뢰 펌프 기사 여러 건이 고신뢰 1건을 눌러 왜곡된다. 무게(weight)로 믿을 만한 기사에 힘을 실어 이를 막는다. 상세 산출·예시는 컬럼값_산출근거_데이터계보.html·공시·뉴스_컬럼구성_도식.html 참조.

06 disclosure_ref — 뉴스↔공시 교차확인 (유지 확정)

삭제한 루머/팩트 이진판별 대신 공식 문서 존재 여부로 뉴스 사실성을 보강한다. 뉴스 대표 event_type과 같은 종목 오늘 공시가 매칭되면 그 filing_no, 없으면 null.

disclosure_ref_for(ticker, event_type): if event_type in (None, "other"): return None # 광범위 오매칭 방지 후보 = 오늘 같은 종목 공시 중 event_type 일치 & accession 있음 return 후보 중 importance 최대 1건의 filing_no # 없으면 None
  • 설계: accumulator(오늘 프로세스 메모리) 기반 · event_type 일치만 · tie는 importance 최대.
  • 순수성: build_news_bundle은 순수 함수로 유지하고 매칭은 스케줄러에서 주입(has_signal일 때만).
  • 한계: 재시작 직후엔 오늘 공시 누적이 비어 잠깐 매칭이 빌 수 있음 → 다음 공시 틱 이후 최신 행에 채워짐.
✅ 이 기능은 팀 최최종 명세 #10 신설안을 구현한 것으로, 유지 확정(제거하지 않음). 뉴스 "실적 호재"에 같은 날 실적 8-K가 매칭되면 → "소문 아니라 공식 확인됨" 신호로 Strategist에 전달.
