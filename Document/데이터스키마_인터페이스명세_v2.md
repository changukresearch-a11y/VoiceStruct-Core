# 데이터 스키마 · 인터페이스 명세 v2 — 정보분석(공시·뉴스) → Strategist

> 작성: 정보분석 담당(정창욱) · 대상: Strategist 담당(이은미) 및 전 팀
> 갱신: 2026-07-06 계약 개편 반영 — 번들 필드 축소·개명·재배치(공시 **18필드**·뉴스 **28필드**), 전달 방식을 **공유 DB 스냅샷 테이블(`tb_disclosure`/`tb_news`)** 로 전환
> 이전: 2026-07-02 팀 회의 피드백(**v1 폐기 → v2**)
> 구현: `app/common/analysis_bundle.py` (`DisclosureBundle` / `NewsBundle`) · `app/storage/db.py` (스냅샷 저장/조회)

## ⚡ TL;DR

1. **공시·뉴스는 완전 별개 2객체** — `DisclosureBundle`(**18필드**) / `NewsBundle`(**28필드**). (v1의 "종목당 1개 합산 Bundle"은 폐기. 합치기는 Strategist가.)
2. **전달은 공유 DB 스냅샷 테이블**로 — 두 객체를 직접 넘기지 않고 `tb_disclosure`/`tb_news`에 **시계열 append**(PK `(ticker, collected_at)`), Strategist가 종목별 **최신 행을 JOIN**해서 읽는다.
3. **모든 점수는 0.0~1.0 소수점**으로 통일. 방향도 **`sentiment_score`(0~1: 0=악재·0.5=중립·1=호재)**로 점수화하고, `sentiment` 라벨을 파생 병기. `importance`는 방향 뺀 강도(별도 축).
4. **종목 식별은 `ticker` + `trade_date`** 기준. `company_name`·`category`는 번들에서 빼고 **`tb_universe` JOIN**으로 읽는다(중복 제거).
5. **스크리닝(50개 선별)은 스크리닝 에이전트 몫** → 우리는 종목 리스트를 입력으로 받는다.

## 2. 공통 규약

### 2-1. 종목 식별 (두 Bundle 공통)

| 필드 | 타입 | 채우는 주체 | 설명 |
|---|---|---|---|
| `ticker` | `str` | 정보분석 | 종목 티커 (대문자). 예: `NVDA` |
| `trade_date` | `str` | 정보분석 | 분석 기준일 `YYYY-MM-DD` (구 `as_of`에서 개명) |
| `collected_at` | `str` | 정보분석 | 레코드 수집 시각(**초 포함** ISO). 스냅샷 PK의 일부(구 `created_at`에서 개명) |

> **`company_name`·`category`는 번들에서 제거** — `tb_universe`에 이미 있으므로 Strategist가 `ticker`로 **JOIN**해서 읽는다(중복 제거). 스크리닝 **입력** 단계에는 여전히 존재하는 필드다.

### 2-2. 점수 규약 (전부 `float` 0.0~1.0)

| 개념 | 규칙 |
|---|---|
| 정규화 | 내부 0~10 척도를 `/10` 하여 0~1로 (`_n10()`), 소수 2자리 반올림 |
| `sentiment_score` | **방향×강도를 0~1로** (`0.5 + 0.5 × signed`). 0=강악재·0.5=중립·1=강호재. **고도화(비대칭 신뢰 감쇠): 호재(+)는 `confidence`로 중립쪽 감쇠**(근거 약한 낙관 과신 방지), **악재(−)는 감쇠 안 함**(안전 우선·경고 유지) |
| `importance` | 강도·중요도만(부호 없음). 거래정지·상폐 계열이 1.0에 가까움 |
| `risk_score` | 하방·위험 강도 |
| `confidence` | 신뢰도. 공시=`certainty_level` 수치화 / 뉴스=`certainty × is_confirmed × source_trust` |
| `source_trust` | (뉴스 전용) 출처 신뢰도 (LLM 판단, 본문 톤+등급) |
| `grade_score` | (뉴스 전용) 출처 **정책등급** 점수. `ALLOW=1.0 · GRAY=0.6 · WATCH=0.3 · BLOCK=0.0` (신뢰가중 평균) |
| `confirmed_score` | (뉴스 전용) 확정도 = `확정건수 / 전체`. 공시는 늘 1.0(정보량 0)이라 번들에서 제거 |

> **불리언 → 0/1 정수:** `has_signal` · `hard_block`은 JSON에서 `1`/`0` 정수로 노출(의미 유지, 타입만 통일). `hard_block=1`은 여전히 매수 override.

### 2-3. 방향 — `sentiment_score` + `sentiment`

- **`sentiment_score`** `float` 0~1 — 방향을 점수화. `0.0`=강한 악재, `0.5`=중립, `1.0`=강한 호재. (방향×강도를 0~1 축에 인코딩)
- **`sentiment`** 라벨 — `positive` · `negative` · `neutral` · `mixed`. `sentiment_score`에서 파생 병기(가독성용). 호재+악재 혼재는 `mixed`.

> 세 필드 역할 분리: `sentiment_score`=방향(+강도 반영) · `importance`=순수 강도 · `sentiment`=방향 라벨.

### 2-4. `event_type` 통일 온톨로지 (11종, 공시·뉴스 공유)

`app/common/ontology.py` 단일 정의. 과세분화 금지(표본 부족).

```
earnings · guidance_change · ma · capital_raise · management_change ·
insider_trade · product_deal · analyst_rating · regulation_legal ·
delisting_halt · other
```

## 3. `DisclosureBundle` — 공시 결과 (단독, **18필드**)

> 2026-07-06 개편: 24→**18필드**. 삭제(6) `company_name`·`category`·`confirmed_score`·`fact_check`·`verdict`·`ref` / 개명 `as_of`→`trade_date`·`created_at`→`collected_at` / `reason`를 `sentiment_score` 바로 아래로 재배치. `tb_disclosure`(18컬럼)의 스키마와 1:1.

```python
class DisclosureBundle(BaseModel):
    # 종목 식별
    ticker: str
    trade_date: str                 # 분석 기준일 YYYY-MM-DD (구 as_of)
    collected_at: str = ""          # 레코드 수집 시각(초 포함, 스냅샷 PK 일부, 구 created_at)
    has_signal: int = 0             # 0/1 (분석된 공시 신호 유무)
    # 공시 원문 식별 (전달용)
    filing_title: str = ""          # 공시 제목
    filing_no: str = ""             # 공시 번호(accession) — 원문 역추적 ID
    filed_at: str = ""              # 업로드 일시(초 포함, acceptanceDateTime 우선)
    # 공시 신호 (점수 0~1)
    event_type: str = "other"
    sentiment: str = "neutral"      # 방향 라벨 (sentiment_score에서 파생)
    sentiment_score: float = 0.5    # 0~1: 0=강악재·0.5=중립·1=강호재
    reason: str = ""                # 한 줄 근거 (sentiment_score 바로 아래로 재배치)
    importance: float = 0.0         # 0~1 (강도, 방향 뺀 별도 축)
    risk_score: float = 0.0         # 0~1
    confidence: float = 0.0         # 0~1
    # 안전장치
    hard_block: int = 0             # 0/1 (매수 override)
    hard_block_reason: str | None = None
    # 전문 요약
    summary: str = ""               # ★ 공시 전문 3~4줄 요약
    keywords: list[str] = []        # ★ 핵심 키워드 5개
```

| 필드 | 타입/범위 | 의미 · 근거 |
|---|---|---|
| `ticker` · `trade_date` | str | 종목 식별. `company_name`·`category`는 `tb_universe` JOIN으로 읽음(번들에서 제거) |
| **`collected_at`** | str | 레코드 수집 시각(**초 포함** ISO). 스냅샷 PK `(ticker, collected_at)`의 일부(구 `created_at`) |
| `has_signal` | 0/1 | LLM 분석 신호가 실제로 있는지. 없으면 아래 신호 필드는 기본값 |
| **`filing_title`·`filing_no`·`filed_at`** | str | 공시 제목·번호(accession)·업로드 일시(**초 포함**, acceptanceDateTime 우선). 신호 유무와 무관하게 전달. `filing_no`(accession)가 원문 역추적 ID라 별도 `ref` 불필요 |
| `event_type` | 11종 | 공시 이벤트 유형 |
| `sentiment_score` | 0~1 | 방향 점수. 0=강악재·0.5=중립·1=강호재 |
| `sentiment` | 라벨 | 방향 라벨(점수에서 파생 병기) |
| `reason` | str | 한 줄 근거(수치 포함 권장). `sentiment_score` 바로 아래로 재배치 |
| `importance` | 0~1 | 강도만. 1~5일 스윙 영향. 거래정지·상폐≈1.0, 대형 M&A·실적쇼크 0.8~0.9 |
| `risk_score` | 0~1 | 하방 위험. 회계·희석·상폐일수록 ↑ |
| `confidence` | 0~1 | `certainty_level`(High=0.9/Med=0.6/Low=0.3) |
| `hard_block` | 0/1 | 상폐·거래정지·going concern 등 → 매수 override. **sentiment에 녹이지 않고 별도 플래그** |
| `hard_block_reason` | str/None | 차단 사유(결정저널·리뷰용) |
| **`summary`** | str | 공시 전문 **3~4줄 요약**(원문 근거만, 환각 방지) |
| **`keywords`** | list | 핵심 **키워드 5개**(검색/태깅용) |

> **삭제 필드(6)**: `company_name`·`category`(→`tb_universe` JOIN) · `confirmed_score`(공시는 늘 1.0, 정보량 0) · `fact_check`(늘 "Verified — official SEC filing" 상수) · `verdict`(`sentiment_score`+`reason` 재진술) · `ref`(`filing_no`가 이미 원문 역추적 ID). **뉴스는 값이 가변이라 `confirmed_score`·`fact_check`·`ref` 유지.**

## 4. `NewsBundle` — 뉴스 결과 (단독, **28필드**)

> 2026-07-06 개편: 31→**28필드**. 삭제(3) `company_name`·`category`·`verdict` / 개명 `as_of`→`trade_date`·`created_at`→`collected_at` / `reason`를 `sentiment_score` 바로 아래로 재배치. **`source_trust`·`grade_score`·`confirmed_score`·`fact_check`·`top_evidence`·`peak_importance`·`ref`는 뉴스 고유값이라 유지.** `tb_news`(28컬럼)의 스키마와 1:1.

```python
class NewsBundle(BaseModel):
    # 종목 식별
    ticker: str
    trade_date: str                 # 분석 기준일 YYYY-MM-DD (구 as_of)
    collected_at: str = ""          # 레코드 수집 시각(초 포함, 스냅샷 PK 일부, 구 created_at)
    has_signal: int = 0             # 0/1
    # 뉴스 원문 식별 (대표 기사)
    news_title: str = ""            # 뉴스 제목
    source: str = ""                # 출처 언론사
    published_at: str = ""          # 업로드 일시(초 포함)
    # 건수
    news_count: int = 0
    news_confirmed: int = 0         # 확정 사실 건수
    news_rumor: int = 0             # 루머/추측 건수
    # 대표 신호 (점수 0~1)
    event_type: str = "other"
    sentiment: str = "neutral"      # 방향 라벨 (sentiment_score에서 파생)
    sentiment_score: float = 0.5    # 0~1: 0=강악재·0.5=중립·1=강호재
    reason: str = ""                # 한 줄 근거 (sentiment_score 바로 아래로 재배치)
    importance: float = 0.0         # 0~1 (강도, 신뢰가중)
    peak_importance: float = 0.0    # 0~1 (집계 전 최고 잠재)
    risk_score: float = 0.0         # 0~1
    confidence: float = 0.0         # 0~1
    source_trust: float = 0.0       # 0~1 (LLM 판단)
    grade_score: float = 0.0        # 0~1 (출처 정책등급: ALLOW1·GRAY.6·BLOCK0)
    confirmed_score: float = 0.0    # 0~1 (확정 비율 = 확정건수/전체)
    fact_check: str = ""            # 팩트체크 한 문장(Verified/Unverified…)
    # 안전장치
    hard_block: int = 0             # 0/1
    hard_block_reason: str | None = None
    # 근거 · 전문 요약 (대표 기사 기준)
    top_evidence: list[str] = []
    summary: str = ""               # ★ 뉴스 전문 3~4줄 요약
    keywords: list[str] = []        # ★ 핵심 키워드 5개
    ref: str = ""                   # 원문 링크(뉴스는 filing_no가 없어 유지)
```

| 필드 | 타입/범위 | 의미 · 근거 |
|---|---|---|
| `ticker` · `trade_date` · `collected_at` | str | 종목 식별·기준일·수집시각(**초 포함**). `company_name`·`category`는 `tb_universe` JOIN으로 읽음(번들에서 제거). `collected_at`은 스냅샷 PK 일부(구 `created_at`) |
| `news_count` / `news_confirmed` / `news_rumor` | int | 분석된 뉴스 건수와 확정·루머 분해. "확정 여럿=신뢰↑, 한 곳뿐=루머 의심" |
| **`news_title`·`source`·`published_at`** | str | **대표 기사**의 제목·출처 언론사·업로드 일시(**초 포함**, RSS pubDate) |
| `sentiment_score` · `sentiment` | 0~1·라벨 | 방향 점수(0=악재·0.5=중립·1=호재)와 파생 라벨 |
| `reason` | str | 한 줄 근거. `sentiment_score` 바로 아래로 재배치 |
| `importance` | 0~1 | 신뢰가중 평균 강도 |
| `peak_importance` | 0~1 | 집계 전 최고 잠재 중요도. "잠재 큰데 저신뢰라 눌린" 경우 병기(`[최고imp..·저신뢰]`) |
| `source_trust` | 0~1 | (뉴스 전용) 출처 신뢰도(LLM 판단). 소셜·루머 방어 |
| `grade_score` | 0~1 | (뉴스 전용) 출처 **정책등급** 점수. ALLOW1·GRAY.6·BLOCK0 (신뢰가중, 소셜/미등록=GRAY) |
| `confirmed_score` | 0~1 | 확정 비율 = 확정건수 / 전체. (뉴스는 매번 값이 달라 **유지**) |
| **`fact_check`** | str | 팩트체크 한 문장(코드 판정). 확정→`Verified …`, 루머→`Unverified (rumor) …`. (Verified/Unverified 가변이라 **유지**) |
| **`hard_block`·`hard_block_reason`** | 0/1·str/None | 매수 override 플래그·사유 |
| `top_evidence` | list[str] | 대표 근거 헤드라인(나머지는 DB) |
| **`summary`·`keywords`** | str·list | **대표 기사**(신뢰가중 선정)의 전문 3~4줄 요약·키워드 5개 |
| `ref` | str | 원문 링크. 뉴스는 `filing_no`(accession)가 없어 역추적 ID로 **유지** |

> **삭제 필드(3)**: `company_name`·`category`(→`tb_universe` JOIN) · `verdict`(`sentiment_score`+`reason` 재진술). 나머지 뉴스 고유 필드는 전부 유지.

## 5. 집계 규칙 — 뉴스 여러 건 → 대표 1개

| 항목 | 규칙 |
|---|---|
| 방향/강도 | 확정·`source_trust`·`certainty` **신뢰가중**. 단 강한 악재(신뢰가중 signed ≤ −0.7)면 그쪽으로 보수적 |
| `sentiment_score` | 위 방향×강도를 `_senti_score`로 0~1 매핑. **호재만 `confidence`로 중립쪽 감쇠, 악재는 유지**(비대칭) |
| `importance` | 신뢰가중 평균(방향 뺀 크기). **감쇠 없음** — 방향/강도 축 분리 |
| `confidence` | `certainty × is_confirmed × source_trust`의 평균 |
| **대표 선정** | `event_type`·`reason`·`top_evidence`는 **`importance × confidence`(신뢰가중) 최대** 기사로. 시끄러운 저신뢰 루머가 대표가 안 되게(확정 위주) |
| `hard_block` | 어느 뉴스든 BLOCK류면 true (안전장치는 하나만 발동해도) |

> **산출 고도화 근거:** ① 방향점수는 **낙관만 신뢰로 깎고 악재는 유지**해 과신·펌프엔 보수적이되 경고는 흐리지 않는다("틀려도 작게 잃기"). ② 대표는 **가장 시끄러운 것(importance)이 아니라 가장 믿을 만하며 중요한 것(importance×confidence)**을 뽑아, 미확정 루머가 헤드라인이 되는 걸 막는다.

## 7. 전달 방식 (2026-07-06 개편)

- **정식 전달 계약 = 공유 DB 스냅샷 테이블.** 두 객체를 직접 넘기지 않고 DB에 저장한다.
  - **`tb_disclosure`(18컬럼)** · **`tb_news`(28컬럼)**, **PK = `(ticker, collected_at)`**. 각 번들 필드가 컬럼과 1:1.
  - 새 공시/기사가 뜰 때마다 **새 행을 append**(덮어쓰기 X, 시계열 누적). Strategist는 종목별 **최신 행**을 읽고, `company_name`·`category`는 `tb_universe`를 `ticker`로 **JOIN**해서 채운다.
  - 저장 함수 `save_disclosure_bundle` / `save_news_bundle`, 읽기 `latest_disclosure` / `latest_news` (`app/storage/db.py`).
  - 수집 주기: 공시=**1시간**, 뉴스=**5분**(새 것만 처리). 중복제거 키: 공시=accession(`filing_no`), 뉴스=google_link.
- **보조 = `to_prompt()` 텍스트 / `model_dump_json()`** — 사람이 읽는 요약·LLM 프롬프트 삽입용. 같은 객체의 표현이며, 정식 계약은 DB 행(또는 그 JSON)이다.
- **별개 유지 = per-signal 분석로그**(`disclosure_signals` / `news_signals`) — 스케줄러·백테스트·결정저널·replay용. `tb_*`(전략가 출력 전용)와는 **별개 테이블**이다.

엔트리: `python run_bundle.py --ticker NVDA --llm --save` → 공시·뉴스 번들 2개 산출 후 `tb_disclosure`/`tb_news`에 append. (회사명·카테고리는 tb_universe JOIN이라 인자 없음)

## 8. 이은미와 합의할 열린 질문

1. **결합 위치** — 공시·뉴스를 Strategist가 어떻게 합칠지(가중치). 우리는 합산 안 함(v2).
2. **집계 가중치 계수** — 신뢰·최신성 초기값은 임의 → 백테스트로 튜닝(자동조정 아님·수동).
3. **`category` 값 체계** — 스크리닝 에이전트가 넘길 카테고리 라벨 표준(GICS 섹터? 자체 분류?) 합의 필요.
