에이전트 뉴스(News) 분석

공시에 없는 정보(애널리스트 의견·외신·업계 루머)를 잡되, 루머·찌라시·펌프가 섞여 있어 신뢰도·사실성 판정이 핵심이다. 다층 방어로 노이즈를 거른다.

Google News RSS (키 불필요) 사전필터로 LLM 전 차단 출처 3단계 + 키워드 is_confirmed · source_trust 출력 NewsBundle (0~1)

01 왜 뉴스인가 — 공시보다 빠르지만 노이즈

뉴스는 공시에 없는 정보(애널리스트 의견·외신·업계 루머)도 다루고, 큰 사건은 공식 공시 전에 언론이 먼저 보도하기도 한다(예: "A사 인수 추진 중인 것으로 알려져" → 며칠 뒤 공시). 즉 공시와 다른 정보원이라 더 빠르고 폭넓게 분위기를 잡는다.

⚠️ 대신 노이즈·추측이 많아 공시로 사실 확인이 필요하다. 문맥·뉘앙스는 LLM이 호재/악재/테마로 해석한다. 뉴스 = 신뢰도 필터가 전부.
🍳 쉽게 말하면 — 공시가 "도장 찍힌 서류"라면, 뉴스는 "빠르지만 진위가 섞인 소문·보도". 그래서 뉴스는 "이거 진짜야?"를 여러 겹으로 검증하는 게 일의 핵심이다.

02 수집 — Google News RSS

키 불필요 — 종목명/티커로 Google News RSS를 실수집한다. API 키가 없어도 되고, RSS의 <source> 태그로 출처(Reuters·Bloomberg 등)를 분류한다.
기사별 유니크 키 — dedup(중복 제거) 키는 google_link(RSS <link>, 기사별 유니크)를 쓴다. url은 출처 도메인이라 같은 언론사 기사가 뭉개지므로 쓰지 않는다.
💡 DB 기반 dedup — news_signals 테이블에 news_key UNIQUE + INSERT OR IGNORE로, 스케줄러가 재시작해도 이미 본 기사는 재분석·재LLM 하지 않는다. 드롭된 뉴스도 "본 것"으로 저장한다.

03 사전 필터 — LLM 호출 전 조기 종료

수집한 기사는 LLM에 가기 전에 출처 등급과 키워드 두 관문을 통과해야 한다. 여기서 걸리면 LLM 비용을 아예 쓰지 않는다(비용↓ + 정확도↑).

3-1. 출처 3단계 정책 (WATCH_ONLY 소셜 등급 제거 — 팀 결정 2026-07-03)

등급출처 예시취급
ALLOWSEC·Reuters·Bloomberg·WSJ·CNBC분석 반영
GRAYBenzinga·SeekingAlpha·Zacks + 소셜(X·Reddit)·미등록단독 매수금지
BLOCK광고·리딩방·AI SEO 복붙즉시 drop

소셜/미등록 도메인은 이제 전부 GRAY로 떨어진다(전용 소셜 등급 폐지). 루머는 키워드 사전필터·LLM is_confirmed가 거른다.

⚠️ ALLOW ≠ 매수허용. 분석에 반영 가능하다는 뜻일 뿐, 시장반응·리스크 필터를 더 통과해야 한다.

3-2. 키워드 필터 (화이트 우선)

  • whitelist 우선 통과 — false negative 방지
  • blacklist 매치 → drop (WATCH_ONLY로 종료)
  • 둘 다 없으면 pass → LLM으로
# 사전필터 통과 못 하면 LLM 스킵:
출처 == BLOCK  → BLOCK_BUY,   LLM 스킵
키워드 == drop → WATCH_ONLY,  LLM 스킵

04 NewsSignal — LLM 출력 스키마 (공시와 별도)

뉴스 전용으로 사실/루머 구분과 출처 신뢰도 필드를 추가로 둔다. 추론 강제(reasoning→certainty)는 공시와 동일 사상.

class NewsSignal:
    reasoning; certainty_level     # 공시와 동일 추론 강제(CoT + 모호성)
    event_type; sentiment          # 통일 12종 · pos/neg/neutral/mixed
    importance: int (0~10)
    risk_score: float (0~10)
    is_confirmed: bool             # ★ 확정 사실인가? '~알려져/전망/소문'=False
    source_trust: float (0~1)      # ★ 출처 신뢰도(정책 등급 + 본문 톤)
    theme: list                    # 테마/섹터 태그
    hard_risk_flag; trade_permission
    reason; evidence_quotes        # 한 줄 근거 · 원문 인용(환각 방지)
    summary: str                   # ★ 뉴스 전문 3~4줄 요약
    keywords: list                 # ★ 핵심 키워드 5개
💡 요약·키워드 (신규) — 점수와 별개로 전문 3~4줄 요약·키워드 5개를 함께 낸다. 원문 근거로만 채워 환각 방지. 확정/추측 뉘앙스는 confirmed_score·fact_check가 담으므로, 재진술이던 호재/악재 판정 verdict은 계약에서 제거됐다.
💡 is_confirmed와 source_trust가 공시엔 없는 뉴스 전용 축이다. "누가, 얼마나 믿을 만하게, 확정된 사실을" 말했는지를 분리해 신뢰도를 방향과 떼어 낸다.

05 다층 방어 — 루머는 세 번 걸러진다

① 사전 필터 — 출처 BLOCK·키워드 blacklist는 LLM 전에 drop.
② LLM 재판정 — 통과분도 LLM이 is_confirmed=False(추측)로 다시 분류.
③ 집계 감점 — 루머 비중만큼 confidence를 깎는다(신뢰가중).
🍳 실작동 (AAPL·NVDA 실데이터 검증) — NVDA 8건 중 확정 1·루머 7 → 신뢰도가 0.14로 하락. "루머뿐이면 신뢰↓"가 실제로 작동한다.
⚠️ 가짜뉴스·조작을 100% 막는 시스템은 없다 → 4겹으로 최대한 거르되, 속아도 손절이 최종 방어선(손절은 뒤 단계 몫).

06 집계 규칙 — 여러 건 → 대표 1개

종목당 뉴스 여러 건을 대표 1개로 집계해서 넘긴다(뉴스 10건 덤프 금지 — 토큰·노이즈 절감).

항목규칙
방향/강도확정·source_trust·certainty 신뢰가중. 단 강한 악재(신뢰가중 signed ≤ −0.7)면 보수적으로 그쪽
importance신뢰가중 평균(방향 뺀 크기). peak_importance는 신뢰검증 기사 중 최댓값을 별도 병기
confidencecertainty × is_confirmed × source_trust 조합 → 루머 많으면 급감
hard_block어느 뉴스든 BLOCK류 권한이면 1 (안전장치는 하나만 발동해도)
💡 peak_importance = 신뢰검증 기사 중 최댓값 (2026-07-07 편향 수정) — 확정 or source_trust≥0.6 or 출처 ALLOW인 기사 중에서만 최댓값을 잡는다. 저신뢰 매체가 자극적으로 부풀린 importance가 가짜 peak를 세우는 것을 차단. 신뢰 대형신호가 평균에 묻히면 [최고imp0.8·low-conf]로 부각하고, 신뢰검증 기사가 없으면 importance로 수렴한다(peak≥importance 불변식 유지).

6-1. 산출 방식 고도화 & 근거

단순 평균은 "시끄러운 루머"에 휘둘린다. 그래서 방향 점수와 대표 선정 두 곳을 신뢰도 기준으로 고도화했다.

① sentiment_score — 비대칭 신뢰 감쇠

방향 점수를 0.5 + 0.5 × signed로 매핑하되, 호재(+)만 confidence로 중립(0.5) 쪽으로 감쇠한다. 악재(−)는 감쇠하지 않는다.
# signed = 방향×강도(-1~+1),  confidence = 집계 신뢰도(0~1)
if signed > 0:              # 호재만 감쇠
    signed *= confidence
sentiment_score = 0.5 + 0.5 * signed

_senti_score(+1.0, conf=1.0) = 1.00   # 확신 호재 → 그대로
_senti_score(+1.0, conf=0.2) = 0.60   # 저신뢰 호재 → 중립쪽으로 눌림
_senti_score(-1.0, conf=0.2) = 0.00   # 저신뢰 악재 → 감쇠 안 함(경고 유지)
🍳 근거 — 루머성 낙관은 과신·펌프 위험이라 신뢰가 낮으면 방향을 덜 단정해야 한다. 반대로 악재는 근거가 약해도 흐리면 위험하다("틀려도 작게 잃기"). 그래서 호재는 깎고 악재는 유지하는 비대칭이 "가장 보수적 채택" 사상과 맞는다. importance(순수 강도)는 감쇠 없이 그대로 둬서 방향/강도 축을 분리 유지한다.

② 대표 신호·근거 — 신뢰가중 선정

여러 건 중 대표 이벤트(event_type·reason·top_evidence)를 importance 만이 아니라 importance × confidence(신뢰가중)가 가장 큰 기사로 뽑는다.
💡 근거 — 대표를 순수 importance로 뽑으면 시끄럽지만 미확정인 루머(예: "인수될 것으로 알려져", imp9·저신뢰)가 헤드라인이 된다. 신뢰가중으로 뽑으면 imp가 좀 낮아도 확정·신뢰 높은 사실(예: "실적 확정 beat", imp6·고신뢰)이 대표가 된다. 검증: 루머(imp9·GRAY·미확정) vs 확정(imp6·ALLOW·확정) → 대표 = 확정 실적.

07 매매권한 · "공시 기다리면 늦지 않나?"

뉴스도 공시와 같은 공통 decision 레이어(6단계, 가장 보수적 채택)를 통과한다. 미확인 뉴스·소셜 단독은 보수적으로 강등된다.

🔗 "공시 기다리면 늦지 않나?" — 맞다. 그래서 진입의 판단 근거는 공시가 아니라 실시간 가격·거래량이다(공시는 늦게 떠 진입 게이트로 못 씀). 단 이미 급등한 뒤 추격은 "고점 매수" 위험(buy the rumor, sell the news). 이 실시간 검증·실제 매수는 이 파트가 아니라 Strategist 몫이다 — 우리는 신뢰도 판정까지만.
⚠️ 이 파트는 뉴스만으로 매수를 확정하지 않는다. 시장반응(가격·거래량) 확인과 진입 타이밍은 범위 밖(Strategist·리스크 게이트).

08 산출물 — NewsBundle (Strategist 입력)

뉴스 최종 산출물. 공시와 완전 별개 객체로 넘어가고, 모든 점수는 0~1 소수점으로 통일한다. 라벨성 값(확정·출처등급)도 점수로 노출한다.

class NewsBundle:                     # 28필드 (company_name·category는 tb_universe JOIN)
    ticker; trade_date; collected_at  # ★ trade_date=거래일 · collected_at=수집시각(초, DB append)
    has_signal
    news_title; source; published_at  # ★ 대표기사 제목·언론사·업로드일시(초 포함)
    news_count; news_confirmed; news_rumor   # 건수 분해(확정 n·루머 m)
    event_type
    sentiment; sentiment_score: 0~1   # 방향 라벨 + 점수(0=악재·0.5=중립·1=호재)
    reason                            # ★ 한 줄 근거 — sentiment_score 바로 아래
    importance: 0~1                   # 강도(신뢰가중)
    peak_importance: 0~1              # 신뢰검증 기사 중 최댓값
    risk_score: 0~1
    confidence: 0~1
    source_trust: 0~1                 # 뉴스 전용(LLM 판단)
    grade_score: 0~1                  # 출처 정책등급(ALLOW1·GRAY.6·BLOCK0)
    confirmed_score: 0~1              # 확정 비율 = 확정건수 / 전체
    fact_check                        # ★ 팩트체크 한 문장(Verified/Unverified…)
    hard_block: 0/1 + hard_block_reason
    top_evidence: list                # 대표 기사 원문 인용
    summary; keywords                 # ★ 대표 기사 요약·키워드5
    ref                               # 원문 링크(공시와 달리 accession 없어 유지)
    def to_prompt() -> str
💡 대표 기사 기준 — 뉴스는 여러 건을 대표 1개로 집계하므로, summary·keywords·reason과 원문 식별(news_title·source·published_at)은 신뢰가중(§6-1)으로 뽑힌 대표 기사의 값이다. published_at은 초까지 포함(RSS pubDate). 시끄러운 루머가 아니라 확정·신뢰 높은 기사가 실린다.
💡 source_trust(LLM 판단)와 grade_score(정책 등급)를 분리 노출한다. 확정 여부도 bool이 아니라 confirmed_score(확정건수/전체)로 점수화 — 모든 값이 숫자라 파싱·검증이 안정적. source_trust·grade_score·confirmed_score·fact_check·top_evidence·peak_importance·ref는 뉴스 전용 축이라 유지된다(공시는 제거).
🗄️ 전달 = 공유 DB 스냅샷 — 정식 계약은 2객체 직접 전달이 아니라 tb_news(28컬럼) 스냅샷 테이블이다. PK=(ticker, collected_at), 새 기사가 뜰 때마다 새 행 append(덮어쓰기 X, 시계열 누적)하고 Strategist는 종목별 최신 행을 tb_universe(company_name·category 보유)와 JOIN해 읽는다. 저장 save_news_bundle · 읽기 latest_news. 수집 주기 = 5분(새 것만 처리, 중복제거 키 = google_link). to_prompt()는 보조. 기존 news_signals(per-signal 로그)는 그대로 유지되는 별개 테이블.

8-1. to_prompt() 실출력 예시

[NVDA] News · 2026-07-03
 News: 3 items (confirmed2·rumor1) → positive(0.84) imp0.67 trust0.88 grade0.95 conf0.6 [peak imp0.8·low-conf]
       Source: reuters.com · published 2026-07-03T14:30:05+00:00
       Reason: "Confirmed Q2 earnings beat, guidance raised"
       Fact-check: Verified — 2/3 confirmed, trust 0.88, grade 0.95.
       Summary: Q2 results beat consensus. Revenue +22% YoY. Guidance raised.
       Keywords: earnings, surprise, revenue +22%, guidance raised, semiconductors
       Evidence: "reuters.com: Nvidia earnings beat expectations Q2"
 Overall: hard_block=0 · generated 2026-07-03T14:31:00+00:00
💡 출력 언어 = 영어 — 자유서술(summary·reason·fact_check)과 to_prompt() 라벨은 전부 영어(전략가 에이전트가 읽음). enum·점수는 공통.

09 역할 경계 — 하는 것 / 안 하는 것

✔ 이 파트가 한다

  • Google News RSS 실수집(키 불필요) + DB 기반 dedup
  • 출처 3단계 + 키워드 사전필터(LLM 전 조기 종료)
  • LLM 사실/루머·출처신뢰 판정(is_confirmed·source_trust) + 다층 방어
  • NewsBundle 산출·전달 + 백테스트 수익률(측정만)

✘ 이 파트가 안 한다 (Strategist·이후 몫)

  • 실시간 가격·거래량 검증 · 실제 매수 확정(고점 추격 판단)
  • 공시와 뉴스의 결합·가중치 — 두 Bundle을 Strategist가 직접 합침
  • 가중치 자동 튜닝 — 과적합 방지로 측정·리포트까지만
🍳 한 줄 요약 — 뉴스 파트는 "빠르지만 진위가 섞인 정보를, 여러 겹으로 신뢰도 판정해서" Strategist에게 넘긴다. 화제성≠신뢰라 확정·출처를 점수로 분리해 넘기고, 진입 판단은 넘긴다.
