# Quantinue · 공시·뉴스 개발·고도화 상세 보고서

> **정보분석 파트 ①공시(05) ②뉴스(06)** · 담당 정창욱(collector-agent) · 스택 Python · Pydantic AI · OpenAI gpt-4o-mini · SEC · Google News(키 불필요 실수집)
> 산출물 **DisclosureBundle(18) · NewsBundle(28)** · 핵심축 **강도 · 방향 · 신뢰** · 문서일 **2026-07-07**

SEC 공시와 실시간 뉴스를 구조화 신호로 바꿔 Strategist에게 넘기는 파트의 **기능·데이터 흐름 전체**와, 수집→해석→집계→전달 파이프라인이 **어떤 개발 과정을 거쳐 어떻게 고도화됐는지**를 코드·공식·검증 결과까지 담은 종합 문서.

## 01 · 개요 · 파이프라인 위치

전체 11단계 파이프라인 중 **05 공시 · 06 뉴스**가 우리 파트. 스크리닝이 넘긴 오늘의 50종목에 대해, 각 종목의 공시·뉴스를 구조화 신호로 만들어 **07 Strategist**에게 넘긴다. **구조화 신호 + 보수적 1차 판단까지만** — 실제 매수권한·시장반응 검증은 Strategist 몫(범위 밖).

```
01~03 유니버스·기술·데일리픽 → 04 매크로 → [05 공시] [06 뉴스] → 07 Strategist → 08~11 Critic·리스크·주문·회고
                                          └── 우리 담당 ──┘
```

| 단계 | 내용 |
|---|---|
| **입력** | `tb_daily_pick`의 50 티커. 카테고리·랭크는 스크리닝이 전달(우리는 통과). |
| **처리** | 공시 **1시간** · 뉴스 **5분** 간격 수집 → 새 건만 LLM 분석 → 신뢰가중 집계. |
| **출력** | `tb_disclosure`(18) · `tb_news`(28) 스냅샷. Strategist가 최신 행 JOIN. |

**설계 사상 3줄** — ① **"코드=사실, LLM=해석만"**(숫자·정형데이터는 코드가 확정). ② 모든 결과값은 **강도·방향·신뢰 3축 분리**. ③ **"가장 보수적 채택"** — 틀려도 작게 잃기(악재 우선, 근거 약한 낙관 감쇠).

## 02 · 아키텍처 · 모듈맵

공시·뉴스는 **수집·분석 로직은 완전 분리**하되 **공통 배관(집계·저장·결정·온톨로지)은 공유**한다. 중복 없이 각 소스 특성만 어댑터로 처리하는 Walking Skeleton 구조.

```
수집(collectors) → 라우팅·필터(router·policies) → 분석(analyzers, LLM) → 집계(analysis_bundle) → 저장(storage/db)
```

| 레이어 | 공시 모듈 | 뉴스 모듈 | 공유 |
|---|---|---|---|
| 수집 | `sec_collector`·`sec_xbrl_collector`·`sec_form4_collector` | `news_collector`(Google News RSS) | `NormalizedItem` 스키마 |
| 라우팅·필터 | `form_type_router`·`form_8k_item_router`·하드리스크 | `source_trust_policy`·키워드 사전필터 | — |
| 분석(LLM) | `disclosure_analyzer`·`form4_scorer`(코드) | `news_analyzer` | Pydantic AI · 프롬프트별 분리 |
| 집계 | `build_disclosure_bundle` | `build_news_bundle` | `analysis_bundle`·`ontology` |
| 결정·저장 | `decision/trade_permission_policy`(보수적 1차 권한) · `storage/db`(SQLite) | (공통) | 공통 재사용 |

## 03 · 공시 파이프라인 — 3축

공시는 Form 종류에 따라 **코드/LLM을 차등 라우팅**한다. "숫자=코드, LLM=해석만" 원칙을 Form별로 구현. `form_type_router`가 8-K=Item / 10-Q·K=XBRL / Form4=코드로 분기.

- **① 8-K (Item 라우터)** — `sec_collector`가 submissions→Item 청킹 → `item_event_map`이 Item번호로 1차 분류·기본권한 부여 → 하드룰 → LLM 해석. 예: 3.01 상폐통지=BLOCK_BUY·hard_risk / 2.02 실적 / 5.02 경영진변경.
- **② 10-Q/K (XBRL)** — `sec_xbrl_collector`가 companyconcept API로 YoY 수치 계산(코드) → LLM은 **수치 해석만**. 본문 리스크문구(going concern·material weakness)는 원문 별도 스캔.
- **③ Form 4 (코드 스코어링)** — `sec_form4_collector`가 raw XML 파싱 → `form4_scorer`가 **LLM 없이** 코드로 스코어(공개매수=호재·매도=약한악재·베스팅/세금/옵션=중립, 직급 가중).

> **공통 배관 재사용** — 세 축 모두 결과를 `DisclosureSignal`로 변환해 같은 집계·저장·결정 파이프라인에 흘린다. Form4는 코드 산출이지만 `event_type=insider_trade`로 동일 배관 사용.

**안전장치 — 규칙 기반 확정 차단**: 공시의 `hard_block`은 확률적 감점이 아니라 **규칙 기반**이다. 8-K Item 3.01/4.01(상폐·감사인교체)은 `default_permission=BLOCK_BUY`라 즉시 차단. 최종 권한은 **하드룰/LLM/Item 중 가장 보수적**으로 채택.

## 04 · 뉴스 파이프라인

Google News RSS로 실수집(키 불필요). 공시와 로직은 분리하되 배관은 공유. **LLM 호출 전에 두 겹의 사전필터**로 노이즈·저신뢰를 먼저 걸러 비용·오탐을 줄인다.

```
수집(RSS→NormalizedItem) → 사전필터①(출처등급 BLOCK drop) → 사전필터②(키워드 추측·낚시 drop) → 분석(LLM→NewsSignal) → 집계(신뢰가중→NewsBundle)
```

**한 기사 = 한 묶음**: RSS `<item>` 하나가 `NormalizedItem`으로 — 제목·본문(RSS 빈약→헤드라인 중심)·언론사(`<source>`)·링크 2종(도메인=등급용 / `google_link`=dedup)·발행시각(`pubDate`).

**사전필터 (LLM 호출 전 차단)**

| 필터 | 기준 | 동작 |
|---|---|---|
| 출처 등급 | BLOCK 도메인/조건 | LLM 스킵 + drop |
| 키워드(블랙) | reportedly·sources say·class action·stocks to watch 등 | drop |
| 키워드(화이트 우선) | share buyback·acquisition·fda approval 등 | 우선 통과(false negative 방지) |

> **다층 신뢰 방어** — 뉴스는 진위가 섞여 위험하므로 신뢰를 한 곳에 몰지 않는다: **사전필터(코드) → LLM `is_confirmed`(사실/추측) → LLM `source_trust`(본문 신뢰) → 코드 `grade_score`(도메인 등급) → 교차확인**의 여러 겹.

## 05 · 집계 계약 — DisclosureBundle 18 · NewsBundle 28

Strategist에게 넘기는 종목당 스냅샷. 공시·뉴스 **완전 별개 2객체**로, Strategist가 두 소스를 직접 결합한다(팀 명세 수용, 2026-07-06).

- **DisclosureBundle (18필드)** — **공시 1건을 1:1 복사**(집계 없음 — 원본이 유일). event_type·sentiment_score·importance·risk_score·confidence + 원문식별(filing_title·filing_no·filed_at) + hard_block + summary·keywords. *신뢰/출처 필드 없음 — 출처가 SEC 하나로 고정(항상 최고 신뢰).*
- **NewsBundle (28필드)** — **N개 기사 → 종목 1개로 신뢰가중 집계**. 위 공시 필드 + 건수(count/confirmed/rumor) + 신뢰 3축(confidence·source_trust·grade_score) + confirmed_score·fact_check·peak_importance·top_evidence. *대표 기사(신뢰가중 최대)에서 제목·출처·요약 채움.*

> **정식 전달 = JSON**(`model_dump_json()`) · `to_prompt()`는 사람이 읽는 보조 표현. DB 스냅샷 `tb_disclosure`/`tb_news`에 시계열 append, Strategist는 최신 행만 읽음.

## 06 · 0~1 산출 지표 — 공식과 의미

모든 점수는 **0~1 소수점 통일**(2026-07-02 개편). 강도·방향·신뢰 3축으로 분리되며, 뉴스는 기사별 신뢰도 `_news_conf`로 가중한다.

**기사별 신뢰 가중치 (뉴스 집계의 근간)**

```
_news_conf = certainty(High.9/Med.6/Low.3) × 확정계수(확정1.0·루머0.5) × source_trust
# 확정된 High + 신뢰높은 출처일수록 1.0, 저신뢰 루머는 0에 근접
```

| 지표 | 축 | 산출 | 의미 |
|---|---|---|---|
| `importance` | 강도 | 강도의 신뢰가중 평균(방향 무관) | 사건 크기 · 호·악재 섞여도 상쇄 안 됨 |
| `peak_importance` | 강도 | **신뢰검증 기사 중 최댓값** ★ | 묻힌 대형 신호 감지(§09 고도화) |
| `sentiment_score` | 방향 | 방향×강도 가중평균 → 비대칭 감쇠 | 0=강악재·0.5=중립·1=강호재 |
| `sentiment` | 방향 | score에서 파생 | positive/negative/neutral/mixed |
| `risk_score` | — | 위험도 신뢰가중 평균 | 호재/악재와 별개 축 |
| `confidence` | 신뢰 | `_news_conf` 단순평균 | 루머 많으면 급락(정직) |
| `source_trust` | 신뢰 | LLM 출처신뢰 가중평균 | 본문 톤+등급, 사후 판단 |
| `grade_score` | 신뢰 | 도메인 등급 가중평균(ALLOW1·GRAY.6·BLOCK0) | 코드·사전, 감사 가능 |
| `confirmed_score` | 신뢰 | 확정건수/전체 | 3건 중 2건 → 0.67 |

**방향의 비대칭 감쇠 — "가장 보수적 채택"**: **호재만 confidence로 감쇠, 악재는 유지.** 근거 약한 낙관은 과신·펌프 위험이라 중립(0.5)쪽으로 눌러 덜 단정하고, 악재는 근거가 약해도 흐리면 위험하므로 그대로 살린다. 또 **강한 악재(≤ −0.7) 1건이면** 가중평균 무시하고 그쪽으로. "틀려도 작게 잃기".

## 07 · 출처 신뢰 정책 · 선정 기준

URL 도메인으로 3등급 판정(코드·LLM 무관·재현성 100%). **미등록은 보수적 기본 GRAY**. 등급→점수(`grade_score`)는 신뢰가중 평균된다.

| 등급 | 점수 | 부류·선정 근거 | 대표 도메인 |
|---|---|---|---|
| **ALLOW** | 1.0 | ① 1차 공식출처 ② 편집책임 1급언론 ③ 공식 배포사 | sec·nasdaq·finra·fda · reuters·bloomberg·wsj·ft·cnbc · businesswire·prnewswire |
| **GRAY** | 0.6 | 2차 매체·논평·집계(단독 신뢰 부적합) + 소셜·미등록 | benzinga·seekingalpha·fool·zacks·tipranks |
| **BLOCK** | 0.0 | 편집책임 없는 유료홍보·수익보장 | sponsored·paid promotion·원출처 없음 |

**선정 원칙(2026-07-07 문서화)** — 외부 신뢰지수를 그대로 쓴 게 아니라 **"원본(1차 출처) 근접성 + 편집·정정 책임"** 기준의 큐레이션.
- **ALLOW 세 부류**: ① 1차 공식출처(규제기관·거래소: sec/nasdaq/nyse/finra/fda = 사실상 원본) ② 1급 금융언론(편집·팩트체킹·정정보도 책임: reuters/bloomberg/wsj/ap/cnbc/ft/barrons/marketwatch) ③ 공식 보도자료 배포사(기업 원문 그대로: businesswire/prnewswire/globenewswire)
- **한계**: 업계 통념 기반이라 외부 감사표준(NewsGuard·MBFC) 미앵커링 → 후속 과제. 단 `grade_score`는 단일 관문이 아니라 다층 방어의 한 겹.

> **⚠️ ALLOW ≠ 매수허용** — "분석 반영 가능"일 뿐. 실제 매수는 시장반응·리스크 필터를 추가로 통과(Strategist 몫). `grade_score`(코드·사전) ≠ `source_trust`(LLM·사후) — 두 축 분리 노출.

*(상세: 결과값_필드_레퍼런스 §7-1)*

## 08 · 저장 · 증분 · 백테스트

같은 산출물이 두 갈래로 저장된다 — 기사/공시 1건씩의 **분석 로그**와, 집계된 **전달 스냅샷**.

| | `*_signals` (로그) | `tb_*` (스냅샷) |
|---|---|---|
| 한 행 | 기사/공시 1건 | 종목 1개(집계) |
| 용도 | 백테스트·증분처리 | Strategist 전달 |
| 키 | news_key(google_link) / accession | PK(ticker, collected_at) |

**메타레벨 증분 수집**: `peek_recent_filings`가 submissions를 **1회만** 호출해 form별 최신 accession을 문서 다운로드 없이 확인 → `processed_filings`로 처리여부 판정 → **새 건일 때만** 문서·LLM·저장. 뉴스도 DB 기반 dedup(재시작에도 유지).

**백테스트 피드백**: Yahoo 일봉으로 신호 발생일(공시 `filed_at`/뉴스 `published_at`) 기준 **1·3·5일 전방수익률**을 채우고, 이벤트/센티먼트/출처별 적중률 리포트. **가중치 자동조정은 안 함**(과적합 방지 — 측정·리포트까지만, 표본 쌓이면 튜닝).

## 09 · 고도화 타임라인 ★

단건 코어 → MVP 한바퀴 → Strategist 계약 → 정밀 고도화로 이어진 개발 과정. 각 단계마다 실데이터 검증을 붙였다.

| 시점 | 마일스톤 | 내용 |
|---|---|---|
| ~2026-07-01 | **단건 코어 + MVP 한바퀴** | 공시 3축 + 뉴스 실수집. 수집 스케줄러(메타증분)·뉴스 DB저장·유니버스 시총확장(2000)·백테스트 피드백까지 = MVP 한바퀴 완성. |
| 2026-07-02 | **Strategist 계약 개편 · 0~1 통일** | 공시·뉴스 완전 별개 2객체로 분리. 모든 점수 **0~1 소수점 통일**, sentiment 점수화(비대칭 신뢰 감쇠). 회사명·카테고리·원문식별 추가. |
| 2026-07-03 | **산출 고도화 · 출처 3등급화 · 영어화** | importance 방향 독립·라벨↔점수 일치 등 집계 버그 5종 수정. 출처 WATCH_ONLY(소셜) 제거→3등급. Strategist-facing 출력 영어화. 화제성(buzz) 과기능 제거. |
| 2026-07-06 | **번들 18/28 개편 · 스냅샷 DB** | 팀 명세 수용 — Disclosure 24→18·News 31→28, as_of→trade_date·created_at→collected_at 개명. `tb_disclosure`/`tb_news` 시계열 스냅샷 테이블 신설. |
