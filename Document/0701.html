01 개요 & 확정 사항

사용자는 ① 시드머니 ② 투자유형(안정형/균형형/공격형)만 정한다. 이후 에이전트가 24시간 자율 매매하고, 사용자는 입출금 시 잠깐 멈추는 "일시중지/재개" 외엔 개입하지 않는다.

✅ 공통 확정 (단계 무관)
  • ① 시장 = 미국만 (공시=SEC · 매매=토스)
  • ② 제어 = 완전 자율 + 일시중지/재개 (매 거래 승인 아님)
  • ③ LLM 혼합 (모델은 교체 쉬움 · 예정)
      · 분석가(공시·뉴스·매크로 등 호출 많음) = OpenAI GPT-4o mini (저가·대량)
      · 결정자(Strategist·최종 판단 호출 적음) = Claude Opus 4.8 (가성비 대안 Sonnet 4.6)
  • ④ 키 = 서버측 보관 (.env · 깃 커밋 금지)
  • ⑤ DB = Postgres + TimescaleDB
  • ⑥ 계좌 = 5인 각자 본인 토스 계좌 (잔고 있는 2명은 부모님 토스 API로 격리)
  • ⑦ 체결 = 최종은 무조건 실계좌 — 개발 3주 페이퍼(모의체결)로 연습 → 마지막 1주 실계좌 라이브(실투입 1·3·5·7·9) · 1·2차 단계와 무관

🧭 핵심 — 투자유형은 정책 객체(config)로 추상화해두고 1차엔 균형형 값만 채운다 → 2차에 안정·공격 config 두 개만 추가(코드 안 갈아엎음). 1차(각자 로컬) → 2차(AWS 중앙 5계좌 멀티운용)로 전환. 페이퍼→실계좌 전환은 단계와 무관하게 개발 3주 뒤 마지막 1주에.

설계를 관통하는 5원칙

  • LLM은 "해석", 예측·타이밍은 "규칙+ML" — LLM에 가격 예측을 맡기지 않는다.
  • 예측보다 리스크 관리 — "틀려도 작게 잃기"가 수익보다 우선.
  • 확신 없으면 안 산다 — 기대값(EV)이 +이고 신호가 합의될 때만 진입.
  • 실제 돈 전에 무조건 검증 — 페이퍼(가상) + 백테스트로 거른 뒤 실거래.
  • 정직 — "꾸준한 수익"은 약속 못 함. 시스템 완성도 + 낭만이 목적.

04 에이전트 vs 비에이전트 — 총괄

LLM 에이전트ML(학습) 규칙·계산데이터 수집코드·실행

표는 프로그램이 한 사이클을 도는 작동 순서(⓪→⑩)대로다. 에이전트 = 팀원 4명(1인 1에이전트), 비에이전트 = 팀장 김지현. 거장 페르소나는 Strategist(시그널)에 합쳐 이은미가 함께 맡는다.

【목표 1】 유니버스 구성 — 감시할 ~2,000개 고르기 (주1회 갱신 · 정적)

📋 사전 준비 유니버스 구성 — "어떤 2,000개를 감시할까?" (스크리너 전에 먼저)

왜 먼저냐 — 미국 상장 ~16,000개를 다 볼 순 없다. ⓪ 스크리너가 돌려면 "감시할 2,000개"가 이미 정해져 있어야 한다. 이건 매 사이클이 아니라 주1회 갱신하는 정적 단계. (목표 1)
2,000개 고르는 법
  • 코어 — 토스 거래가능 + 시총 내림차순 top ~2,000 + 일평균 거래대금 하한(유동성 확보) + penny/OTC 제외
  • 급등주 풀 (공격형 전용) — 매일 등락률·거래량 상위 스캔 → 정규 거래소 상장(OTC 제외) + penny 제외 + 시총·거래대금 최소 하한 통과분만 임시 추가(단 하한은 코어 top-2000보다 낮게 — 안 그러면 코어와 겹쳐 무의미), 조건 미달 시 제거. 급등주 1등은 대개 동전주·펌프라 이 필터가 핵심.
→ 안정·균형형 = 코어만 / 공격형 = 코어 + 급등주 풀

⓪ 규칙 스크리너 (유니버스 → 오늘의 후보)

왜 필요한가 — 유니버스가 ~2,000종목인데, 매 사이클마다 전부를 LLM(공시·뉴스·매크로 분석가)에 넣으면 비용·시간이 폭발한다. 그래서 그 앞에 값싼 규칙 필터를 둬서 "지금 볼 만한" 후보 20~50개만 추리고, LLM은 그 후보에만 돌린다. (위 📋 사전 준비에서 구성한 2,000개가 입력)
🍳 쉽게 말하면 — 스크리너 = 서류전형(1차 필터). 지원자 2,000명을 다 면접 볼 순 없으니, "조건 되는" 사람만 추려서 면접관(분석가 AI)에게 넘기는 거야. 앞서 "16,000개 중 AI가 어떻게 좋은 거 찾냐" 했을 때 — 이 스크리너가 그 "찾아주는" 역할을 대신한다.
어떻게 동작하나 — 토스 /prices batch로 ~2,000종목 스냅샷 → 유형에 맞는 트랙으로 점수 매겨 상위 N개 선정. 트랙이 유형 따라 다르다:
트랙고르는 기준주기
모멘텀
(공격형·단기)		거래량 급증 · 가격 급변(전일 대비 ±%) · 변동성 큰 것 · 오늘 공시 발생 · 뉴스 언급 급증매 사이클
가치
(안정형·장기)		저PER·저PBR(저평가) · 고배당 · 재무 우량(ROE↑·부채↓·이익 꾸준) · 장기 추세 양호(우상향)주/월 리밸런싱
혼합 (균형형)두 트랙 섞어서중간
전부 숫자 비교·정렬이라 LLM 안 씀(학습도 아님). 모멘텀=시세/거래량, 가치=재무·밸류 데이터(yfinance/Finnhub).
📊 가치 트랙 용어 풀이
  • PER (주가수익비율) — 주가가 연이익의 몇 배냐. 낮을수록 쌈(저평가). 예: PER 10 = 10년치 이익이 가격.
  • PBR (주가순자산비율) — 주가가 순자산(재산)의 몇 배냐. 1 미만이면 저평가(재산보다 싸게 거래).
  • ROE (자기자본이익률) — 내 돈으로 얼마나 잘 버나(효율). 높을수록 우량(15%+), 버핏이 중시.
  • 고배당 — 이익을 주주에게 나눠주는 돈이 많음(은행 이자 같은 현금흐름).
  • 부채↓ — 빚 적어 망할 위험 낮음(재무 안전). 이익 꾸준 — 매년 안정/성장.
→ 싸고(PER·PBR↓) + 잘 벌고(ROE↑) + 안정적(고배당·낮은 빚) + 우상향 = 가치 트랙이 찾는 우량주. (yfinance/Finnhub 무료 제공)

입력: 2,000종목 { 시세·거래량·공시플래그·뉴스언급수 }
출력: 후보 30개 (거래량 급증 + 오늘 공시 뜬 것 위주)  → 분석가에게 전달
비용 효과 (핵심) — 스크리너 없으면 LLM 호출 2,000번/사이클, 있으면 ~30번 → 약 60배 절감. 후보 개수 N과 필터 기준은 백테스트로 튜닝(많으면 비용↑, 적으면 기회 놓침).
필요 API — 토스 batch 시세(거래량 포함) · 종목목록=공개 미국 상장 리스트/yfinance · 공시 플래그=SEC · 뉴스 언급=뉴스 소스 · 재무·밸류(가치 트랙)=yfinance/Finnhub

① 에이전트 공시(Disclosure) 분석

왜 필요한가 — 기업의 "공식 발표(공시)"는 주가를 직접 움직이는 가장 신뢰도 높은 정보다. 뉴스는 추측·과장이 섞이지만, 공시는 법적으로 의무 제출하고 거짓이면 처벌받는 "도장 찍힌 사실"이라 호재/악재 판단의 기준점이 된다.
🍳 쉽게 말하면 — 공시 = 기업이 정부(SEC)에 내는 공식 신고서. "우리 이런 일 있었습니다"를 법적으로 보고하는 것. 거짓 보고하면 제재받으니 소문·뉴스와 신뢰도 차원이 다르다. SEC=미국 증권 감독기관, EDGAR=그 공시를 누구나 무료로 보는 공개 시스템.
공시 종류(Form)와 호재/악재 경향 — 무슨 서류냐로 1차 성격이 갈린다:
Form내용경향
8-K돌발 주요사건(M&A·대형계약·CEO사임·소송) — 트레이딩에 가장 중요±(사건 따라)
10-Q / 10-K분기 / 연간 실적 보고서실적 좋으면 호재, 쇼크면 악재
Form 4내부자(임원) 매매 — CEO가 자사주 사면 자신감 신호매수=호재 / 매도=약한 악재
S-1 / 424B증자·신주 발행(유상증자류)보통 악재(희석)
미국 상장사 공시가 하루 수백~수천 건이라 사람이 다 못 읽음 → LLM이 대신 읽고 점수화한다.
어떻게 동작하나
  1. 감지 — SEC/EDGAR를 주기적으로 확인해 워치리스트 종목의 새 공시가 뜨면 가져옴(이벤트 트리거)
  2. 발췌 — 공시 원문이 길면 핵심만 추려 LLM 토큰 절약
  3. 해석 — LLM에 "이 종목에 호재냐 악재냐, 얼마나 중요하냐"를 물음
  4. 구조화 출력 → {sentiment: 호재/악재/중립, importance: 0~10, reason: 한 줄 근거}
  5. 전달 — 이 신호가 Strategist(시그널 제안)에게 가서 매매 판단에 반영
중요도(importance) 점수는 어떻게 매기나? — 공시 종류 × 규모 × 종목 영향으로 LLM이 0~10 부여. 예: 거래정지·상폐 = 10, 대형 M&A·실적 쇼크 = 8~9, 자사주 매입 = 6~7, 임원 소폭 매도 = 3, 일상 보고 = 1~2. 점수 높은 것만 매매에 크게 반영, 낮은 건 거의 무시.
입력 → 출력 예시 
입력: "Apple 8-K: 분기 배당 4% 인상 + 900억 달러 자사주 매입 승인"
출력: { sentiment:"호재", importance:7, reason:"자사주 매입=주당가치↑·주가 부양 신호" }

입력: "XYZ S-1: 2억 달러 신주 발행으로 자금 조달"
출력: { sentiment:"악재", importance:6, reason:"신주 발행=기존 주주 지분 희석" }
왜 규칙 아니고 LLM? — 같은 "유상증자"도 호재(우량 기관 대상 대규모 투자유치)일 수도 악재(자금난에 주식 희석)일 수도 있다. 자금 용도·대상·규모를 읽어야 판단되니 문맥 해석이 필수 → 키워드 매칭으론 못 잡아 LLM이 맡는다.
🔗 단, 상장폐지·거래정지 공시는 LLM 판단을 기다리지 않고 하드룰로 즉시 매수금지·강제청산(아래 ⑧ 리스크 게이트). "지능"이 아니라 "안전장치"라서.
한계 — 공시는 정확하지만 느리다(뉴스·가격이 먼저 움직인 뒤 뜨기도). 그래서 공시 단독보다 뉴스·가격과 함께 교차로 본다.
필요 API — SEC EDGAR (키 불필요, User-Agent 헤더만 넣으면 무료) · LLM(싼 모델)

② 에이전트 뉴스(News) 분석

왜 필요한가 — 뉴스는 공시에 없는 정보(애널리스트 의견·외신·업계 루머)도 다루고, 큰 사건은 공식 공시 전에 언론이 먼저 보도하기도 한다(예: "A사 인수 추진 중인 것으로 알려져" → 며칠 뒤 공시). 즉 공시(공식·느림·정확)와 다른 정보원이라 더 빠르고 폭넓게 분위기를 잡는다. 대신 노이즈·추측이 많아 공시로 사실 확인이 필요. 문맥·뉘앙스는 LLM이 호재/악재·테마로 해석.
어떻게 — 영어 뉴스(RSS·Finnhub) 헤드라인 → LLM {sentiment, strength, theme[], target, is_confirmed, source_trust}.
찌라시·루머 어떻게 거르나 (신뢰도 필터 4겹)
  1. 소스 화이트리스트 — Reuters·Bloomberg·CNBC·회사 IR·SEC 등 신뢰 매체에 가중, 무명 블로그·익명 소문은 배제/감점 (Finnhub의 source 필드 활용)
  2. 다중 출처 교차확인 — 같은 사실을 신뢰 매체 여러 곳이 보도해야 신뢰↑, 한 곳만이면 루머 의심
  3. 사실 vs 추측 태깅 — LLM이 "~알려져·전망·소문"(추측) vs "발표·공시"(확정)을 구분해 is_confirmed 표시, 추측성은 가중치↓
  4. 실시간 가격·거래량으로 즉시 교차검증 — 교차검증 주역은 공시가 아니라 실시간 가격·거래량(공시는 늦게 뜨므로 진입 게이트로 못 씀). 신뢰 뉴스 + 거래량 급증·가격 반응이 함께면 = 시장이 진짜로 받아들이는 중 → 진입. 공시는 사후 보강(있으면 신뢰↑, 없어도 가격이 받쳐주면 OK)
+ Risk Critic이 미확인 뉴스 기반 시그널을 적대적으로 기각, 소셜발 급등은 오히려 펌프 의심으로 회피.
"공시 기다리면 늦지 않나?" — 맞다. 그래서 진입 게이트는 공시가 아니라 실시간 가격·거래량이다. 단 이미 급등한 뒤 추격은 "고점 매수" 위험(buy the rumor, sell the news — 공시 확정 시점이 오히려 매도 타이밍일 때가 많음) → 과열(예: 이미 +X%) 시 회피·축소 또는 눌림목 대기 + 트레일링 스탑 보호. 진입 속도는 유형별: 공격형=신뢰뉴스+거래량이면 즉시 추격(손절 타이트), 안정형=더 확인 후(놓쳐도 OK).
⚠️ 가짜뉴스·조작을 100% 막는 시스템은 없다 → 4겹으로 최대한 거르되, 속아도 손절이 최종 방어선.
🧪 왜 감성을 LLM 전담으로? — 처음엔 FinBERT(무료 금융 감성 모델)로 1차 전처리하려 했는데, 직접 설치해 돌려보니 명확한 호재/악재는 잘 잡지만 유상증자·보합 같은 미묘한 건 틀리면서도 0.9+ 고확신을 줬다(캘리브레이션 X → "점수 높으면 믿어도 된다"가 성립 안 함). 그래서 1차 전처리로 부적합 판단 → 감성은 전부 LLM으로 돌리기로. (스크리너가 후보를 20~50개로 좁혀 LLM 비용도 작음 — 6장 상세)
필요 API — 영어 뉴스 RSS / Finnhub news(가입) · 감성은 LLM 전담 (6장 참고)

데이터+LLM ②+ (선택) 소셜 감성 — 뉴스 옆 보조 신호

왜 — "이유 없는 급등"의 상당수는 소셜발(밈·레딧). 공시·뉴스엔 안 뜸. → ② 뉴스 옆에 붙는 추가 신호원.
어떻게 — Reddit 언급량·감성 → "급등+소셜폭증+근거없음=펌프 의심"으로 거르는 용도. 노이즈 커서 보조신호로만.
필요 API — Reddit API(PRAW, 무료) · (X는 유료라 비추)

③ 규칙+ML 기술(Technical) 분석

왜 필요한가 — 공시·뉴스·재무가 "이 회사 좋냐(무엇을 살까)"라면, 기술적 분석은 "지금이 살/팔 타이밍이냐(언제 살까)"를 본다. 아무리 좋은 회사도 비싼 타이밍에 사면 물린다. 또 이유 없는 급등락(공시·뉴스론 안 잡힘)은 오직 가격·거래량으로만 포착된다.

ML은 어떻게 엮이나 (학습한 게 어디로 흐르나) — 위 지표들을 feature(입력)로 LightGBM에 넣어 "상승/보합/하락 확률"을 추정(6장 ML 모듈에서 학습·검증). 규칙(지표 계산)=1차 신호, ML=그걸 확률로 종합(둘 다 LLM 아님). 흐름: ③에서 나온 ML 확률 → ⑤ Strategist 종합 + ⑦ Portfolio Manager의 기대값(EV) 계산에 들어간다.
입력 → 출력 예시 
입력: NVDA 캔들(가격+거래량) 60일치
출력: { 추세:"상승(골든크로스)", RSI:34(과매도 반등권), 거래량:2.1배, ML_상승확률:58% }
      → Strategist에게 "기술 신호"로 전달
단기 vs 장기 — 공격형·단기는 분/일봉 + 모멘텀 지표(RSI·MACD·거래량) 위주, 안정형·장기는 주/월봉 + 장기 추세(장기 이평 우상향) 위주로 본다.
필요 API — 캔들: 토스 / yfinance(키 불필요) · 지표 계산 라이브러리 pandas-ta(무료)

④ 에이전트 Market Regime / 매크로

왜 필요한가 — 개별 종목이 아무리 좋은 호재여도, 시장 전체가 빠지는 날엔 같이 빠진다(종목들이 시장과 함께 움직이는 상관성 때문). 그래서 "지금이 사도 되는 분위기냐"는 시장 전체 국면을 먼저 봐야 한다. 매크로 = 개별 회사가 아니라 경제·시장 전체를 보는 눈.
🍳 쉽게 말하면 — 매크로는 날씨예보다. 공시·뉴스가 "이 가게 장사 잘 되나"라면, 매크로는 "오늘 폭풍이냐 맑냐". 폭풍(risk-off)이면 아무리 좋은 가게도 손님이 없다 → 외출(매수) 자제. risk-on=투자자들이 주식 적극 사는 낙관 분위기, risk-off=현금·안전자산으로 도망가는 공포 분위기.
어떤 지표를 보나 (각각 뜻)
지표뜻해석
지수 (나스닥·S&P500)시장 전체 방향오르면 분위기↑, 급락이면 위험
VIX (공포지수)시장 변동성·공포 수준높을수록(20~30↑) 불안 = risk-off
금리 (미 국채·연준)돈의 값금리↑=주식 불리(자금이 채권으로), 인하 기대=호재
환율 (USD/KRW)달러 강세/약세달러 급강세=글로벌 위험회피 신호 + 우리 환차손익에도 영향
어떻게 동작하나 — 위 지표를 수집(데이터, 학습 아님) → 종합해서 "지금 risk-on이냐 off냐 + 위험도 점수"를 판정 → 매니저/Strategist에 전달.
입력 → 출력 예시 
입력: { 나스닥 -1.8%, VIX 28, 미10년물 금리 급등, 달러 강세 }
출력: { regime:"risk-off", risk_score:7, reason:"VIX 급등+지수 하락+금리 상승 = 위험회피 국면, 신규 진입 자제" }
매매에 어떻게 반영되나 — risk_score가 높으면(risk-off) 신규 매수 줄이고 현금 비중↑, 공격형도 보수적으로. risk-on이면 평소대로. 즉 개별 종목 신호 위에 곱해지는 "시장 전체 필터".
🚀 고도화 (advanced) — 기본은 "4지표 → LLM 한 줄"이지만 다음으로 정교화 가능:
  • ① 지표 확장 — 시장 폭(상승/하락 종목 비율·200일선 위 비율), 신용 스프레드(하이일드 금리차=위험선호 직접 측정), 장단기 금리차(yield curve=침체 신호), DXY·유가·금까지
  • ② 정량 regime 점수 — LLM 한 줄에만 의존하지 말고 지표를 가중합한 객관적 risk score(0~100)를 코드로 산출(예: VIX 30% + 지수vs200일선 30% + 신용스프레드 25% + 금리추세 15%). LLM은 보조 해석 → 재현·검증 가능
  • ③ 국면 세분화 — risk-on/off 2단계 → 강세추세 / 약세추세 / 횡보 / 급락 4~5국면. 국면별 다른 전략(횡보=그리드, 급락=현금)
  • ④ 정책 마스터 스위치 ⭐ — regime이 단순 참고가 아니라 시스템 파라미터를 동적 조정(아래 표). 제일 임팩트 큼
  • ⑤ 전환 조기경보 — VIX 급등·시장폭 급락 등 "국면 전환 신호"를 별도 감지해 전환 확정 전 선제 비중 축소
  • ⑥ (고급·2차) HMM/클러스터링 — 과거 데이터로 국면을 통계적 자동 분류(Hidden Markov Model=regime detection 정석), 다중 시간대(단기/중기) 분리. 여유되면
→ ①②④⑤는 MVP에서도 가능, ③⑥은 2차.
④ 정책 마스터 스위치 예시 — 국면이 유형별 정책 위에 한 번 더 곱해진다:
국면신규 진입종목당 한도손절현금 비중
강세 (risk-on)정상유형 기본유형 기본유형 기본
횡보선별적−20%타이트↑
약세·급락 (risk-off)중단−50%매우 타이트강제 ↑
왜 데이터+LLM? — 지표 수집·계산은 코드(데이터). LLM은 여러 지표를 묶어 "한 줄 국면 해석"만 담당(가벼움). 사실 규칙(예: VIX>30이면 위험)으로도 상당 부분 가능 — LLM은 맥락 종합용.
필요 API — yfinance(지수·VIX·환율·금리, 키 불필요·무료) · LLM(요약용, 가벼움)

⑤ 에이전트 Strategist (시그널 제안)

왜 필요한가 — 분석가 4명(공시·뉴스·기술·매크로)이 각자 신호를 내지만 서로 엇갈릴 수 있다(공시는 호재인데 기술은 약세, 매크로는 risk-off…). 누군가 이걸 종합해 "그래서 이 종목 살까/팔까"를 제안해야 한다 — 그게 Strategist. 펀드의 "전략가/시니어 애널리스트" 역할.
🍳 쉽게 말하면 — 애널리스트 4명이 각자 리포트를 올리면, Strategist는 그걸 다 읽고 "제 의견은 — NVDA 매수, 확신도 7" 처럼 종합 판단을 내는 팀 리더. 단 최종 결재는 아님 — 뒤에서 Risk Critic이 반박하고, Portfolio Manager·리스크 게이트가 거른다(과신 방지).
입력 — ① 4개 분석 신호 ② 투자유형 정책(어디에 가중치) ③ 과거 회고(메모리)("지난번 비슷한 상황에서 이렇게 했다 망함")
어떻게 종합하나
  • 유형별 가중치 — 안정형은 공시·재무 가중, 공격형은 뉴스·테마·모멘텀 가중(9장 "신호 가중치"와 연결)
  • 엇갈림 조정 — 예: 뉴스 호재 + 기술 약세 → 신뢰뉴스+거래량이 받쳐주면 매수, 아니면 보류
  • 거장 페르소나 연결 — 안정형이면 버핏·그레이엄 관점, 공격형이면 우드·린치 관점으로 평가
  • 메모리 참고 — 과거 비슷한 상황의 교훈을 반영해 같은 실수 반복 줄임
출력 필드 — {candidate, side, conviction, rationale}
  • candidate — 종목 (예: NVDA)
  • side — 매수 / 매도
  • conviction — 확신도(0~10) → 나중에 포지션 크기·EV 계산에 쓰임
  • rationale — 근거 → 결정 저널에 기록("왜 샀나")
입력 → 출력 예시 
입력: NVDA — 공시 중립6 · 뉴스 호재7 · 기술 모멘텀↑(상승확률 58%) · 매크로 risk-on / 유형=공격형
출력: { candidate:"NVDA", side:"매수", conviction:7,
        rationale:"뉴스 호재 + 기술 모멘텀 + risk-on 정렬, 공격형 가중" }
다음 단계와의 관계 — Strategist는 "제안"만 한다. 그 제안을 → Risk Critic이 반박(근거 약하면 기각) → 통과하면 Portfolio Manager가 수량·비중 계산 → 리스크 게이트가 한도·손절 검증. 결정권자가 아니라 제안자(혼자 폭주 못 하게).
왜 강한 모델(Claude)? — 4개 신호를 종합하고 엇갈림을 조정하는 가장 어려운 판단이라서. 분석가들은 싼 모델, 여기는 강한 모델(혼합 전략).
필요 API — LLM(강한 모델 = Claude 권장)

에이전트 ⑤+ (선택) 투자 거장 페르소나 — Strategist 옆 결정 보조

왜 — 관점 다양성 + 발표 임팩트. 투자유형 연결(안정형=버핏·그레이엄 / 공격형=우드·린치). → ⑤ Strategist 옆에서 거장 관점을 더해줌.
어떻게 — 같은 종목을 거장별 system_prompt로 평가 → 투표/가중 합의. 단 후보당 거장 전원이 아니라 유형에 맞는 최대 2명만 매칭(안정형→버핏·그레이엄 / 공격형→우드·린치). 2명이 엇갈리면 매니저가 종합.
💸 비용과 절충 — 거장 전원(5명)을 모든 후보에 돌리면 30종목×5 = 150호출/사이클로 비싸다(LLM은 호출당 과금). 그래서 후보당 유형 매칭 거장 2명만 → 30×2 = 60호출(약 60% 절감), 관점 다양성은 유지. 더 줄이려면 확신 높은 후보(top N)에만 페르소나(예: 10×2=20호출) + 싼 모델.
필요 API — LLM 호출 多 → 비용 주의

⑥ 에이전트 Risk Critic (적대적 반박)

왜 필요한가 — Strategist(⑤)가 "사자"고 제안하면, 누군가 반대편에서 따져야 과신·헛매매를 막는다. 단일 시그널 맹신은 깡통의 지름길. Risk Critic은 "이거 진짜 맞아?"를 적대적으로 검증해 약한 시그널을 걸러낸다(손실↓).
🍳 쉽게 말하면 — 악마의 변호인(깐깐한 선배). 후배(Strategist)가 "이 종목 좋아요!" 하면 "근거 있어? 이미 다 오른 거 아냐? 펌프 아냐? 시장 빠지는 중인데?" 하고 깐깐하게 되묻는 역할.
무엇을 반박하나
  • 근거가 충분한가 (공시·가격이 받쳐주나, 뉴스뿐이면 의심)
  • 이미 급등해서 늦은 거 아닌가 (고점 매수?)
  • 펌프·소셜발 급등 아닌가
  • 이벤트(어닝) 임박 / 시장 risk-off 아닌가
어떻게 — Strategist 시그널을 "반박해보라"는 프롬프트로 LLM 실행 → {agree, objection, confidence}. 반박을 못 이기면(근거 탄탄) 통과, 무너지면 기각. 통과한 것만 PM으로.
입력 → 출력 예시 
입력: Strategist "{NVDA 매수, 확신 7, 뉴스 호재}"
출력: { agree:false, objection:"호재 뉴스뿐 + 이미 +12% 급등 = 고점 추격 위험", confidence:0.7 } → 기각
🔁 Strategist↔Critic 무한루프 방지 — 현재는 단방향 1패스(Strategist→Critic→PM). Critic은 통과/기각만 하고 되돌려보내지 않으니 루프가 없다. 만약 "기각 시 Strategist가 수정·재제출 → Critic 재검토" 토론을 넣는다면 → 최대 라운드 수 제한(예: 1~2회) + 합의 안 되면 보수적 기본값(매매 안 함)으로 강제 종료 + 비용·시간 cap. 매 사이클 같은 종목 반복 기각은 아래 "중복 방지"로 거른다.
♻️ "같은 시그널 계속 들어오면?" (중복 방지 4겹) — 루프가 아니라 "매 사이클 같은 종목 재평가" 낭비 문제(재평가 자체는 정상). ① 쿨다운: 기각·손절된 종목은 N일/N사이클 재제안 금지 타이머. ② 메모리 피드백: Strategist가 "최근 기각·손절 이력"을 입력으로 받아 안 내거나 약하게(⑩→⑤). ③ 변화 트리거 ⭐: 새 정보(공시·뉴스·가격 변동) 없으면 재평가 스킵("상황 안 바뀌면 결론도 같다"→ LLM 호출 안 함=비용↓). ④ 기각 캐시: 기각 이유 저장 → 동일 조건이면 LLM 없이 즉시 스킵. → 재평가는 상태가 바뀔 때만.
필요 API — LLM

⑦ 코드 Portfolio Manager (비중·수량·EV)

왜 필요한가 — 통과한 "이거 사자"는 아직 방향일 뿐. 실제로 "몇 주·얼마어치"를 정해야 주문이 된다. 확신·확률을 돈(수량)으로 옮기는 계산 단계. LLM 아님 — 수식·코드(돈 계산은 정확·재현돼야 하니까).
🍳 쉽게 말하면 — 회계·집행 담당. "안정형이니 총자산의 1%만 리스크, 손절 -4%니까 → 2.5만원어치 사세요" 하고 정확한 수량을 뽑는 사람.
균등이냐 차등이냐? → 차등 배분 — 20만원을 2만원씩 똑같이(균등) 나누는 게 아니라, 좋은 종목에 더, 약한 종목에 덜 넣는다(conviction-weighted). 이게 포트폴리오 매니저의 핵심. 배분 4단계:
  1. 신호 강도(확신·EV·ML 상승확률)가 클수록 기본 크게
  2. 변동성 조정 — 출렁임 큰 종목은 같은 확신이라도 줄임(손절폭이 넓어 수량↓ = 리스크 균등화)
  3. 상한 캡 — 종목당 최대비중·동시보유 수(투자유형)를 넘으면 잘라냄
  4. 상관관계 분산 — 비슷한 종목(예: 반도체 3개)이 몰리면 합산해서 캡(쏠림 방지)
배분 예시 (총 20만원, 후보 4개) — 유형이 "차등 정도"를 가른다 
공격형(확신에 집중): A(확신9) 8만 · B(7) 5만 · C(6) 4만 · D(5) 3만   → 차이 큼
안정형(분산 우선):   A 4만 · B 3.5만 · C 3만 · D 2.5만 + 현금 多     → 고르게·작게
※ 안정형은 종목당 5% 상한이라 몰빵 못 함 → 자연히 균등에 가깝고,
   공격형은 25~30% 허용이라 확신 종목에 집중된다.
어떻게 (공식)
  • 포지션 크기 = 거래당 리스크 한도 ÷ 손절폭 (예: 총자산 1% 리스크, 손절 -4%)
  • 기대값(EV) = 상승확률×기대수익 − 하락확률×기대손실 → EV가 +일 때만 산다
  • 투자유형 비중 상한·동시보유 한도 안에서 수량 확정 (미국이라 소수점 가능)
입력 → 출력 예시 
입력: {NVDA 매수, 확신 7, ML 상승확률 58%, 손절 -4%} / 안정형, 총자산 10만원
계산: EV +0.7%(양수) → 진입. 리스크 1%(1,000원) ÷ 손절 4% ≈ 2.5만원어치
출력: { 매수, NVDA, $18.x(소수점), 손절 -4% }

데이터+규칙 ⑧앞 (선택) 이벤트 회피 — 진입 차단 필터

왜 필요한가 — "이벤트" = 실적발표(어닝)·FOMC(미국 금리 결정)·CPI(물가) 같은 예정된 큰 일정. 발표 직전엔 결과를 아무도 모르고, 발표 순간 ±10~20% 갭(점프)으로 튄다 — 좋으면 급등, 나쁘면 급락. 예측 불가 = 도박. 어닝 쇼크로 하루 -20%도 흔하다.
🍳 쉽게 말하면 — 룰렛이 돌아가는 동안엔 새 칩을 안 건다. 결과(발표)가 나오고 판세를 보고 베팅. "모르면 베팅 안 한다"의 일정 버전.
어떻게 — 어닝 캘린더(어느 종목이 언제 실적발표) + 경제 캘린더(FOMC·CPI 일정)를 받아 → 이벤트 임박(예: 1~2일 내) 종목·시장은 신규진입 금지 + 보유 비중 축소. 발표 지나 결과 확인되면 그때 다시 판단. → ⑧ 리스크 게이트 직전에서 진입을 막는 역할.
🔗 뉴스 카드의 "buy the rumor, sell the news"와 모순 아니다 — 이벤트는 두 종류:
① 예정 발표(어닝·FOMC) = 날짜는 알지만 결과를 몰라 도박 → 직전 회피 (이 카드).
② 예정 안 된 루머→공시(M&A 등) = 루머에 이미 오른 걸 공시 확정 때 추격하면 고점 (② 뉴스 카드).
→ 공통 교훈은 같다: "발표·확정 시점에 추격 진입 말고, 지나서 결과·가격 확정되면 판단." 어닝도 기대감에 미리 올랐다 발표 후 차익실현(sell the news) 잦아서 — 이벤트 회피가 곧 그 함정 회피다.
필요 API — Finnhub 어닝/경제 캘린더(가입) / yfinance 어닝(키 불필요)

⑧ 규칙 리스크 게이트·엔진

왜 필요한가 — 예측은 틀릴 수 있다(보장 못 함). 그래서 "틀려도 작게 잃게" 막는 마지막 방어선이 필요하다. AI가 "사자"고 해도 한도·안전 규칙을 어기면 이 게이트가 거부권을 행사한다. 여기서 진짜 돈을 지킨다.
🍳 쉽게 말하면 — 최종 안전 검문소. 위(에이전트들)에서 "통과!"라 해도 규칙 위반이면 막는 경비원. 지능(LLM)이 아니라 무조건 지키는 규칙이라 폭주를 못 한다.
무슨 규칙 (전부 코드)
  • 트레일링 스탑 = 고점 대비 N% 하락 시 자동 청산(상승 따라 손절선도 올라감)
  • 분할매수/매도 · 분할익절 = 한 번에 안 사고 나눠 진입/청산
  • Protections = 연속손실·낙폭(MDD) 초과 시 자동 거래중단·쿨다운
  • 한도 검사 = 종목당 비중·동시보유 종목수·일일 손실 한도 초과 시 거부/축소
  • 하드룰 = 상폐·거래정지·관리종목 → 매수금지·강제청산 (LLM 판단 전에)
입력 → 출력 예시 
입력: {NVDA 2.5만원 매수} / 현재 NVDA 비중 이미 4% (안정형 상한 5%)
출력: 통과(5% 이내) → 실행. 만약 상한 초과면 → 거부 또는 수량 축소
필요 API — 토스 /warnings(거래정지·관리종목)

⑨ 코드 실행 / 백테스트 / 알림 / 대시보드

왜 필요한가 — 게이트를 통과한 결정을 실제로 주문 넣고, 기록하고, 검증하고, 사람에게 보여주는 마무리 묶음. 전부 코드(LLM·학습 아님).
🍳 쉽게 말하면 — 트레이더 + 서기 + 회계사. 주문 넣고(트레이더), "왜 샀는지" 적고(서기), 과거로 검증하고(회계사), 화면에 보여준다.
① 실행 — 결정 → 토스 주문 호출 + 체결·근거 DB 기록. 개발 땐 PaperBroker(가상), 발표 7일 전 LiveBroker(실제)로 스위치.
② 백테스트 — 과거 캔들 재생 + walk-forward + 수수료·세금·슬리피지 반영(현실적). 데이터: yfinance(키 불필요).
③ 알림 — 매매·손익을 텔레그램 봇으로 통지(무료, 휴대폰 푸시). 안 봐도 파악.
④ 대시보드 — "왜 샀/팔았나" 결정 저널 + 수익률 곡선. 웹(Next.js)=1차 / 앱=2차.

⑩ 에이전트 Reviewer (회고) → 페르소나 메모리

왜 필요한가 — 매매하고 끝이 아니라 "그 판단이 맞았나"를 나중에 복기해 같은 실수를 안 하게 한다. 이게 있어야 자율운용이 시간이 갈수록 똑똑해진다(회고 없으면 같은 실수 반복).
🍳 쉽게 말하면 — 오답노트. "지난번 뉴스만 보고 추격했다 -8% 물렸지 → 다음엔 거래량 안 받쳐주면 패스" 같은 교훈을 쌓아 다음 판단(Strategist)에 참고.
언제 회고하나? (매도 때만 아님) — 매도 시점에만 하면 "팔고 나서 오른 것"을 못 잡는다. 그래서 두 시점 + 주기 배치:
  • 진입 후 N일(예: 5/10일) — "그때 산 게 맞았나"를 그 시점 가격으로 채점 (보유 중이든 이미 팔았든)
  • 청산(매도) 후 N일 — "그때 판 게 맞았나" 채점 ← "팔았는데 올랐다"를 잡는 부분
  • 주기 배치(일/주 1회) — 그동안 닫힌·열린 포지션 묶어서 검토
채점은 코드, 교훈은 LLM — 케이스가 많으니 매 건 LLM 안 돌린다. ① 코드가 자동 채점·집계(실현손익·시장대비 알파·손절 후 반등 여부·진입 타이밍) → ② 두드러진 패턴(반복 실패)만 LLM이 교훈화 → 페르소나 메모리(마크다운)에 저장 → Strategist가 비슷한 상황에서 참고. (벡터DB 없이 종목·최근성 검색)
⚠️ "팔았는데 오르면?" — 결과론 편향 조심 — 손절 후 반등했다고 그 한 번으로 손절 규칙을 바꾸면 안 된다(손절은 리스크 관리상 옳은 행동일 수 있고, 한 케이스는 운=생존편향). 여러 번 반복되는 패턴일 때만(예: "손절 후 반등이 70%") → "손절선이 너무 타이트한가?"를 통계로 검토. 한 케이스 ≠ 규칙 변경.
🎯 회고의 핵심 — "한 케이스로 규칙 바꾸지 말고, 패턴(통계)으로 배워라." 이게 회고의 함정(결과론·생존편향)을 피하는 열쇠.
입력 → 출력 예시 
[진입 회고] "NVDA 뉴스 추격 매수 → 5일 뒤 -8% (시장 +1%)"
  교훈: "고변동주, 뉴스 단독 + 이미 급등 시 추격 금물."
[매도 회고] "손절 후 3일간 +6% 반등 — 이번 분기 7회 중 5회 반등"
  교훈: "안정형 손절 -3%가 변동성 대비 타이트할 수 있음 → 백테스트 검토(한 건 아닌 패턴)."
필요 API — LLM

06 ML 예측 모듈 — "통계 외우기 기계"

기술 분석의 확률 추정 담당. "지금 이 종목, 며칠 뒤 오를 확률 / 제자리 확률 / 내릴 확률"을 숫자로 뱉는다. 딥러닝·점쟁이 아님 — LightGBM 권장.

🍳 쉽게 말하면 — ML 모듈 = "과거에 이런 모양이었을 땐 며칠 뒤 보통 이렇게 됐다"를 수만 번 외운 기계. 미래를 보는 게 아니라 과거 통계를 외워서 비슷한 상황을 찾아주는 것뿐이다. 그래서 결과도 "오른다!"가 아니라 "이런 모양이었던 과거 케이스 중 58%가 올랐다"는 확률로 나온다.

1️⃣ 먼저 — "학습 데이터"가 뭔지 (제일 헷갈리는 부분)

ML은 예시를 잔뜩 보고 패턴을 외우는 방식이다. 그럼 "예시"가 필요한데, 예시 한 줄은 이렇게 생겼다:

[입력]  오늘 이 종목 상태 (RSI 70, 20일선 위, 거래량 폭증 …)
   │
   ▼
[정답]  → 그래서 "5일 뒤" 실제로 +3% 올랐다  =  "상승"

핵심은 이 예시를 사오는 게 아니라 과거 주가로 직접 만든다는 것:

  • 애플 과거 5년치 일봉을 공짜로 받는다 (yfinance)
  • 2021-03-10 시점의 지표들을 계산 → 입력
  • 그날로부터 5일 뒤 주가를 보니 +3% → 정답 = 상승
  • 이걸 모든 종목 × 모든 날짜에 반복 → 예시 수만 개 자동 생성
💡 "이 상황이면 오른다"는 완성된 정답표는 세상에 안 판다(있으면 만든 사람이 부자겠지). 그래서 공짜 주가(OHLCV)로 우리가 직접 만든다. 사야 하는 데이터는 없다. (안정성 원하면 Finnhub/Polygon 가입 — yfinance는 비공식이라 가끔 깨짐.)

2️⃣ 단어 딱 두 개만 — feature(입력) · label(정답)

용어뜻우리 예시
feature (입력)판단 재료. 종목의 "현재 모양"을 숫자로RSI·이동평균·MACD·볼린저·ATR·거래량 (+ 감성·매크로 추가 가능)
label (정답)맞춰야 할 답. N일 뒤 결과상승 / 보합 / 하락

"보합"이 왜 끼냐? → 주가는 오르거나 내리는 것만 있는 게 아니라 그냥 제자리(±2% 안)인 경우가 제일 많다. 그래서 답을 3개로 나눈다:

  • N일 뒤 +2% 초과 → 상승
  • N일 뒤 −2% 미만 → 하락
  • 그 사이 → 보합 (제자리) = "의미있게 안 오르고 안 빠짐" = "건드리지 마라" 신호
🔧 개발자 관점 — feature는 종목×날짜마다 벡터 하나. 지표는 pandas-ta로 계산. 임계값(±2%)·N(며칠 뒤)은 튜닝 파라미터.

3️⃣ 모델이 뱉는 건 → "확률 3개"

학습이 끝난 기계에 지금 종목 상태를 넣으면, 점치는 게 아니라 확률 분포를 준다:

[ 하락 20%,  보합 22%,  상승 58% ]     ← 셋을 더하면 항상 100%

predict_proba → [0.20, 0.22, 0.58]
"ML_상승확률 = 0.58"  =  "이런 모양이었던 과거 케이스 중 58%가 올랐다"

이게 softmax / predict_proba라는 말의 정체 — 그냥 "합쳐서 100%인 확률 3개"다. "58% 확신"이 아니라 "과거 비슷한 케이스의 58%가 올랐다"는 뜻인 게 핵심.

⚠️ 단기 예측에서 "상승 90%"가 나오면 기뻐할 게 아니라 의심해야 한다. 현실은 50%대 근처에서 살짝 기우는 정도가 정상. 90%면 "과거를 통째로 외워버린(과적합)" 고장 신호다.

4️⃣ 이 확률을 어디 쓰나 (← 진짜 의도)

ML 확률 하나로 매수하지 않는다. "재료"로 두 군데에 흘러간다:

(a) ⑤ Strategist — 공시·뉴스·기술(ML)·매크로 신호를 다 모아 판단할 때, ML은 그중 한 표.
(b) ⑦ Portfolio Manager — 확률을 기대값(EV)이라는 "돈 계산"으로 바꾼다: 
EV = 오를확률 × 먹을이익  −  내릴확률 × 볼손해
   = 0.58 × (+5%)        −  0.20 × (−4%)
   = +2.1%   → 양수니까 살 만하다. 확률 높을수록 더 크게 베팅.

(보합 확률이 높으면 → "애매" → 진입 안 함)
✅ ML의 진짜 역할은 "맞히기"보다 "애매한 건 거르고 EV 플러스인 것만 통과시키는 필터".
🔧 개발자 관점 · 검증 — walk-forward(과거로 학습 → 그보다 미래 구간으로 테스트, 시간순) + look-ahead 차단(미래 정보가 feature에 새는 것 방지) + 캘리브레이션(확률이 실제 적중률과 맞는지 점검).

왜 LLM 파인튜닝으로 "가격 예측"을 안 하나

🍳 쉽게 말하면 — 파인튜닝 = "이미 똑똑한 AI(GPT 같은 거)에 우리 데이터를 더 먹여 우리 입맛에 맞게 길들이는 것". 가능은 한데, LLM은 "글 쓰는 기계"라 숫자·가격 맞히기엔 구조적으로 약하다. 숫자는 숫자 전문 기계(LightGBM)한테 맡기는 게 맞다.

표로 비교하면:

기준LLM 파인튜닝ML (LightGBM)
수치·시계열 예측❌ 텍스트 생성기라 구조적으로 약함✅ 표·시계열에 최적
확률 신뢰성❌ "55%"가 진짜 55%란 보장 X (캘리브레이션 어려움)✅ predict_proba + 캘리브레이션 검증
데이터·비용❌ 수천~수만 고품질 예시 + GPU 비용✅ 가볍고 무료, CPU로 OK
검증·재현❌ walk-forward·재현 어려움✅ 엄밀히 가능
4주 적합성❌ 과함✅
⚠️ 결론: 가격·타이밍 = ML(LightGBM), LLM은 텍스트 해석 전용. LLM 파인튜닝은 가성비·신뢰성 모두 불리.

텍스트 감성 = LLM 전담 (FinBERT 검토 후 제외)

뉴스·공시 감성은 LLM이 전담한다. FinBERT(무료 사전학습)를 직접 테스트했으나 제외:

  • 고확신 오답 — 유상증자·보합 같은 맥락 필요한 걸 틀리면서도 0.9+ 확신을 줌(캘리브레이션 X) → 점수로 거를 수 없음
  • 비용 이점 없음 — 스크리너가 후보를 20~50개로 좁혀 LLM 호출량이 적음 → FinBERT의 유일한 장점(싸다)이 무의미
  • 단순성 — 한 시스템(LLM)이 둘보다 유지·디버깅 쉬움
✅ 결론: 감성은 LLM-only. (테스트: 명확한 호재/악재는 FinBERT도 맞췄으나 — Apple +0.91 / Tesla −0.95 — 미묘한 건 고확신으로 틀려 신뢰 불가)

학습 데이터셋 — 어디서, 어떻게 (자세히)

우리가 "학습"시키는 건 ML(가격 예측, LightGBM)뿐이다. 텍스트 감성은 LLM 프롬프트로 처리(학습 안 함). 그래서 데이터도 둘로 나뉜다:

① 가격 예측 ML — 데이터 직접 생성 (핵심) — "이 상황→오를 확률" 같은 완성 데이터셋은 없다(있으면 다들 부자). 무료 OHLCV로 직접 만든다: 
1) yfinance/Finnhub로 종목별 과거 캔들(OHLCV) 수년치 받기
2) 각 날짜마다 지표 계산(이평·RSI·MACD·ATR·거래량…) = feature(입력)
3) "N일 뒤 수익률"로 라벨 = +2%↑ 상승 / -2%↓ 하락 / 사이 보합
4) 시간순으로 학습/검증 분할 (walk-forward · 미래 누수 금지)
→ 공짜 가격데이터로 (입력,정답)을 직접 만든다. 별도 구매 불필요.
② 텍스트 감성 — 학습 안 함 (LLM) — 감성은 LLM이 프롬프트로 바로 판단하니 학습 데이터가 필요 없다. 단 공개 감성 라벨 데이터는 "검증·평가용"으로 쓸 수 있다(우리 LLM 감성이 얼마나 맞나 채점 / few-shot 예시):
  • HuggingFace — FinancialPhraseBank·FiQA(금융 감성 라벨) → LLM 감성 정확도 검증·예시용
  • Kaggle — "Daily Financial News 6000+ Stocks"(~400만 헤드라인) · "Historical financial news archive"(미국 800+ 기업 12년) → 과거 뉴스를 가격과 결합해 백테스트·ML feature 보강용
✅ 정리: 가격 예측 데이터는 무료 OHLCV로 직접 생성, 텍스트 감성은 LLM(학습 X) — 공개 감성셋은 검증용. 직접 파인튜닝은 4주엔 과함 → LLM(텍스트) + LightGBM(가격)으로 충분. (FinBERT는 검토 후 제외 — 틀린 것도 고확신, 5장 ② 참고)

08 에이전트는 어떻게 작성하나 (PydanticAI)

에이전트 = 모델 + system_prompt + 출력 스키마(+도구). PydanticAI가 출력을 검증된 JSON으로 강제 → 파싱 깨짐 없음.

# 공시 분석 에이전트 예시
from typing import Literal
from pydantic import BaseModel
from pydantic_ai import Agent

class DisclosureSignal(BaseModel):     # ← 출력 스키마(검증)
    sentiment: Literal["호재", "악재", "중립"]
    importance: int                    # 0~10
    reason: str

disclosure_agent = Agent(
    "openai:gpt-4o-mini",              # 분석가=싼 모델 (교체 가능)
    output_type=DisclosureSignal,
    system_prompt="너는 공시 분석가다. SEC 공시를 읽고 호재/악재와 "
                  "중요도(0~10)를 판단하라. 확실하지 않으면 중립.",
)
result = disclosure_agent.run_sync(filing_text)   # → DisclosureSignal(검증됨)
작성 순서 — ① 출력 스키마 → ② system_prompt(역할·규칙) → ③ 필요시 도구 연결 → ④ Strategist가 결과 종합.
💡 모델 분담: 분석가 = 싼 모델(호출 많음), Strategist/결정 = Claude. LLMClient로 추상화해 교체 쉽게.
🍳 쉽게 말하면 — Pydantic = "이 데이터는 반드시 이 모양이어야 해"라고 틀(서식)을 정해두고, 안 맞으면 거부하는 파이썬 도구. PydanticAI = 그 틀을 LLM(AI)한테 강제하는 도구. "AI야, 답은 반드시 이 칸(호재/악재/중립 + 0~10 점수 + 이유)에 맞춰서 줘. 딴소리하면 다시 시켜"라고 못박는 것. 그래서 AI가 제멋대로 줄글로 답해서 코드가 깨지는 사고를 막아준다.

위 코드, 한 줄씩 뜯어보기

코드뜻 (쉽게)
class DisclosureSignal(BaseModel)"답안지 양식"을 정의 — AI가 채워야 할 빈칸 3개
sentiment: Literal["호재","악재","중립"]이 칸엔 이 셋 중 하나만 허용. AI가 "좀 호재?" 같은 딴말 하면 거부
importance: int # 0~10중요도는 정수여야 함. "높음" 같은 글자면 거부
Agent("openai:gpt-4o-mini", …)이 일 시킬 AI 모델 지정 (싼 모델 / 나중에 한 줄로 교체)
output_type=DisclosureSignal"답은 반드시 위 양식으로" — 이 한 줄이 PydanticAI의 핵심
system_prompt="너는 공시 분석가다…"AI에게 역할·규칙을 글로 지시 (캐릭터 부여)
result = …run_sync(filing_text)공시 원문을 넣으면 → 양식대로 검증된 결과가 나옴. 바로 코드에서 result.sentiment로 꺼내 씀
⚠️ 왜 이게 중요하냐 — AI한테 그냥 물어보면 어떤 날은 {"감정":"호재"}, 어떤 날은 "이건 호재로 보입니다~" 처럼 매번 모양이 달라진다. 자동매매는 사람이 안 보고 24시간 도는데, 모양이 틀어지면 코드가 멈추거나 엉뚱한 주문을 낸다. PydanticAI는 "양식 안 맞으면 AI한테 자동으로 다시 시킴" → 뒷단(Strategist·주문)이 안심하고 받아쓴다.

PydanticAI vs LangGraph — 왜 우리는 PydanticAI인가

둘 다 "AI 에이전트 만드는 도구"지만 결이 다르다.

🟢 PydanticAI (우리 선택)🔵 LangGraph
한마디로"AI 한 명한테 일 시키고 답을 양식대로 받기""AI 여러 명·여러 단계를 그래프(흐름도)로 묶어 돌리기"
잘하는 것출력 검증·타입 안정성. 개별 에이전트가 깨지지 않게복잡한 분기·반복·되돌아가기(루프) 흐름 제어
흐름 제어그냥 평범한 파이썬 코드(if/for)로 우리가 직접전용 그래프 API(node/edge/state)를 배워야 함
학습 난이도낮음 — Pydantic 쓰던 감각 그대로높음 — 그래프·상태 개념을 새로 익혀야
4주·초보 섞인 팀✅ 적합⚠️ 오버스펙·러닝커브
🧩 핵심 차이 — LangGraph의 강점은 "복잡한 흐름 제어"인데, 우리 파이프라인은 이미 단방향 1패스다(스크리너 → 분석가 → Strategist → Risk Critic → PM → 게이트 → 실행, 되돌아가는 루프 없음). 이런 직선 흐름은 평범한 파이썬으로 충분해서 LangGraph의 그래프 기능이 거의 안 쓰인다. 반대로 우리가 진짜 필요한 건 "AI 출력이 매번 같은 모양일 것" → 그게 PydanticAI의 본업.
✅ 결론 — 흐름은 우리가 파이썬으로 짜고, "AI 각 명이 양식대로 답하게 만드는 일"만 PydanticAI에 맡긴다. 가볍고, 배우기 쉽고, 자동매매에 제일 무서운 "출력 깨짐"을 정면으로 막아준다. (나중에 흐름이 정말 복잡해지면 그때 LangGraph 도입 검토 — 지금은 YAGNI.)
🔧 개발자 관점 — CrewAI/AutoGen도 후보였으나 둘 다 "협업 오케스트레이션" 프레임워크라 우리 단방향 파이프라인엔 과함. PydanticAI는 프레임워크라기보다 "타입 세이프한 LLM 호출 래퍼"에 가까워 의존성·블랙박스가 적다. 모델 교체는 Agent() 첫 인자 문자열만 바꾸면 됨(LLMClient 추상화로 한 곳에서 관리).

09 투자유형 3개 = 결정층 정책

투자유형은 코드가 아니라 결정층에 적용하는 설정값. (무위험 허들: 예금 ~3% / 국채 ~4.2% 초과가 최소 목표)

유형주식:현금종목당 최대분산손절거래당 리스크주로 보는 것
안정형~40 : 605%넓게(10+)-3~5%1%/거래공시·재무 · 대형·고배당·채권ETF
균형형~65 : 3510%8종목 내외-7%2%/거래고르게 · 대형+중형+ETF
공격형~100 : 020~30%3~6종목-15~20%4%/거래뉴스·테마·모멘텀 · 성장·고변동·나스닥
💵 "주식:현금"이 뭐야? — 시드 중 몇 %를 주식에 넣고 몇 %를 현금·안전자산으로 남기나의 비율(현금도 포지션). 현금을 일부러 남기는 이유: ① 폭락 방어(안정형 40:60이면 시장 −20%여도 내 계좌는 −8%만 노출) ② 기회 실탄(급락 저점에 살 돈) ③ risk-off 대피(매크로 마스터 스위치가 위험할 때 현금 비중↑). 즉 "안 사는 것도 전략" — 항상 풀투자 안 한다.
📊 "분산" 숫자는 강제 개수가 아니라 목표 범위 — 반드시 N개를 채우는 게 아니다. 실제로 강제되는 건 종목당 최대비중(몰빵 방지)이고, 그게 자동으로 "최소 몇 종목"을 만든다(안정형 주식40%÷5%=최소 8종목, 얇게 나누면 10+ / 공격형 100%÷25%≈3~4종목). 좋은 신호가 부족하면 그보다 적게만 사고 나머지는 현금(억지로 안 채움). → 안정형=얇게 많이(한 종목 터져도 타격↓), 공격형=소수 집중(잘되면 크게·터지면 크게).
🌐 risk-off는 이 표에 없다 — 유형별 정책이 아니라 매크로(5장 ④) 공통 레이어라서. 안정·균형·공격 모든 유형 위에 곱해져 시장이 위험할 때 상한·비중을 깎는다(5장 ④ 매크로의 "정책 마스터 스위치" 표 참고).
⏳ 시간축(보유기간)도 유형이 가른다 — 안정형 = 장기·가치(저평가 우량 천천히 모아 수주~수개월 보유 · 가치 선정 + 정기 리밸런싱) / 공격형 = 단기·모멘텀(급등·이벤트 · 수일~수주 회전 · 모멘텀 스크리너 + 트레일링) / 균형형 = 중기·혼합. → 그래서 ⓪ 스크리너 트랙도 유형 따라 달라진다(모멘텀 vs 가치).
⚠️ 미국 소수점 가능 → 소액 분산 유리. 종목 수는 백테스트로 튜닝.
