🔑 ERD 표기 — 🔑 PK(기본키) · 🔗 FK(외래키) · 🕐 시계열(날짜로 쌓임 → TimescaleDB hypertable 후보). 거의 모든 표가 tb_daily_pick(오늘의 50)에서 파생된다.

🗄️ DB 전체 관계도 (ERD) — 테이블끼리 어떻게 연결되나

🍳 흐름도(위)와 다른 그림 — 흐름도는 "일 순서", 이 ERD는 "DB 구조". 박스 = 테이블, 박스 안 = 컬럼·키, 선 = 관계(FK). 중심은 tb_daily_pick — 주1회 tb_universe(2,000)에서 50을 뽑고, 그 50개에 기술·공시·뉴스가 붙고, 시장 국면(tb_macro)을 얹어 tb_strategist_signals→tb_critic_verdict→tb_order→tb_fill, tb_review가 다음 판단으로 되돌아온다(점선 피드백).
📐 관계 기호(까치발) — || 정확히 1개 · o{ 0개 이상(여러 개) · o| 0 또는 1개. 예: tb_universe ||--o{ tb_daily_pick = "유니버스 1개(그 주)에 tb_daily_pick 여러 행". tb_daily_pick ||--|| tb_technical = 1:1(한 종목당 지표 딱 한 줄). 점선(..) = 피드백(물리 FK 아님, 조회로 참고).

🧭 왜 한 DB(Postgres 1개) — 표들이 FK로 촘촘히 엮여(대부분 tb_daily_pick 조인) 조인·트랜잭션·정합성이 필요. 두 DB로 나누면 FK가 깨진다(대용량 tb_candle만 나중에 분리 고려). 시계열(🕐)은 TimescaleDB hypertable 후보. 7~11 컬럼은 초안 — 담당이 확정.

01 1차 스크리너 — 감시할 2,000개 고르기

📥 입력: NASDAQ 상장 전체 ~16,000📤 출력: tb_universe 2,000🔁 주1회🗃️ 데이터: NASDAQ screener API(무료) · yfinance(sector)
⏰ 01~03 배치는 언제 도나 — 시장 공백 10분 (KST 기준)
미국장 세션: 정규장 22:30~05:00 · 애프터마켓 05:00~08:50 · 데이마켓 09:00~16:50 · 프리마켓 17:00~22:30. 애프터마켓 종료(08:50)~데이마켓 시작(09:00) 사이 10분 공백에 하루 1회 배치(01 유니버스는 주1회 · 02 기술분석 · 03 데일리픽)를 돌린다.
왜 이 틈 — 시스템은 24시간 운영이 목표라, 무거운 배치를 거래 세션과 안 겹치는 유일한 공백에 밀어넣는다. ※ 04~06(매크로·공시·뉴스)과 07 Strategist는 이 배치와 별개로 장중 상시 돈다.

흐름

  1. NASDAQ screener API로 미국 상장 전체(~16,000) 스냅샷 수집 (ticker·company_name·market_cap).
  2. 시가총액 내림차순 정렬 → 상위 2,000개만 남긴다. (감시 대상을 감당 가능한 크기로)
  3. 각 종목 sector(11 GICS)를 yfinance로 붙인다 — 국면·섹터 로테이션 파악용(분산 강제 아님).
  4. 결과를 tb_universe에 저장(ticker·company_name·market_cap·sector). 주1회만 갱신하는 정적 단계(매일 아님).
⚠️ 1차는 가격 필터를 하지 않는다 · 페니주 컷은 2차 — 1차는 시총 top 2,000만 자른다. 페니주(종가 < $5)·유동성(거래대금) 컷은 03 2차 공통필터가, 02 기술분석에서 새로 불러온 캔들의 최신 종가로 판정한다(주1회 스냅샷이 아니라 그날 실시간 가격). 그래서 last_price를 tb_universe에 저장하지 않는다 — 어차피 2차에서 신선한 값으로 다시 본다.

02 기술 분석 — 지표 계산 (규칙, LLM 아님)

📥 입력: tb_universe 2,000 티커📤 출력: tb_technical (최종 50개)🔁 매일 · 배치 로드 ~4분🗃️ 데이터: yfinance 캔들(+^GSPC)

흐름 — 매일, tb_universe의 2,000개에 대해

  1. tb_universe(주1회 확정)에서 2,000 티커를 읽는다 — tb_universe는 감시 리스트일 뿐, 안 건드림.
  2. 그 2,000개 캔들(OHLCV 1년치)을 yfinance 배치로 한 번에 로드(~4분) + 지수 ^GSPC 1개. 지표마다 API 호출이 아니라 캔들 한 번 — 지표는 이 캔들로 계산.
  3. 캔들로 모든 규칙 지표를 pandas로 한 번에 계산(선정용 + 판단용 rsi·macd·trend까지 같이 — 캔들이 메모리에 있으니 판단용도 공짜) → 메모리 2,000행 지표 DF(pandas_ta 안 씀, numpy 2.x 이슈).
  4. 이 2,000행에 03 공통필터 → 5버킷 적용 → 오늘의 50.
  5. 그 50행(이미 지표 다 계산돼 있음)만 tb_technical에 저장 — 다시 계산 안 함. 나머지 1,950행은 버림(다음날 새로). ※ ml_probs(2차 ML)만 추론이 비싸 최종 50개에만 — 1차엔 빈값이라 무관.
⏱️ 왜 매일 다시 로드하나 — tb_universe는 주1회(감시 리스트, 잘 안 변함), 캔들·지표는 매일. 공통필터가 오늘 종가로 판정해야 하니, 주1회 낡은 값이 아니라 매일 신선한 캔들이 필요하다.
🧠 2,000행은 저장 안 하는 임시 — 매일 만드는 2,000행 지표 DF는 메모리에서 필터용으로만 쓰고 버린다(DB 테이블 아님). 저장하는 건 필터·버킷 통과한 50개뿐(tb_technical). 원본 캔들도 1차엔 메모리/CSV 캐시, DB 저장 X(확장 시 tb_candle 테이블).
🍳 선정용 vs 판단용 = "계산 시점"이 아니라 "용도" — 둘 다 2,000행에서 같이 계산된다. 앞(rs_20·atr_pct 등)은 종목 고를 때(필터·버킷) 쓰고, 뒤(rsi·macd·trend)는 고른 뒤 "살까?" 판단할 때 전략·critic이 쓴다. "선정 후"는 쓰이는 시점이지 계산이 미뤄진다는 뜻이 아님. ml_probs만 예외(2차 ML, 50개에·1차 빈값).

04 매크로 분석 — 시장 전체 국면(날씨)

📥 입력: 지수·VIX·금리·달러📤 출력: tb_macro (regime·risk_score)🔁 1시간마다🗃️ 데이터: yfinance(키 불필요)

흐름

  1. 시장 지표 수집(데이터 수집, 학습 아님): 지수(나스닥·S&P500) 등락 · VIX(공포지수) · 금리(미 국채) · 달러(환율).
  2. 각 지표를 0~100 sub-score로 환산.
  3. 가중합으로 risk_score(0~100) 계산 — 클수록 위험(risk-off).
  4. risk_score를 구간으로 잘라 regime 라벨을 붙인다.
  5. 1시간마다 갱신 → tb_macro 저장. 개별 종목 아님 — 시장 전체 1행. 07 전략이 이 국면을 얹어 판단.

risk_score 가중식

risk_score = 0.40×VIX + 0.30×지수등락 + 0.15×금리 + 0.15×달러   (0~100)
regime대략 구간뜻매매
risk_on ☀️risk_score ≤ 30낙관 — 주식 적극 매수 분위기평소대로 매수
neutral ⛅30 ~ 70애매 — 방향 없음(혼조)평소~약간 조심
risk_off 🌩️risk_score ≥ 70공포 — 현금·안전자산 도피신규 매수 줄이고 현금↑
🍳 왜 25% 균등이 아니냐 — 지표마다 "위험을 얼마나 직접 말해주냐"가 다르다. VIX가 공포를 가장 직접 보여줘서 가장 무겁게(0.40). 경계값(30/70)은 백테스트로 튜닝.

05 공시 분석 — 50개 종목의 공식 발표

📥 입력: tb_daily_pick 50 티커📤 출력: tb_disclosure🔁 1시간 간격(새 공시만)담당: 정창욱

흐름

  1. 오늘의 50개 각 종목의 최근 공시(8-K·10-Q·실적 등)를 1시간 간격으로 수집. 새 공시만 처리(중복 제거 = accession 번호).
  2. LLM으로 분석 → 호재/악재(sentiment_score)·중요도·위험도·event_type(사건 종류)·한 줄 근거. + 코드가 파산·상폐 등 hard_block 스캔(매수 차단).
  3. 새 공시가 있으면 tb_disclosure에 새 행 append(같은 날 여러 건이면 여러 행 쌓임). 07 전략(Strategist)이 DB에서 JOIN해 최신 행 읽음.
🕐 공시도 "새 공시 뜰 때마다" 쌓인다 (덮어쓰기 ❌ · 뉴스와 동일 구조)
왜 — 1시간 간격으로 가져오니 같은 날 8-K가 오전·오후 여러 건 뜰 수 있다. 덮어쓰면 앞 공시(파산 8-K 등)가 사라지므로 새 공시마다 새 행 append. PK = (ticker, collected_at⏱️). Strategist는 최신 행을 읽는다.
새 공시 없으면 안 씀(accession 번호로 중복 제거) → 빈 행 낭비 없음. 회고가 "언제 그 공시 떴나" 추적 가능.

06 뉴스 분석 — 50개 종목의 실시간 뉴스

왜 여러 행인가 — 뉴스를 5분 간격으로 가져오니 하루 중 계속 새 기사가 뜬다. 그때마다 새 행을 append(덮어쓰기 ❌) → 종목당 하루 0~여러 행. PK = (ticker, collected_at⏱️). Strategist는 판단 시점의 최신 행을 읽는다.
매크로와 같은 점 — 타임스탬프로 시계열 누적 → 회고(문성혁)가 "이 종목 언제 악재 떴나" 추적 가능.
매크로와 다른 점 — 매크로(VIX)는 연속이라 5분마다 무조건 1행이지만, 뉴스는 이벤트(이산적)라 대부분 5분간 새 기사 0건 → 새 기사 없으면 안 쓴다(빈 행 낭비 방지).

07 전략 종합 (Strategist) — 한 종목을 고른다

📥 입력: tb_daily_pick 50 + tb_technical·tb_macro·tb_disclosure·tb_news + 실시간 시세📤 출력: tb_strategist_signals🔁 장중 상시담당: 이은미
🧭 이은미님 설계 반영 — 아래는 이은미님 Strategist 문서(v2.2) 기준. 투자유형 공격형 단일. 판단 = 코드 게이트 + GPT 샌드위치(코드가 앞뒤 문 잠그고, GPT는 가운데서 판단만 — GPT가 우겨도 게이트 못 넘으면 소용없음).

흐름 (이은미 STEP 2~7)

  1. 신호 읽기 — 5개 테이블을 ticker+cycle_id로 매칭해 SELECT. + 실시간 스냅샷 즉석 조회 — 현재가·당일고저·거래량·거래대금·52주 최고/최저를 50개 배치로 한 번에(아래 2층 구조 · 협의 필요).
  2. 전처리(코드) — 🔒hard_block=1이면 즉시 보류(GPT 안 감) → 매수신호 consensus 카운트(0~4) → 교차확인 시 +0.15 → risk-off면 매크로 감점.
  3. 정책(코드) — 공격형 POLICY 문턱 적용(아래 표).
  4. 메모리 — tb_memory_entries에서 그 종목 최근 5개 기록을 GPT 프롬프트에 넣음.
  5. GPT-4o 판단 — 종합 → conviction(0~10) + side(매수/보류). ※ 계산 아니라 판단만.
  6. 검증·전달(코드) — 🔒min_conviction(5.0) 미달이면 매수→보류 강등. 매수만 08 Critic으로(설득 payload 풀 저장) · 보류는 Critic 안 가고 가볍게만 저장(회고용).
📊 POLICY 문턱값 — 공격형(B 표준) · ⚠️신호점수는 0~1인데 conviction·문턱은 0~10(스케일 다름!)
공시 sentiment≥0.60 · importance≥0.25 · 뉴스 sentiment≥0.60 · grade≥0.50 · confirmed≥0.40 · 기술 ml_prob_up≥0.50 · 필요 합의 2/4 · 매크로 risk-off conviction 감점(거부권 폐지) · min_conviction 5.0. ※ 무조건 차단은 hard_block뿐.
🕐 왜 현재 시세는 DB에 없나 — "정보 2층 구조" (팀 공통 질문 · 07 담당 필독)
① 느린 층(설정·맥락) = 기술지표·공시·뉴스·매크로 → 하루 1번 계산해 DB 테이블에 저장. "이 종목이 요즘 좋은 흐름인가"를 판단(며칠~몇 주 스윙 관점이라 초 단위로 안 바뀌어도 됨).
② 빠른 층(실시간 스냅샷) = 현재가 + 장중 계속 변하는 원값(당일고저·거래량·거래대금·52주 최고/최저) → technical(하루 고정)엔 안 넣고, Strategist가 판단 순간에 50개 배치로 즉석 조회. "지금 이 가격·거래량에 들어가도 되나"(진입 타이밍)를 판단. (회고 원하면 이 스냅샷을 판단 기록 evidence에 같이 저장 가능)
왜 ②를 저장 안 하나 — 현재가는 몇 초만 지나도 낡은 값이라, 테이블에 넣는 순간 거짓말이 된다. 그래서 tb_technical.close는 "그날 종가"일 뿐 실시간 현재가가 아니다. 진짜 현재가는 필요한 그 순간에 즉석 조회가 정답.
상시 폴링 아님 — 1차는 스케줄러가 정해진 시각에 깨우고, 그때 후보 현재가를 1회 조회 → 판단 → 지정가·손절·익절을 브래킷 주문으로 걸어두면 이후 감시·체결은 Alpaca 서버가 대신 한다(초 단위 폴링 불필요).
※ 한계 — 지표(RSI·MACD) 자체를 장중 실시간 갱신하려면 일봉이 아니라 분봉/실시간 캔들이 필요(별도 확장). 일 단위 스윙 MVP에선 일봉으로 충분.

① 실시간 스냅샷 — 이은미 문서엔 아직 없음(4신호 DB만 읽음). 07에서 실시간 스냅샷 세트(현재가·당일고저·거래량·거래대금·52주 최고/최저)를 50개 배치로 즉석 조회 추가 필요(장중 변하는 원값은 technical에 안 넣음).
② cross_source_confirmed — 이은미가 뉴스에서 씀(교차확인 +0.15). 창욱님 스키마엔 없음 → 신설 요청 필요(원자료는 있어 산출 가능).
③ 표 이름 통일 완료 — tb_signal → tb_strategist_signals(이은미 이름)로 문서 반영함.
④ 손절 필드 빼기 — sizing_hint의 손절(−5%)은 09(포트폴리오)가 정하므로 이은미 스키마에서 제거 요청.
⑤ 계좌 확인은 09 몫(Strategist는 잔고 안 봄). ⑥ 이은미 ❌필드를 📄(GPT맥락)로 · source_trust+grade 둘 다 쓰기 제안.
⑦ 보류 저장 방식 — 매수=풀 저장(payload 전부) / 보류=가볍게 저장(side·conviction·1줄 사유만, 회고용). 보류를 tb_strategist_signals에 넣을지/방식 이은미와 확정.

08 크리틱 검증 (Risk Critic) — 매수 제안에 적대적 반박

📥 입력: tb_strategist_signals(매수만) + 분석가 신호📤 출력: tb_critic_verdict🔁 매수 제안마다담당: 김미연
🧭 김미연님 설계 반영 — Risk Critic = "악마의 변호인". Strategist 매수 제안을 일부러 적대적으로 반박해 과신·헛매매를 거른다. 구조 = 3계층 하이브리드(코드→LLM→코드). LLM은 OpenAI gpt-4o-mini. 통과 → 09(PM) · 기각 → 11(Reviewer). 구현 완료·테스트 51개 통과.

흐름 (3계층 하이브리드)

  1. Layer 0 · 하드룰(코드) — 명백한 부실은 LLM 없이 즉시 기각: 거래정지·어닝임박·risk-off(위험도≥7)·펌프(소셜+15%↑)·고점추격(+12%↑)·근거불충분.
  2. Layer 1 · LLM 적대적 반박(PydanticAI) — 애매한 것만 5각도로 반박(아래) → agree·objection·confidence 산출.
  3. Layer 2 · 게이트(코드) — LLM을 그대로 안 믿고 임계값으로 통과/기각 확정(공격형: 반박 confidence ≥ 0.70이면 기각).
🎯 반박 각도 5개(Layer 1 LLM) — ①근거 충분성(뉴스뿐? 공시·거래량 받쳐주나) · ②고점 추격(이미 급등 후 뒤늦게?) · ③펌프·소셜발(근거 없이 소셜만 폭증?) · ④이벤트·매크로(어닝/FOMC 임박·risk-off?) · ⑤근거 대비 과신(확신은 높은데 뒷받침 빈약?). 2차 후보: ⑥논리적 모순("호재라며 확신9인데 폭락위험").
🔄 1차 단방향 1패스 · 2차 다회전 토론 — 1차: 하드룰→LLM 1회→게이트, 기각이면 그 종목 그냥 스킵(되돌림 없음 · 단순·저렴). 2차: 기각 시 Strategist에 되돌려 재제안(최대 라운드, 합의 불가면 보수적 기각). + 중복방지 4겹(쿨다운 5사이클·변화트리거·기각캐시·메모리).
🚧 역할 경계 — gics_sector는 맥락 참고만("섹터 테마 편승 아니냐"). "같은 섹터 몰빵" 집중도 캡은 보유현황이 필요해 PM(09) 몫 — Critic은 단일 시그널만 보므로 섹터 집중도로는 거절하지 않는다.

09 리스크·포트폴리오 — 얼마 살까 · 살지 말지 (코드)

📥 입력: 통과한 tb_strategist_signals + 실시간 시세 + tb_account(잔고)📤 출력: 주문 계획 → tb_order🔁 매수 발생 시담당: 김지현(코드)
🧭 여기는 에이전트 아니라 코드 — 결정론적 규칙·계산(차단·비중·손절)만. Strategist(LLM 판단)와 달리 숫자 비교로 굴러간다.

흐름

  1. 🚫 이벤트 회피 체크 — 어닝·FOMC·CPI 임박(1~2일 내)이면 매수 스킵(아래 카드).
  2. 💰 계좌 잔고 확인 — tb_account 현금을 읽어 살 돈 있나 확인(없으면 스킵/축소).
  3. 📈 현재가 즉석 조회 — Alpaca 실시간 호가(07의 "정보 2층 구조"와 동일).
  4. ⚖️ 비중 계산(위험 기반) — "한 번에 계좌의 4%까지만 손실 허용" → 손절폭 크면 적게·작으면 많이 삼. 한 종목 최대 25%.
  5. 🎯 수량·손절·익절 세팅 — 수량 = 배정금액 ÷ 현재가. 손절 −15% + 트레일링(오르면 따라 올림)을 미리 정해 tb_order 주문계획으로.
🚫 이벤트 회피 — 예정된 큰 발표 직전엔 안 산다 (담당 김지현 · 데이터+규칙)
어닝(회사 실적발표)·FOMC(미국 금리결정, 연 8회)·CPI(물가발표, 매달) = 날짜는 알지만 결과는 발표 전까지 아무도 모름 → 발표 순간 ±10~20% 갭으로 튐 = 도박. 그래서 1~2일 내 임박한 종목·시장은 신규매수 차단 + 보유 축소, 발표 지나 결과 확인되면 다시 판단.
왜 Critic(에이전트) 아니고 코드? — Critic은 "이 논리 말 되나" 주관 판단(LLM, 틀릴 수 있음). 이벤트 회피는 "어닝 1일 전인가" 날짜 사실 확인 → 코드가 정확(LLM에 날짜 맡기면 착각·환각 위험).
왜 Critic "뒤"에? — 정석이다 — 안전장치는 코드 하드룰을 맨 마지막에 두는 게 원칙. Critic(LLM)이 실수로 통과시켜도 코드가 "어닝 직전이니 무조건 차단"으로 받쳐준다. 창욱님 hard_block(파산 차단)과 같은 패턴 = 이중 안전망(중복 아님).
🔄 매수는 "판단", 매도는 "미리 걸어둔 자동" (1차 MVP)
매수(진입) = Strategist가 능동 판단("이거 사자"). 매도(청산) = 여기서 살 때 손절가·트레일링을 미리 계산 → 10이 브래킷 주문으로 걸어둠 → Alpaca가 자동 매도(사람·에이전트 개입 없음).
손절 % 권한 = 09(여기) — 이은미 sizing_hint의 −5%는 참고 제안일 뿐, 최종은 09가 −15%로 정한다. (→ 이은미 스키마에서 손절 관련 필드 빼기 협의)
🔜 2차 — 능동 매도(Strategist가 "이제 팔자" 판단)까지 확장. 그땐 side=매도로 청산도 판단이 담당(테이블은 side로 구분, 별도 테이블 아님).
💰 "살 돈 있나"는 여기서 확인 — Strategist 아님 — Strategist는 "좋은 매수인가" 판단만 하고 계좌 잔고는 안 본다. 현금 확인·자금 배분은 09가 tb_account 읽어서 함. 돈 없으면 제안이 와도 09가 스킵/축소. → 역할 분리: Strategist=판단 / 09=자금·리스크.

투자유형 — 결정층 설정값(코드 아님) · 1차 = 공격형만

유형주식:현금종목당 최대분산손절거래당 리스크주로 보는 것
안정형 (2차)~40 : 605%넓게(10+)−3~5%1%/거래공시·재무 · 대형·고배당·채권ETF
공격형 ⭐1차 MVP~100 : 020~30%3~6종목−15~20%4%/거래뉴스·테마·모멘텀 · 성장·고변동·나스닥
🧭 투자유형은 코드가 아니라 config 설정값 — 1차는 공격형 하나(aggressive.yaml)만 쓴다. 안정형은 2차에 conservative.yaml을 같은 형식으로 추가만 하면 됨(코드 수정 X). 균형형은 안 함 — 공격형·안정형 2종만.
📊 공격형 리스크 정책(aggressive.yaml) — 한 종목 최대 25% · 목표 5종목 · 1회 리스크 4% · 손절 −15% + 트레일링 · risk-off면 신규 축소·현금↑.
💡 이 숫자들은 감으로 정한 게 아니라 업계 관용값 — 거래당 리스크 1~5%(공격형이라 4%) · 모멘텀주 손절 넓게(−15%)가 표준이고, "4% ÷ 15% ≈ 25%"로 종목당 비중이 자동으로 딸려나온다(서로 맞물림). 초기값은 관용값으로 시작 → 11 백테스트/회고로 검증·튜닝(감이 아니라 실제 성과가 최종 결정).

10 주문·체결 (Alpaca) — 페이퍼 매매 실행

📥 입력: 주문 계획📤 출력: tb_account·tb_order·tb_fill🔘 설계 예정담당: 김지현(실행)

흐름

  1. Alpaca 페이퍼 계정에 09의 주문계획을 브래킷 주문(진입 + 손절 + 익절/트레일링을 한 세트)으로 제출(전부 가상매매 — 실제 돈 X).
  2. 진입 체결되면 손절·익절은 Alpaca 서버가 자동 감시·실행(초 단위 폴링 불필요). 체결 결과(가격·수량·시각)를 기록.
  3. 계좌 잔고·보유 종목 갱신 → 11 리뷰로.

11 리뷰·회고 — 복기 → 다음 판단으로

📥 입력: Strategist 판단·Critic 검증·체결·계좌 + 이후 가격📤 출력: tb_review · tb_memory_entries🔘 설계 예정담당: 문성혁
🔘 개념 초안 — 담당(문성혁)이 로직·컬럼 확정. 아래는 "무엇을 복기하나" 큰 그림.

흐름 (개념)

  1. 모은다 — 하루치 판단(tb_strategist_signals: 매수/보류/매도, 수백~수천 행) + 검증(tb_critic_verdict: 통과/거절) + 실제 체결·잔고(tb_fill·tb_account: 뭘 얼마에 샀고 계좌가 어떻게 됐나).
  2. 결과를 붙인다(채점) — 각 판단·체결 이후 가격이 어떻게 됐나(전방 수익률 +1/3/5일)를 붙여 옳았는지 판정: 매수는 올랐나 · 보류는 놓쳤나 · 손절은 적절했나.
  3. 교훈을 뽑는다 — 잘된/못된 패턴을 tb_review(리포트·통계) + tb_memory_entries(종목별 메모리)로 압축. 수천 행 → 요약 교훈.
  4. 피드백 루프 — 이 메모리를 다음 사이클 Strategist(07)가 종목별 최근 5개 읽어 프롬프트에 넣음 → 같은 실수 반복↓.
🔎 회고는 셋을 다 이어본다: ① Strategist 판단표(매수/보류/매도) · ② Critic 통과/거절표 · ③ 실제 체결·계좌 잔고. 여기에 "그래서 가격이 어떻게 됐나"(전방 수익률)를 붙여 채점하는 게 회고의 핵심. → "판단(제안)"만이 아니라 "실제로 산 것"과 "결과"까지 이어봐야 진짜 성과를 안다.

테이블담는 것키
tb_review회고 리포트 — 교훈(lesson)·통계(수익률·적중률 등 stats)🔑 id · 🔗 tb_daily_pick
