2 공통 규약

2-1. 종목 식별 (두 Bundle 공통)

필드타입채우는 주체설명
tickerstr정보분석티커(대문자). 예 NVDA
trade_datestr정보분석분석 기준일 YYYY-MM-DD (구 as_of 개명)
collected_atstr정보분석수집 시각(초 포함 ISO). 스냅샷 PK 일부 (구 created_at 개명)
company_name·category는 번들에서 제거
tb_universe에 이미 있어 Strategist가 ticker로 JOIN해 읽는다(중복 제거). 스크리닝 입력 단계에는 여전히 존재하는 필드.

2-2. 점수 규약 float 0.0~1.0

정규화내부 0~10 → /10 → 0~1 (소수 2자리)
sentiment_score방향×강도 0~1. 0.5=중립. 호재만 conf 감쇠·악재 유지(비대칭)
importance강도만·부호 없음 (상폐≈1.0)
risk_score하방·위험 강도
confidence신뢰도
source_trust(뉴스) 출처 신뢰도(LLM)
grade_score(뉴스) 출처 등급 ALLOW1·GRAY.6·BLOCK0
confirmed_score(뉴스 전용) 확정비율. 공시는 늘 1.0이라 제거
bool 플래그has_signal·hard_block → 0/1 정수

2-3. 방향 sentiment_score + sentiment

sentiment_score 0~1 — 0.0=강악재 · 0.5=중립 · 1.0=강호재

positive negative neutral mixed ← 점수에서 파생 병기

역할 분리: score=방향 · importance=강도 · label=가독성

2-4. event_type 온톨로지 (12종)

earnings · guidance_change · ma · capital_raise · buyback · management_change · insider_trade · product_deal · analyst_rating · regulation_legal · delisting_halt · other

공시·뉴스 공유(ontology.py). 과세분화 금지.

3 DisclosureBundle — 공시 결과 공시 18필드

공시 분석 결과 단독. 종목당 1개, 뉴스와 완전 분리. tb_disclosure(18컬럼)와 1:1.

2026-07-06 개편: 24 → 18필드
삭제(6) company_name category confirmed_score fact_check verdict ref · 개명 as_of→trade_date · created_at→collected_at · reason를 sentiment_score 바로 아래로 재배치.
class DisclosureBundle:
    ticker: str
    trade_date: str                 # 분석 기준일 YYYY-MM-DD (구 as_of)
    collected_at: str               # ★ 수집시각(초, 스냅샷 PK 일부, 구 created_at)
    has_signal: int                 # 0/1 (분석된 공시 신호 유무)
    filing_title; filing_no; filed_at # ★ 공시 제목·번호(accession)·업로드일시(초 포함)
    event_type: str                 # 통일 12종
    sentiment: str                  # 방향 라벨(파생)
    sentiment_score: float          # 0~1: 0=악재·0.5=중립·1=호재
    reason: str                     # 한 줄 근거 (sentiment_score 바로 아래로 재배치)
    importance: float               # 0~1 (강도)
    risk_score: float               # 0~1
    confidence: float               # 0~1
    hard_block: int                 # 0/1 상폐·going concern → 매수 override
    hard_block_reason: str | None
    summary: str                    # ★ 공시 전문 3~4줄 요약
    keywords: list[str]             # ★ 핵심 키워드 5개
필드범위의미 · 근거
ticker·trade_datestr종목 식별·기준일. company_name·category는 tb_universe JOIN으로 읽음(번들에서 제거)
collected_atstr수집시각(초 포함 ISO). 스냅샷 PK (ticker, collected_at)의 일부(구 created_at)
filing_title·filing_no·filed_atstr공시 제목·번호(accession)·업로드 일시(초 포함, acceptanceDateTime 우선). filing_no가 원문 역추적 ID라 별도 ref 불필요
sentiment_score0~1방향 점수. 0=강악재·0.5=중립·1=강호재
reasonstr한 줄 근거(수치 포함 권장). sentiment_score 바로 아래로 재배치
importance0~1강도만. 1~5일 스윙 영향. 거래정지·상폐≈1.0, 대형 M&A·실적쇼크 0.8~0.9
risk_score0~1하방 위험. 회계·희석·상폐일수록 ↑
confidence0~1certainty_level 수치화 (High=0.9/Med=0.6/Low=0.3)
hard_block0/1상폐·거래정지·going concern → 매수 override. sentiment에 안 녹이고 별도 플래그
summary·keywordsstr·list★ 전문 3~4줄 요약·키워드 5개. 원문 근거만(환각 방지)
삭제 필드(6) 근거
company_name·category→tb_universe JOIN · confirmed_score(공시는 늘 1.0, 정보량 0) · fact_check(늘 "Verified — official SEC filing" 상수) · verdict(sentiment_score+reason 재진술) · ref(filing_no가 이미 역추적 ID). 뉴스는 값이 가변이라 confirmed_score·fact_check·ref 유지.

4 NewsBundle — 뉴스 결과 뉴스 28필드

뉴스 분석 결과 단독. 공시와 완전 분리. tb_news(28컬럼)와 1:1.

2026-07-06 개편: 31 → 28필드
삭제(3) company_name category verdict · 개명 as_of→trade_date · created_at→collected_at · reason를 sentiment_score 바로 아래로 재배치. source_trust·grade_score·confirmed_score·fact_check·top_evidence·peak_importance·ref는 뉴스 고유값이라 유지.
class NewsBundle:
    ticker: str
    trade_date: str                 # 분석 기준일 YYYY-MM-DD (구 as_of)
    collected_at: str               # ★ 수집시각(초, 스냅샷 PK 일부, 구 created_at)
    has_signal: int                 # 0/1
    news_title; source; published_at # ★ 대표기사 제목·언론사·업로드일시(초 포함)
    news_count: int                 # 분석된 뉴스 총 건수
    news_confirmed: int             # 확정 사실 건수
    news_rumor: int                 # 루머/추측 건수
    event_type: str
    sentiment: str                  # 방향 라벨(파생)
    sentiment_score: float          # 0~1: 0=악재·0.5=중립·1=호재
    reason: str                     # 한 줄 근거 (sentiment_score 바로 아래로 재배치)
    importance: float               # 0~1 (강도, 신뢰가중)
    peak_importance: float          # 0~1 (집계 전 최고 잠재)
    risk_score: float               # 0~1
    confidence: float               # 0~1
    source_trust: float             # 0~1 (LLM 판단)
    grade_score: float              # 0~1 (출처등급 ALLOW1·GRAY.6·BLOCK0)
    confirmed_score: float          # 0~1 (확정 비율)
    fact_check: str                 # ★ 팩트체크 한 문장(Verified/Unverified…)
    hard_block: int                 # 0/1
    hard_block_reason: str | None
    top_evidence: list[str]
    summary: str                    # ★ 뉴스 전문 3~4줄 요약(대표 기사)
    keywords: list[str]             # ★ 핵심 키워드 5개
    ref: str                        # 원문 링크(뉴스는 filing_no가 없어 유지)
필드범위의미 · 근거
ticker·trade_date·collected_atstr종목 식별·기준일·수집시각(초 포함). company_name·category는 tb_universe JOIN(번들에서 제거). collected_at은 스냅샷 PK 일부(구 created_at)
news_count/confirmed/rumorint건수 분해. "확정 여럿=신뢰↑, 한 곳뿐=루머 의심"
news_title·source·published_atstr대표 기사의 제목·출처 언론사·업로드 일시(초 포함)
sentiment_score·reason0~1·str방향 점수(0=악재·0.5=중립·1=호재)와 한 줄 근거. reason은 sentiment_score 바로 아래로 재배치
importance0~1신뢰가중 평균 강도
peak_importance0~1집계 전 최고 잠재. "잠재 큰데 저신뢰라 눌린" 경우 병기
source_trust0~1(뉴스 전용) 출처 신뢰도(LLM). 소셜·루머 방어
grade_score0~1출처 정책등급 점수. ALLOW1·GRAY.6·BLOCK0 (신뢰가중, 소셜=GRAY)
confirmed_score0~1확정 비율 = 확정건수 / 전체. 뉴스는 값이 가변이라 유지
fact_checkstr팩트체크 한 문장(확정→Verified·루머→Unverified (rumor)). Verified/Unverified 가변이라 유지
top_evidencelist대표 근거 헤드라인(나머지는 DB)
summary·keywordsstr·list★ 대표 기사(신뢰가중 선정)의 전문 3~4줄 요약·키워드 5개
refstr원문 링크. 뉴스는 filing_no가 없어 역추적 ID로 유지

5 집계 규칙 — 뉴스 여러 건 → 대표 1개

방향/강도신뢰가중. 단 강한 악재(signed ≤ −0.7)면 보수적으로 그쪽
sentiment_score방향×강도를 0~1로. 호재만 conf 감쇠·악재 유지(비대칭)
importance신뢰가중 평균(방향 뺀 크기, 감쇠 없음)
confidencecertainty × is_confirmed × source_trust 평균
대표 선정importance × confidence(신뢰가중) 최대 기사로 — 루머가 헤드라인 안 되게
hard_block어느 뉴스든 BLOCK류면 1
산출 고도화 근거
① 방향점수는 낙관만 신뢰로 깎고 악재는 유지 — 과신·펌프엔 보수적, 경고는 안 흐림("틀려도 작게 잃기"). ② 대표는 가장 시끄러운 것(importance)이 아니라 가장 믿을 만하며 중요한 것(importance×confidence)을 뽑아 미확정 루머가 헤드라인이 되는 걸 막는다.

7 전달 방식 · 열린 질문

전달 방식 (2026-07-06 개편)

  • 정식 계약 = 공유 DB 스냅샷 — tb_disclosure(18컬럼)·tb_news(28컬럼), PK (ticker, collected_at). 필드=컬럼 1:1.
  • 새 공시/기사마다 새 행 append(덮어쓰기 X, 시계열 누적). Strategist는 최신 행을 읽고 tb_universe를 ticker로 JOIN해 회사명·카테고리 채움.
  • 저장 save_disclosure_bundle/save_news_bundle · 읽기 latest_disclosure/latest_news (app/storage/db.py).
  • 수집 주기 공시=1시간·뉴스=5분(새 것만). 중복제거 키 공시=accession(filing_no)·뉴스=google_link.
  • 보조 = to_prompt()/model_dump_json() — 사람 읽기·프롬프트 삽입용. 정식 계약은 DB 행(또는 그 JSON).
  • 별개 유지 — per-signal 로그 disclosure_signals/news_signals(스케줄러·백테스트)는 tb_*와 별개 테이블.
  • 엔트리 run_bundle.py --ticker NVDA --llm --save → 번들 산출 후 tb_*에 append. (회사명·카테고리는 tb_universe JOIN이라 인자 없음)

  3. category 체계 — GICS 섹터? 자체 분류? 라벨 표준 합의
