Overview

개요 · 파이프라인 위치

전체 11단계 파이프라인 중 05 공시 · 06 뉴스가 우리 파트. 스크리닝이 넘긴 오늘의 50종목에 대해, 각 종목의 공시·뉴스를 구조화 신호로 만들어 07 Strategist에게 넘긴다. 구조화 신호 + 보수적 1차 판단까지만 — 실제 매수권한·시장반응 검증은 Strategist 몫(범위 밖).

Disclosure

공시 파이프라인 — 3축

공시는 Form 종류에 따라 코드/LLM을 차등 라우팅한다. "숫자=코드, LLM=해석만" 원칙을 Form별로 구현. form_type_router가 8-K=Item / 10-Q·K=XBRL / Form4=코드로 분기.

① 8-K Item 라우터

sec_collector가 submissions→Item 청킹 → item_event_map이 Item번호로 1차 분류·기본권한 부여 → 하드룰 → LLM 해석.

예: 3.01 상폐통지=BLOCK_BUY·hard_risk / 2.02 실적 / 5.02 경영진변경.

② 10-Q/K XBRL

sec_xbrl_collector가 companyconcept API로 YoY 수치 계산(코드) → LLM은 수치 해석만. 본문 리스크문구(going concern·material weakness)는 원문 별도 스캔.

③ Form 4 코드 스코어링

sec_form4_collector가 raw XML 파싱 → form4_scorer가 LLM 없이 코드로 스코어(공개매수=호재·매도=약한악재·베스팅/세금/옵션=중립, 직급 가중).

공통 배관 재사용 — 세 축 모두 결과를 DisclosureSignal로 변환해 같은 집계·저장·결정 파이프라인에 흘린다. Form4는 코드 산출이지만 event_type=insider_trade로 동일 배관 사용.

안전장치 — 규칙 기반 확정 차단

공시의 hard_block은 확률적 감점이 아니라 규칙 기반이다. 8-K Item 3.01/4.01(상폐·감사인교체)은 default_permission=BLOCK_BUY라 즉시 차단. 최종 권한은 하드룰/LLM/Item 중 가장 보수적으로 채택.

News

뉴스 파이프라인

Google News RSS로 실수집(키 불필요). 공시와 로직은 분리하되 배관은 공유. LLM 호출 전에 두 겹의 사전필터로 노이즈·저신뢰를 먼저 걸러 비용·오탐을 줄인다.

사전필터①출처등급(BLOCK drop)
사전필터②키워드(추측·낚시 drop)
분석LLM → NewsSignal
집계신뢰가중 → NewsBundle

한 기사 = 한 묶음

RSS <item> 하나가 NormalizedItem으로: 제목·본문(RSS 빈약→헤드라인 중심)·언론사(<source>)·링크 2종(도메인=등급용 / google_link=dedup)·발행시각(pubDate).

Contract

집계 계약 — DisclosureBundle 18 · NewsBundle 28

Strategist에게 넘기는 종목당 스냅샷. 공시·뉴스 완전 별개 2객체로, Strategist가 두 소스를 직접 결합한다(팀 명세 수용, 2026-07-06).

DisclosureBundle 18필드

공시 1건을 1:1 복사(집계 없음 — 원본이 유일). event_type·sentiment_score·importance·risk_score·confidence + 원문식별(filing_title·filing_no·filed_at) + hard_block + summary·keywords.

신뢰/출처 필드 없음 — 출처가 SEC 하나로 고정(항상 최고 신뢰).

NewsBundle 28필드

N개 기사 → 종목 1개로 신뢰가중 집계. 위 공시 필드 + 건수(count/confirmed/rumor) + 신뢰 3축(confidence·source_trust·grade_score) + confirmed_score·fact_check·peak_importance·top_evidence.

대표 기사(신뢰가중 최대)에서 제목·출처·요약 채움.

정식 전달 = JSON(model_dump_json()) · to_prompt()는 사람이 읽는 보조 표현. DB 스냅샷 tb_disclosure/tb_news에 시계열 append, Strategist는 최신 행만 읽음.
Metrics

0~1 산출 지표 — 공식과 의미

모든 점수는 0~1 소수점 통일(2026-07-02 개편). 강도·방향·신뢰 3축으로 분리되며, 뉴스는 기사별 신뢰도 _news_conf로 가중한다.

기사별 신뢰 가중치 (뉴스 집계의 근간)

_news_conf = certainty(High.9/Med.6/Low.3) × 확정계수(확정1.0·루머0.5) × source_trust
# 확정된 High + 신뢰높은 출처일수록 1.0, 저신뢰 루머는 0에 근접
지표축산출의미
importance강도강도의 신뢰가중 평균(방향 무관)사건 크기 · 호·악재 섞여도 상쇄 안 됨
peak_importance강도신뢰검증 기사 중 최댓값 ★묻힌 대형 신호 감지(§10 고도화)
sentiment_score방향방향×강도 가중평균 → 비대칭 감쇠0=강악재·0.5=중립·1=강호재
sentiment방향score에서 파생positive/negative/neutral/mixed
risk_score—위험도 신뢰가중 평균호재/악재와 별개 축
confidence신뢰_news_conf 단순평균루머 많으면 급락(정직)
source_trust신뢰LLM 출처신뢰 가중평균본문 톤+등급, 사후 판단
grade_score신뢰도메인 등급 가중평균(ALLOW1·GRAY.6·BLOCK0)코드·사전, 감사 가능
confirmed_score신뢰확정건수/전체3건 중 2건 → 0.67

방향의 비대칭 감쇠 — "가장 보수적 채택"

호재만 confidence로 감쇠, 악재는 유지. 근거 약한 낙관은 과신·펌프 위험이라 중립(0.5)쪽으로 눌러 덜 단정하고, 악재는 근거가 약해도 흐리면 위험하므로 그대로 살린다. 또 강한 악재(≤ −0.7) 1건이면 가중평균 무시하고 그쪽으로. "틀려도 작게 잃기".
Source Trust

출처 신뢰 정책 · 선정 기준

URL 도메인으로 3등급 판정(코드·LLM 무관·재현성 100%). 미등록은 보수적 기본 GRAY. 등급→점수(grade_score)는 신뢰가중 평균된다.

등급점수부류·선정 근거대표 도메인
ALLOW1.0① 1차 공식출처 ② 편집책임 1급언론 ③ 공식 배포사sec·nasdaq·finra·fda · reuters·bloomberg·wsj·ft·cnbc · businesswire·prnewswire
GRAY0.62차 매체·논평·집계(단독 신뢰 부적합) + 소셜·미등록benzinga·seekingalpha·fool·zacks·tipranks
BLOCK0.0편집책임 없는 유료홍보·수익보장sponsored·paid promotion·원출처 없음
선정 원칙(2026-07-07 문서화) — 외부 신뢰지수를 그대로 쓴 게 아니라 "원본(1차 출처) 근접성 + 편집·정정 책임" 기준의 큐레이션. 한계: 업계 통념 기반이라 외부 감사표준(NewsGuard·MBFC) 미앵커링 → 후속 과제. 단 grade_score는 단일 관문이 아니라 다층 방어의 한 겹. (상세: 결과값_필드_레퍼런스 §7-1)
⚠️ ALLOW ≠ 매수허용 — "분석 반영 가능"일 뿐. 실제 매수는 시장반응·리스크 필터를 추가로 통과(Strategist 몫). grade_score(코드·사전) ≠ source_trust(LLM·사후) — 두 축 분리 노출.
