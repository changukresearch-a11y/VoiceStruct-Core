공시 · 뉴스 파이프라인 플로우 & ERD

두 파이프라인은 산출 테이블 구조가 닮았지만(종목당 1행 시계열 누적) 내부 처리가 근본적으로 다르다. 공시는 Form 종류별 라우팅으로 코드/LLM 비중이 갈리고 공시 1건 → 1행. 뉴스는 출처 신뢰가중 + 대표선정으로 N기사 → 1행. 각각의 흐름과 ERD를 따로 정리한다.

공시 tb_disclosure · 18필드 · 1시간 뉴스 tb_news · 24필드 · 5분 PK 공통 (ticker, collected_at) 담당 정창욱
LLM 분석 코드 집계·파생 대표기사 복사 수집·원문 식별 하드 스캔(차단) 정렬 상위 종목·시각 메타
한 문장

공시 = SEC 공식 문서 1건을 Form 종류에 맞춰 코드/LLM 차등 처리 → 신호 1행. 뉴스 = 신뢰도 제각각인 기사 N건을 출처가중·대표선정으로 종합 → 신호 1행. 그래서 뉴스에만 source_trust·grade_score·peak_importance·top_evidence·news_title 같은 "여럿 중 고르고 믿는" 컬럼이 있다.

05

공시 분석 파이프라인

SEC EDGAR · 1시간 간격
50종목 · 정창욱

app/pipeline.py → run_disclosure_pipeline(). 핵심은 Form 종류가 처리 경로를 결정한다는 것 — 같은 "공시"라도 8-K·10-Q·Form4·S-1이 각기 다른 코드/LLM 비중으로 흐른다.

1

수집 + 정규화 SEC

50종목 최근 공시(8-K·10-Q·10-K·Form4 등)를 1시간 간격 수집. 새 공시만 처리(중복 제거 = accession_no). → NormalizedItem

2

Form 유형 라우팅 route_form()

Form 종류마다 코드/LLM 비중을 다르게 배정한다 — 코드로 잡을 수 있는 건 코드가(정확·저렴), 해석이 필요한 뉘앙스만 LLM이.

Form별 처리 경로

8-K
Item 라우터(코드)가 사건종류 분류 → 해당 Item 본문만 LLM 이벤트 해석
10-Q / 10-K
숫자=XBRL 추출(코드) + 본문 리스크문구 스캔 → LLM은 가이던스 뉘앙스만
Form 4
XML 파싱 100% 코드 스코어링(score_form4). LLM 배제
S-1 / 424B
희석 계산(코드) + 자금용도 LLM 해석
코드 LLM
3

하드룰 스캔 check_hard_risk() · LLM보다 우선

본문에서 파산·상폐·거래정지·going concern(계속기업 불확실성)·material weakness 등 치명 문구를 코드가 검사. 걸리면 LLM 판단과 무관하게 매수 차단 강제. (수치로 안 잡히는 위험을 원문에서)

4

LLM 분석 DisclosureSignal

reasoning(CoT) → event_type·sentiment·importance(0~10)·risk_score(0~10)·certainty_level·reason·summary·keywords. 하드리스크 감지 시 risk_score를 최댓값으로 끌어올림.

5

최종 권한 종합 decide_final_permission()

하드룰 / LLM / Item 기본값 세 제안 중 가장 보수적인 것을 채택(6단계: TRADE_ELIGIBLE … BLOCK_ALL). 안전장치를 맨 뒤에 두는 원칙.

6

집계·저장 build_disclosure_bundle() → tb_disclosure

신호를 18필드 번들로 변환, sentiment_score·confidence를 코드가 파생. 종목당 1행으로 tb_disclosure에 append(PK (ticker, collected_at) · 시계열 누적). 공시 없으면 has_signal=0 빈 행.

왜 뉴스와 달리 "대표선정"이 없나

공시는 SEC 공식 문서라 출처 신뢰도·루머 판별이 불필요하다(항상 사실 1.0). 그래서 공시 파이프라인엔 source_trust·grade_score·peak·top_evidence가 없고, 여러 건이어도 신뢰가중 대표선정 대신 사건별 그대로 집계한다. 대신 Form 종류별 라우팅이 뉴스엔 없는 공시만의 축이다.

06

뉴스 분석 파이프라인

Google News 등 · 5분 간격
50종목 · 정창욱

app/news_pipeline.py → run_news_pipeline() + build_news_bundle(). 핵심은 출처 신뢰도 처리와 대표선정 — 신뢰도가 제각각인 기사 N건을 가중 종합하고, 그중 대표 1건을 골라 종목의 "얼굴"로 삼는다.

1

수집 + 정규화 News

50종목 최근 뉴스를 5분 간격 수집. 새 기사만 처리(중복 제거 = google_link). → NormalizedItem

2

출처 등급 분류 classify_source(url)

도메인으로 ALLOW / GRAY / BLOCK 판정. 미등록(소셜 포함)은 보수적 기본 GRAY. → grade_score의 씨앗

3

키워드 사전필터 keyword_screen() · LLM 전

화이트(fda approval…) > 블랙(rumor·class action·top picks…) > 통과. 출처=BLOCK 또는 키워드=drop이면 LLM 호출 없이 즉시 종료(비용·노이즈 차단).

4

LLM 분석 (기사별) NewsSignal

기사 1건씩 → importance(0~10)·sentiment·is_confirmed·source_trust(0~1)·certainty_level·event_type·reason·summary·keywords. 출처 등급을 입력에 함께 주입.

5

신뢰가중 집계 + 대표선정 build_news_bundle()

weight = certainty × is_confirmed × source_trust → 대표 rep = argmax(importance × weight) · importance·sentiment·risk·source_trust·grade = 신뢰가중 평균 · peak = max(신뢰검증 기사 강도) · top_evidence = 상위 3.

6

하드 스캔 + 최종 권한 hard_block

어느 기사든 파산·상폐류 BLOCK이면 hard_block=1. decide_final_permission()로 최종 권한 종합.

7

저장 → tb_news

종목당 1행으로 tb_news에 append(PK (ticker, collected_at) · 시계열 누적). 뉴스 없으면 has_signal=0 빈 행. Strategist는 종목별 최신 행을 읽음.

뉴스만의 축

3단계(사전필터)·5단계(대표선정·신뢰가중)가 뉴스의 정체성이다. 자세한 대표선정·곱셈가중·peak 논리는 별도 문서 뉴스_대표기사선정·24필드_상세.html 참조.

한 줄 정리

공시는 "믿을 수 있는 하나를 Form에 맞게 잘 읽는" 문제, 뉴스는 "못 믿을 수 있는 여럿에서 믿을 것을 골라 종합하는" 문제. 그래서 공시엔 Form 라우팅이, 뉴스엔 신뢰가중·대표선정이 각각의 심장이다.
