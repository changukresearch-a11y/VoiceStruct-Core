결과값 산출방식 · 의미 · 전략가 활용

각 결과값이 어떤 방식으로 산출되는지, 정확히 무슨 값인지, 비슷해 보이는 항목끼리 뭐가 다른지, 그리고 이 값들이 전략가(Strategist) 에이전트에게 어떤 정보를 주는지를 코드·공식 근거로 설명하는 문서.

출력 0~1 소수 2자리 통일 원천 SEC · RSS · LLM · 정책코드 읽는 이 Strategist 에이전트
Orientation

이 문서가 답하는 네 가지

정보분석 파트는 공시·뉴스를 읽어 종목당 두 개의 구조화 객체를 만들고, 그걸 전략가 에이전트에게 넘긴다. 전략가는 원문을 다시 읽지 않고 이 숫자·문장만 보고 판단해야 하므로, 값 하나하나가 "무엇을·어떻게·왜"를 스스로 설명할 수 있어야 한다.

① 어떻게 산출하나

각 값의 출처(SEC 수치·RSS 메타·LLM 판단·정책 코드)와 계산식. "LLM이 지어내는 값"과 "코드가 확정하는 값"을 명확히 구분한다.

② 정확히 무슨 값인가

0~1 스케일에서 0/0.5/1이 각각 무슨 뜻인지. 방향인지 강도인지 신뢰인지 — 축이 다르면 같은 0.3도 의미가 다르다.

③ 비슷한 항목끼리 뭐가 다른가

importance↔sentiment_score, confidence↔source_trust↔grade_score 처럼 헷갈리기 쉬운 쌍을 §10에서 나란히 비교한다.

④ 전략가에게 뭘 주나

각 값이 전략가의 어떤 결정(진입 여부·비중·타이밍·차단)에 쓰이는지. §12에서 종합 시나리오로 묶는다.

핵심 사상우리 파트는 "살까 말까"를 결정하지 않는다. 사실을 구조화하고, 위험을 차단(hard_block)하고, 얼마나 믿을 값인지(신뢰 3축)를 함께 붙여 넘긴다. 최종 매수·비중·시장반응 판단은 전략가 몫이다. 그래서 모든 점수에 "이 값을 얼마나 신뢰해야 하는가"가 항상 따라붙는다.
Contract

산출물은 종목당 별개 2객체

공시와 뉴스는 성격이 완전히 다르다(공시=법적 확정 사실 / 뉴스=신뢰도 검증이 필요한 보도). 그래서 하나로 합치지 않고 DisclosureBundle·NewsBundle 두 객체로 각각 넘긴다. 둘을 결합하는 건 전략가의 몫이다.

DisclosureBundle SEC 공시

8-K·10-Q/K·Form4에서 나온 단일 신호. 공시는 그 자체가 확정 사실이라 confirmed_score(늘 1.0)·fact_check(늘 Verified)를 아예 뺐다(정보량 0). 신뢰 축은 confidence(LLM 확신도) 하나. 18필드.

고유 필드: filing_title·filing_no·filed_at

NewsBundle 뉴스 RSS

여러 기사를 신뢰가중 집계한 대표 신호. 루머가 섞이므로 신뢰 축이 3개(confidence·source_trust·grade_score)로 갈라지고, 건수(news_count·confirmed·rumor)·peak_importance·confirmed_score·fact_check가 추가된다(공시와 달리 값이 매번 달라 유지). 28필드.

고유 필드: source_trust·grade_score·peak_importance·news_count·confirmed_score·fact_check

전달 형식 — DB 스냅샷정식 계약은 공유 DB 테이블 tb_disclosure(18)·tb_news(28)다. PK=(ticker, collected_at)로 새 공시/기사마다 새 행 append(덮어쓰기 X, 시계열 누적) → 전략가는 종목별 최신 행을 JOIN해 읽는다. to_prompt()/JSON은 보조. 자유서술(summary·reason)과 라벨은 영어(전략가가 영어로 읽음), 점수는 언어 무관. ※ 회사명·카테고리는 tb_universe에 있어 JOIN으로 읽으므로 번들에서 뺐다.

Direction Axis

방향 축 — sentiment / sentiment_score

"이 사건이 주가에 어느 쪽인가"를 나타내는 축. 라벨(sentiment)과 점수(sentiment_score)가 항상 같은 방향을 가리키도록 점수에서 라벨을 파생한다.

sentiment_score — 방향×강도를 0~1로

0.0
강한 악재
0.5
중립 / mixed
1.0
강한 호재

부호 있는 방향값(-1~+1, = 방향 × 강도)을 0.5 + 0.5×signed로 옮긴다. 강도가 실릴수록 0.5에서 멀어진다.

# app/common/analysis_bundle.py
def _senti_score(signed, confidence=1.0):
    signed = max(-1.0, min(1.0, signed))
    if signed > 0:                  # ★ 호재(+)만 신뢰로 감쇠
        signed *= confidence         #   근거 약한 낙관을 0.5쪽으로
    return round(0.5 + 0.5 * signed, 2)  # 악재(−)는 감쇠 없음
비대칭 신뢰 감쇠 — 이 파트의 핵심 안전장치호재는 신뢰도로 눌러 0.5(중립)에 가깝게, 악재는 그대로 둔다. 근거 약한 낙관(펌프·루머성 호재)에 덜 휘둘리고, 경고 신호는 절대 흐리지 않기 위함 — "가장 보수적 채택" 사상. 그래서 같은 강도라도 신뢰 낮은 호재는 점수가 0.5로 수렴하지만, 신뢰 낮은 악재는 여전히 0쪽에 남는다.

실측: 강한 호재(imp8) — 확신 High → 0.82 / 확신 Low → 0.56(감쇠). 강한 악재(imp8) — 확신 Low여도 0.10 유지.

sentiment — 점수에서 파생되는 라벨

라벨은 sentiment_score를 계산한 뒤 거기서 뽑는다. 그래서 라벨과 점수가 어긋나는 일이 없다. 뉴스에는 호·악재가 둘 다 유의미할 때 mixed가 추가된다.

라벨조건의미
negative강한 악재(≤ −0.7) 1건이라도 있으면 우선, 또는 최종 score < 0.5보수적으로 악재 쪽 확정
mixed 뉴스만호재·악재 가중치가 각각 30% 이상방향이 진짜 갈림 — 전략가가 직접 판단하라는 신호
positive최종 score > 0.5호재 (신뢰 감쇠 반영된 뒤)
neutralscore = 0.5방향 없음
주의 — mixed vs negative 우선순위"강한 호재 + 강한 악재"가 섞이면 mixed가 아니라 negative가 나온다. 강한 악재(≤ −0.7)는 mixed 판정보다 먼저 발동하기 때문(보수 우선). 버그가 아니라 설계다.
Magnitude Axis

강도 축 — importance / peak_importance

importance는 방향과 무관한 순수 강도다. "얼마나 큰 사건인가"만 재고, 호재냐 악재냐는 담지 않는다. 이게 방향 축과 분리된 이유가 중요하다.

0.0
일상보고
0.5
보통
1.0
상폐·M&A·실적쇼크

산출 — LLM 0~10 → 코드 0~1 정규화

LLM이 importance: int (0~10)을 반환(거래정지·상폐=10, 대형 M&A·실적쇼크=8~9, 일상보고=1~2). 번들이 _n10()으로 0~1로 정규화한다. 뉴스는 여러 기사의 강도를 신뢰가중 평균한다.

왜 방향과 분리하나 — "상쇄 붕괴" 방지예전엔 강도를 |방향값|으로 냈다. 그러면 중요한 호재 + 중요한 악재가 섞일 때 방향이 상쇄돼 강도까지 0으로 붕괴했다("큰 사건이 없다"는 잘못된 신호). 지금은 강도를 방향과 독립으로 집계해, 같은 강도면 방향 구성과 무관하게 항상 같은 값이 나온다. 실측: 강호+강악 섞여도 imp=0.8 유지, 0으로 안 붕괴.

peak_importance 뉴스 전용 — 신뢰검증 기사 중 최고값

신뢰검증된 기사(확정 or source_trust≥0.6 or 출처 ALLOW) 중 가장 큰 강도(max([importance]+cred_mags)). 신뢰가중 평균한 importance보다 크면, "믿을 만한 대형 사건이 평균에 묻혔다"는 뜻이다. to_prompt()에 [peak imp0.9·low-conf]로 표시돼, 전략가가 "묻힌 대형 신호"를 놓치지 않게 한다. 2026-07-07 편향 수정: 저신뢰 매체가 자극적으로 부풀린 importance는 peak 자격에서 제외(가짜 peak 방지). 신뢰검증 기사가 없으면 importance로 수렴한다(peak≥importance).

Risk Axis

위험 축 — risk_score / hard_block

방향(악재)과는 또 다른 축. 악재라도 하방 위험이 작을 수 있고(단기 노이즈), 중립이라도 구조적 위험이 클 수 있다(희석·회계). 그래서 위험을 별도 두 값으로 낸다: 연속적 risk_score와 이진 안전장치 hard_block.

risk_score LLM 0~1

하방·위험의 연속 점수. 회계문제·희석·상폐 위험일수록 높다. LLM 0~10 → _n10 정규화, 뉴스는 신뢰가중 평균. 비중을 줄이라는 정도의 신호 — 차단은 아님.

hard_block 정책코드 0 / 1

최종권한이 BLOCK_BUY·BLOCK_ALL이면 1. 절대 진입 금지의 이진 스위치. LLM 판단과 무관하게 하드룰·Item기본값이 켠다. hard_block_reason에 사유(예: going concern).

hard_block은 어떻게 켜지나 — 6단계 권한의 최종 결과

세 소스(하드룰 / LLM 제안 / 8-K Item 기본값)의 권한 후보 중 가장 보수적인 것을 채택하고, 그게 BLOCK 계열이면 hard_block=1이 된다.

# 제한도(높을수록 보수적) — 종합 시 max 채택
BLOCK_ALL 6 > BLOCK_BUY 5 > RISK_REDUCE 4 > WATCH_ONLY 3
          > TRADE_ELIGIBLE_SMALL_SIZE 2 > TRADE_ELIGIBLE 1
# 시장반응 미확인이면 매수계열(1·2)은 WATCH_ONLY로 강등 (MVP는 항상 미확인)
하드리스크 유형트리거 키워드(원문)강제 권한hard_block
trading_halttrading halt / suspensionBLOCK_ALL1
delistingdelisting notice / non-complianceBLOCK_BUY1
bankruptcybankruptcy / chapter 11 / going concernBLOCK_BUY1
accounting_issuerestatement / material weaknessBLOCK_BUY1
auditor_changedismissal of auditorBLOCK_BUY1
dilutionsecondary offering / convertible notesBLOCK_BUY1
전략가에게 주는 정보hard_block=1은 "우리가 이미 위험을 확인했으니 이 종목은 매수 후보에서 빼라"는 강한 신호다. risk_score가 "조심해서 비중 줄여"라면, hard_block은 "손대지 마"다. 둘은 층위가 다르다.
Trust — three axes

신뢰 3축 — confidence / source_trust / grade_score

뉴스에서 가장 헷갈리는 부분. "믿을 만한가"를 서로 다른 세 관점에서 잰다. 셋 다 0~1이지만 묻는 질문이 다르다. 공시는 이 중 confidence 하나만 쓴다(출처가 SEC로 고정이라).

confidence

Q. 우리 분석을 얼마나 확신하나?

공시 = LLM 확신도(High .9 / Med .6 / Low .3). 뉴스 = 기사별 종합신뢰(_news_conf)의 평균. 저신뢰 기사뿐이면 낮게.

LLM 원천

source_trust 뉴스

Q. 이 매체·본문을 믿나?

LLM이 출처 등급 + 본문 톤을 함께 보고 0~1로 판단. 신뢰가중 평균. 광고·과장 톤이면 낮아진다.

LLM 판단

grade_score 뉴스

Q. 정책상 어느 등급 매체인가?

도메인 화이트리스트로 코드가 기계적으로: ALLOW 1.0 / GRAY 0.6 / BLOCK 0.0. 신뢰가중 평균.

정책코드
셋의 결정적 차이 — 누가 판단하는가grade_score는 코드가 도메인만 보고 정하는 기계적 등급(reuters.com=1.0), source_trust는 LLM이 본문까지 읽고 정하는 주관 신뢰(같은 Reuters라도 낚시성 톤이면 낮게), confidence는 우리 분석 파이프라인 전체의 확신도다. 그래서 "1급 매체(grade 1.0)인데 본문이 추측성이라 source_trust는 0.4" 같은 조합이 나올 수 있고, 이게 정보다.

_news_conf — 기사 한 건의 종합 신뢰가중치

뉴스 집계의 가중치이자 confidence의 재료. 세 요소의 곱이다.

# 확신도 × 확정여부 × 출처신뢰
c = {High:0.9, Med:0.6, Low:0.3}[certainty]
c *= 1.0 if is_confirmed else 0.5   # 미확정 루머는 절반
c *= source_trust if source_trust is not None else 1.0
# ★ 'or' 아님 — source_trust=0.0(무신뢰)을 1.0으로 오인하던 버그 수정본
놓치기 쉬운 함정source_trust or 1.0으로 쓰면 0.0(완전 무신뢰)이 falsy라 1.0으로 뒤집힌다. 명시적 is not None 검사로 막았다. 실측: trust 0 기사만 있으면 confidence=0.0으로 정직하게 떨어진다.
Confirmation · 뉴스 전용

확정도 — confirmed_score / fact_check (뉴스 전용)

"이게 사실로 굳었나, 아직 소문인가"를 잰다. 신뢰 축(§7)과 다르다 — 신뢰는 "믿을 매체인가", 확정도는 "사실로 발표됐나"다. 공시는 늘 확정(1.0/Verified)이라 두 필드를 뺐고, 값이 매번 다른 뉴스에만 남긴다.

confirmed_score 0~1

확정건수 / 전체건수. 10건 중 3건만 확정 보도면 0.30. 소문 비율이 곧 드러난다. (공시는 늘 1.0이라 필드 자체를 제거 — 정보량 0.)

fact_check 코드 판정

확정비율·신뢰·등급을 코드가 한 문장으로(LLM 아님). 아래 임계값으로 3분류. (공시는 늘 Verified인 상수라 제거.)

뉴스 fact_check 상태조건
Verified확정비율 ≥ 0.5 그리고 source_trust ≥ 0.6
Unverified (rumor)확정건수 = 0 또는 source_trust < 0.4
Partially verified그 사이 (일부 확정)
전략가에게 주는 정보같은 "호재 뉴스"라도 confirmed_score 1.0 · Verified(공식 발표 인용)와 0.2 · Unverified(rumor)(트위터발 소문)는 완전히 다른 베팅이다. 전략가는 이 값으로 진입 크기와 타이밍을 조절한다 — 확정이면 지금, 소문이면 확인될 때까지 관찰. 공시는 이미 확정 사실(별도 값 불필요)이라, 확정 여부는 뉴스에서만 판별한다.

숫자만으로 부족한 맥락. LLM이 쓴 서술 4종과, 코드가 붙이는 원문 추적 메타.

★ Disambiguation

헷갈리는 항목쌍 — 나란히 비교

이 문서에서 가장 중요한 절. 비슷해 보여 자주 혼동되는 값들을 "무엇이 다른가" 기준으로 짝지어 정리한다.

쌍AB결정적 차이
강도 vs 방향 importance
얼마나 큰 사건		 sentiment_score
어느 쪽 + 얼마나		 importance는 부호 없음(호·악재 무관). sentiment_score는 방향을 품음. 큰 악재는 importance 높고 sentiment_score 낮다.
라벨 vs 점수 sentiment
positive/negative/…		 sentiment_score
0~1 연속값		 라벨은 점수에서 파생(항상 일치). 라벨은 빠른 분기용, 점수는 정밀 비교·정렬용.
연속 위험 vs 이진 차단 risk_score
0~1, 비중 줄여		 hard_block
0/1, 손대지 마		 risk_score는 LLM의 정도 판단, hard_block은 정책 코드의 강제 차단. 층위가 다르다(주의 vs 금지).
우리 확신 vs 매체 신뢰 confidence
분석 확신도		 source_trust
매체·본문 신뢰		 confidence는 우리 파이프라인 전체의 확신, source_trust는 그 재료 중 하나(뉴스 출처). confidence ⊃ source_trust.
코드 등급 vs LLM 신뢰 grade_score
도메인 기계등급		 source_trust
본문까지 본 신뢰		 grade_score는 도메인만(reuters=1.0 고정), source_trust는 본문 톤 반영(같은 매체도 가변). 어긋나면 그게 정보.
신뢰 vs 확정 source_trust
믿을 매체인가		 confirmed_score
사실로 발표됐나		 1급 매체가 "~할 전망"으로 보도하면 trust 높고 confirmed 낮다. 신뢰와 확정은 독립.
눌린 평균 vs 최고 importance
신뢰가중 평균		 peak_importance
신뢰검증 중 최고		 peak > importance면 "믿을 만한 대형 신호가 평균에 묻힘". 전략가에게 "묻힌 대형건 주의" 신호(저신뢰 과장은 제외).
LLM 원천 vs 집계 결과 certainty_level
High/Med/Low		 confidence
0~1		 certainty는 LLM이 기사별로 낸 원천 등급, confidence는 그걸 확정여부·출처신뢰와 곱해 집계한 최종 0~1.
한 줄 정리방향(sentiment) · 강도(importance) · 위험(risk/hard_block) · 신뢰(confidence·trust·grade) · 확정(confirmed) — 이 다섯 축은 서로 독립이다. 한 축의 값으로 다른 축을 추측하면 안 된다. 예: importance 높다고 위험한 게 아니고(대형 호재일 수도), grade 높다고 확정도 아니다(전망 보도일 수도).
News aggregation

뉴스 집계 공식 — 여러 기사를 하나로

공시는 단일 신호라 정규화만 하면 되지만, 뉴스는 N건을 하나의 대표 신호로 접어야 한다. 전 과정이 신뢰가중이다.

# 1) 기사별 가중치 = _news_conf (확신 × 확정 × 출처신뢰)
raw_weights = [_news_conf(r.signal) for r in analyzed]
confidence  = mean(raw_weights)              # 저신뢰뿐이면 낮게

# 2) 전부 0이면 등가중 폴백 — 강도·위험이 0으로 붕괴하는 것 방지
weights = raw_weights if sum(raw_weights) > 0 else [1.0]*n

# 3) 강도: 방향 무관 신뢰가중 평균 (상쇄 없음)
importance      = Σ(mag·w) / Σw
# peak: 신뢰검증 기사(확정·trust≥0.6·ALLOW) 중 최댓값. 없으면 importance로 수렴
cred_mags       = [mag for 기사 if _news_credible(기사)]
peak_importance = max([importance] + cred_mags)

# 4) 방향: 강한 악재(≤−0.7) 있으면 그쪽, 아니면 신뢰가중 평균
agg_signed     = min(signed) if strong_neg else Σ(signed·w)/Σw
sentiment_score = _senti_score(agg_signed, confidence)   # 호재만 감쇠

# 5) risk·source_trust·grade_score 도 전부 신뢰가중 평균
# 6) confirmed_score = 확정건수 / 전체,  대표기사 = imp×w 최대
설계 원칙모든 집계가 신뢰가중이라, 확정·고신뢰 기사가 결과를 지배하고 루머는 가중치로 눌린다. 단 강도·위험만은 "전부 저신뢰"일 때 등가중 폴백으로 0 붕괴를 막고, 대신 confidence가 정직하게 낮게 나와 "값은 있지만 못 믿는다"를 전달한다.
★ For the Strategist

전략가는 이 값들로 무엇을 하나

개별 값의 의미를 넘어, 전략가가 두 번들을 받아 어떤 결정에 어떤 값을 쓰는지를 종합한다. 전략가는 원문을 안 읽고 이 값들만으로 판단한다.

전략가의 결정주로 보는 값읽는 법
후보에서 뺄까?hard_block · hard_block_reason1이면 즉시 제외. 우리가 이미 위험 확인함.
방향은?sentiment · sentiment_score · reason라벨로 빠른 분기(mixed면 직접 판단), 점수로 강도까지, reason으로 근거 확인.
얼마나 큰 건?importance · peak_importance강도로 반응 크기 가늠. peak>imp면 묻힌 대형건 주의.
지금 vs 관찰?confirmed_score · fact_check확정이면 지금, 소문이면 확인 대기.
이 값 믿어도 되나?confidence · source_trust · grade_score세 축 모두 높아야 강하게 베팅. 하나라도 낮으면 축소.
비중은?risk_score + 위 신뢰 3축risk 높거나 신뢰 낮으면 소액. 소수점 분할 진입.
공시 vs 뉴스 결합두 번들 event_type 일치 여부같은 이벤트를 공시가 확정하면 뉴스 루머의 신뢰 상승(교차확인).

종합 예시 — 같은 "호재"의 세 얼굴

강한 매수 후보

senti 0.9 · imp 0.85 · conf 0.9 · risk 0.2 · hard_block 0

공시로 확정된 대형 호재, 고신뢰(공시는 늘 확정이라 confirmed 불필요). → 지금 진입, 비중 있게.

관찰 대상

senti 0.56 · imp 0.9 · confirmed 0.2 · conf 0.3 · peak 0.9

신뢰출처의 미확정 대형 호재(peak 0.9)지만 전체는 미확정 우세(confirmed 0.2)·호재 감쇠로 senti 0.56. → 확정 뉴스 나올 때까지 관찰.

차단

senti 0.7 · imp 0.8 · hard_block 1
reason: going concern

LLM은 호재라 봤지만 본문에 존속의문 문구. → hard_block이 우선, 매수 제외.

결론우리가 넘기는 건 "방향·강도·위험·신뢰·확정"이라는 다섯 개의 독립 눈금과, 그걸 코드가 검증한 안전장치다. 전략가는 이 다섯 눈금을 조합해 "진입 여부 × 타이밍 × 크기"를 정한다. 각 값이 "얼마나 믿을 값인지"를 항상 함께 들고 있기 때문에, 전략가는 확신에 비례해 베팅할 수 있다 — 그게 이 파트가 존재하는 이유다.
