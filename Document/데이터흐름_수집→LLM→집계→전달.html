뉴스·공시 데이터 흐름 정리

언론사에서 기사 한 건을 가져온 순간부터, LLM 분석이 붙고, 여러 건이 하나의 종목 신호로 집계돼, 두 개의 DB 테이블을 거쳐 Strategist에게 전달되기까지의 전체 데이터 경로를 실제 코드·필드·렌더 출력으로 정리한 문서.

Overview

한눈에 보는 흐름

한 기사가 NormalizedItem으로 수집되고, 코드 판정(출처등급·키워드)과 LLM 분석(NewsSignal)이 덧붙어 NewsResult가 되며, 이 결과가 두 갈래로 저장된다 — 기사 1건씩 남기는 로그(news_signals)와, 여러 건을 신뢰가중으로 집계한 스냅샷(tb_news). Strategist는 후자의 최신 행만 읽는다.

Collection

수집 — 한 기사 = 한 데이터 묶음

Google News RSS의 각 <item>이 NormalizedItem 객체 하나로 묶인다(news_collector.py). 이 객체 하나만 들고 다니면 그 기사의 모든 정보가 따라온다.

LLM Extension

LLM 확장 — NewsSignal

run_llm=True이면 analyze(item)이 헤드라인을 보고 NewsSignal을 만든다. 수집 데이터엔 없던, LLM이 새로 판단한 값들이다.

추론 강제 LLM

reasoning (CoT) · certainty_level (High/Medium/Low = 단정 vs 추측)

핵심 신호 LLM

event_type · sentiment · importance(0~10) · risk_score(0~10)

뉴스 전용 신뢰·사실성 LLM

is_confirmed(확정사실?) · source_trust(0~1, 본문톤+출처등급) · theme

요약 3필드 LLM

summary(3~4줄) · keywords(5개) · verdict(판정 한 문장)

이중 신뢰 방어 — source_trust는 LLM이 본문 보고 매긴 신뢰, 다음 장의 grade_score는 코드가 도메인만 보고 매긴 신뢰. 일부러 분리해 두 방향에서 방어한다.
Metric

grade_score 산출 — 출처 품질 0~1

"이 종목 뉴스가 얼마나 믿을 만한 매체에서 나왔나"를 나타내는 지표. 코드로만 판정(LLM 개입 없음)해 재현성 100%. 도메인 → 등급 → 점수 → 신뢰가중 평균의 3단계.

1단계 · URL 도메인 → 3등급 코드

등급의미도메인 예시 (config yaml)
ALLOW공식·1급 매체sec.gov · reuters.com · bloomberg.com · wsj.com · cnbc.com · apnews.com · businesswire.com …
GRAY2급, 단독 매수 금지benzinga.com · seekingalpha.com · fool.com · zacks.com · tipranks.com …
BLOCK분석 제외(조기 drop)sponsored / paid promo 등 block_conditions

미등록 도메인(소셜 x·reddit·stocktwits 포함)은 전부 보수적 기본값 GRAY로 분류. (구 소셜용 WATCH_ONLY 4번째 등급은 2026-07-03 팀 결정으로 제거)

2·3단계 · 등급→점수, 신뢰가중 평균

# 2단계: 등급 → 점수 (_GRADE_SCORE)
ALLOW = 1.0   GRAY = 0.6   BLOCK = 0.0

# 3단계: 기사별 점수를 신뢰가중(_news_conf)으로 평균
grade_score = round(sum(_GRADE_SCORE[grade] * w
              for grade, w in zip(grades, weights)) / tw, 2)
흔한 오해 3가지 — ① grade_score ≠ 매수 허용(ALLOW여도 "분석 반영 가능"일 뿐). ② grade_score(코드·사전) ≠ source_trust(LLM·사후) ≠ confidence(분석 확신). ③ 출처 등급과 매매권한의 WATCH_ONLY는 완전 별개.
Aggregation · News

뉴스 집계 — N개 기사 → 1개 NewsBundle

핵심은 단순 평균이 아니라 기사별 신뢰도로 가중평균. 가중치 _news_conf가 전체를 관통한다.

가중치 = _news_conf (기사 1건의 신뢰도)

c = certainty(High .9 / Medium .6 / Low .3)   # LLM 확신도
c *= 1.0 if is_confirmed else 0.5          # 확정이면 그대로, 루머면 반토막
c *= source_trust (0~1)                     # LLM이 매긴 출처신뢰
# → certainty × 확정계수 × source_trust

이 가중치로 각 지표 합산

번들 필드계산포인트
confidenceraw_weights 단순평균저신뢰뿐이면 정직하게 낮음
importanceΣ(강도×w)/Σw방향 무관 — 호·악재 섞여도 상쇄 안 됨
peak_importancemax(강도)가장 센 기사 1건
sentiment_score방향×강도 가중평균 → _senti_score아래 ★ 보수 처리
sentiment(라벨)sentiment_score에서 파생라벨↔점수 항상 일치
risk_score · source_trust · grade_score각 Σ(값×w)/Σw전부 신뢰가중
confirmed_scoreconfirmed / count단순 확정비율
★ 방향(sentiment_score)의 보수적 처리 — ① 강한 악재(≤ −0.7) 1건이면 가중평균 무시하고 그쪽으로(경고를 호재로 희석 안 함). ② 호재는 confidence로 감쇠, 악재는 안 함(근거 약한 낙관은 중립 0.5로 끌어당겨 펌프 방지). ③ 호·악재 가중치 둘 다 ≥30%면 mixed.
대표 기사 선정 — 번들에 실리는 event_type·reason·summary·원문메타는 importance×confidence 최대 기사 1건에서 뽑는다. 시끄러운 저신뢰 루머가 헤드라인 되는 걸 막는다.
Aggregation · Disclosure

공시 집계 — 왜 단순한가

build_disclosure_bundle은 disclosure_result 딱 1건을 받아 필드만 옮긴다. 가중치·평균·대표선정이 없다.

이유 — 한 종목의 특정 공시(예: AAPL 8-K)는 원본이 유일하다. 여러 매체가 제각각 기사화하는 뉴스와 달리, 공시는 SEC 문서 하나가 진실원천이라 합칠 대상이 없다.
뉴스공시
입력기사 N건1건
집계신뢰가중 평균·대표선정없음 (1:1 복사)
confidence_news_conf 평균certainty_level 하나 그대로
없는 필드—source_trust · grade_score · confirmed_score · fact_check · peak_importance · news_count류
필드 수2818

신뢰/출처 필드가 없는 이유: 출처가 SEC 하나로 고정 = 항상 최고 신뢰라 "출처 얼마나 믿냐" 지표가 무의미. 대신 "공시 = 법적 사실"로 취급. 안전장치도 확률적 감점이 아니라 규칙 기반 확정 차단(final_permission이 BLOCK류면 hard_block=1). 방향점수는 뉴스와 같은 _senti_score를 쓰지만, 공시는 대개 certainty=High(0.9)라 감쇠가 거의 안 걸린다.

Storage

두 테이블 관계 — 로그 vs 전달물

같은 파이프라인 산출물이 두 갈래로 저장된다. news_signals의 여러 기사 행이 집계되어 tb_news 한 행이 된다.

용도분석 로그 → 백테스트·증분처리스냅샷 → Strategist가 읽음

Strategist는 latest_news(ticker) = ORDER BY collected_at DESC LIMIT 1로 가장 최신 행 하나만 읽는다. 시계열이라 새 기사가 뜰 때마다 새 collected_at 행이 append된다.

실제 렌더 샘플 — to_prompt()

to_prompt()는 사람이 읽기 쉬운 보조 표현이고, Strategist에게 가는 정식 계약은 model_dump_json(). 아래는 합성 신호(공시 1건 + 뉴스 3건: 확정1·루머2)로 실제 실행한 출력.

buyback 갭 → 수정 완료

위 렌더에서 자사주매입 공시의 event_type이 other로 새던 문제(코드 버그가 아니라 온톨로지 갭)를 이번에 수정했다.

원인

통일 온톨로지에 자사주매입 카테고리가 없어서 norm_event가 buyback 라벨을 other로 떨어뜨렸다. 정작 키워드 필터엔 shareholder_return 화이트리스트(share buyback·repurchase)가 이미 있어 입력은 통과시키되 분류에서 뭉개지는 비대칭 상태였다.

수정 내용

  • 온톨로지에 buyback 추가 (11 → 12종). 증자·희석의 반대·주주환원 개념이라 capital_raise 바로 뒤 대칭 위치에 배치.
  • alias 등록: share_repurchase · repurchase → buyback (표기 흔들림 흡수). 배당(dividend·special_dividend)은 별개 이벤트라 매핑 안 함 → 계속 other.
  • 프롬프트 힌트: 8-K·뉴스 프롬프트에 "자사주매입은 event_type=buyback (증자는 capital_raise, 배당은 buyback 아님)" 한 줄씩 추가.

검증

norm_event('buyback')          -> 'buyback'
norm_event('share_repurchase')  -> 'buyback'
norm_event('repurchase')        -> 'buyback'
norm_event('dividend')          -> 'other'   # 의도대로 유지
EVENT_TYPES count: 12 · 전체 import OK
남은 후속(문서) — 우리 문서(데이터스키마명세_v2·결과값_필드_레퍼런스·상세설계서·개발보고서 등)는 2026-07-07 전부 12종으로 갱신 완료. 팀 파이프라인 명세(김지현) 05·06만 아직 "11종"이라 회의 때 조율 필요.
