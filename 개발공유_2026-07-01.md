# Quantinue 정보분석 파트 — 개발 공유 (2026-07-01)

> 작성: 정보분석(공시·뉴스) 담당
> 대상: 팀 "여름이었다" 팀원
> 목적: 기존 설계 대비 **보완/수정한 점 + 현재 개발 상황** 공유
> (이 버전은 수집 스케줄러·뉴스 저장·유니버스 시총 확장·백테스트까지 반영한 최신본)

---

## 0. 한눈에

- 정보분석 파트(**①공시 + ②뉴스**)가 **수집→분석→저장→유니버스→주기실행→성과측정 한 바퀴** 완성.
- 구체적으로: **공시 3종(8-K·10-Q/K·Form4) + 뉴스**를 신호로 바꾸고, **시총 상위 종목을 주기적으로**
  자동 순회하며, 결과를 DB에 쌓고, 나중에 **"그 신호가 실제로 주가에 먹혔는지"까지 채점**함.
- 아래 전부 **샘플이 아니라 실데이터(SEC·Google News·Yahoo)로 검증** 완료.
- 이 파트의 역할 경계: **"구조화 신호 + 보수적 1차 권한 제안"까지만**.
  실제 매수결정·시장반응 검증은 **별도 Strategist 에이전트**가 담당 (아래 3번 참고).

| **수집 스케줄러** | 우선순위 순회 · **문서 받기 전 증분 skip** · 공시+뉴스 통합 · 주기 자동 실행 | ✅ |
| **유니버스 확장** | Yahoo로 **시총 채워 우선순위=시총순위**, 상위 N만 활성 (2,000개까지 확장 가능) | ✅ |
| **백테스트 피드백** | 신호 이후 +1/3/5일 수익률 채우고 **이벤트/출처별 적중률 리포트** | ✅ |

## 2. 기존 설계 대비 보완/수정한 점 (핵심)

> 처음 보는 팀원도 이해할 수 있게, 각 항목을 **① 기존엔 이랬다 → ② 이렇게 바꿨다 → ③ 그래서 뭐가 좋아졌다** 순서로 풀어 씀.
> 배경 설계는 `Finance.html`(05장)과 고도화 초안 V1/V2. 아래는 그걸 실제로 만들면서 내린 결정들.

---

### (1) AI에게 다 맡기지 않고, "틀리기 어려운 구조"를 먼저 짰다

**① 기존 생각**
초안에서는 "성능 좋은 AI(LLM)한테 공시·뉴스를 통째로 주고 판단을 뽑자"는 그림이었음.

**② 바꾼 것**
AI를 **판단의 전부가 아니라 한 부품**으로 위치시킴. 판단의 뼈대(어떤 걸 걸러낼지·어떤 규칙으로 위험을 막을지·결과를 어떤 형식으로 낼지)는 **사람이 만든 고정 규칙**이 잡고, AI는 그 울타리 안에서 **해석만** 하게 함.
특히 정보 종류마다 AI를 쓰는 비중을 다르게 둠:
| 정보 종류 | 처리 방식 | 이유 |
|---|---|---|
| **내부자 거래(Form 4)** | **AI 안 씀. 100% 계산 규칙**<br>(임원이 사면 호재 / 팔면 약한 악재 / 스톡옵션·세금목적 매도는 중립, 직급 높을수록 가중치↑) | 숫자·거래유형이 명확해 AI가 지어낼 여지 자체를 없앰 |
| **실적 보고서(10-Q/K)** | **숫자는 계산기로, 해석만 AI** | 매출·이익 증감은 계산이 정확. AI는 "그래서 좋다/나쁘다"만 |
| **속보성 공시(8-K)** | 본문 문장만 AI가 해석 | 자유 서술이라 AI가 필요한 유일한 영역 |

**③ 좋아진 점**
AI가 사실을 **지어내는(할루시네이션) 사고를 구조적으로 차단**. 돈이 걸린 판단에서 "AI가 매출 숫자를 잘못 읽는" 최악의 실수가 원천적으로 안 나옴. AI 호출도 꼭 필요한 데만 써서 **비용·속도도 절감**.

---

### (2) AI에 넘기기 전에 "쓰레기 정보를 미리 걸러내는" 관문을 뒀다

**① 기존 생각**
들어온 뉴스·공시를 일단 AI한테 다 보내서 판단.

**② 바꾼 것**
AI에 보내기 **전에** 두 겹의 자동 필터를 통과시킴:
- **키워드 필터**: 노이즈성 단어가 들어간 기사는 버림.
- **출처 등급(4단계)**: 공식 발표·대형 언론은 신뢰, 출처 불명·루머성은 아예 차단.

그리고 여기서 실수하기 쉬운 함정을 하나 미리 막아둠 → **"좋은 신호가 나쁜 단어 때문에 억울하게 버려지는" 문제.** 예를 들어 호재성 공시에 우연히 "investigation(조사)" 같은 단어가 섞여 있어도 버리지 않도록, **"신뢰 키워드"가 "차단 키워드"보다 우선**하게 설계.

**③ 좋아진 점**
- 불필요한 AI 호출이 줄어 **비용·속도 개선**.
- 애초에 **믿을 만한 정보만 분석**에 들어가 결과 품질↑.
- "억울한 오기각"까지 막아 **진짜 기회를 놓치지 않음**.

---

### (3) "사도 되나?"를 애매하게 두지 않고, 6단계 신호등으로 통일했다

**① 기존 생각**
매매 허용 여부의 단계가 문서마다 제각각이고 기준이 흐릿했음.

**② 바꾼 것**
모든 판단 결과를 **딱 6단계 신호등**으로 통일:
```
전면 금지  >  매수 금지  >  위험 축소  >  관망만
                                          >  소액만 매수 가능  >  매수 가능
   (빨강) ────────────────────────────────────────────── (초록)
```
그리고 두 가지 안전장치:
- 여러 규칙이 서로 다른 등급을 내면 → **항상 가장 보수적인(안전한) 등급을 채택**.
- 아직 **시장이 실제로 반응했는지 확인 안 된 상태**면, 매수 신호라도 일단 **"관망"으로 한 단계 낮춤** (성급하게 사지 않게).
- 추가: **10-Q/K 재무보고서 본문에서 "계속기업 불확실성(going concern)"·"중대한 내부통제 결함" 같은 위험 문구가 보이면** 숫자가 좋아도 자동으로 **매수 금지**로 강등.

**③ 좋아진 점**
- 판단 결과가 **누가 봐도 똑같이 읽히는 공통 언어**가 됨(다음 단계 담당자·백테스트가 바로 활용).
- 시스템이 **의심스러울 땐 무조건 안전한 쪽**으로 기울어 **큰 손실 위험을 줄임**.

---

### (4) 공시와 뉴스는 "판단은 따로, 배관은 공유"로 만들었다

**① 기존 생각**
공시 파트와 뉴스 파트를 각각 별개로 개발.

**② 바꾼 것**
- **판단 로직은 분리** (공시와 뉴스는 성격이 다르니 각자의 분석기·규칙을 가짐).
- 대신 **공통 뼈대**(정보를 표준 형식으로 정리 → 최종 등급 결정 → 저장)는 **하나로 공유**.

**③ 좋아진 점**
- **같은 코드를 두 번 만들지 않음** → 버그·유지보수 부담↓.
- 나중에 세 번째 정보원(예: SNS·실적 컨센서스)을 붙일 때 **공통 뼈대에 꽂기만** 하면 됨 → **확장이 쉬움**.

---

### (5) "바뀐 것만, 그것도 문서 받기 전에" 걸러내 비용을 줄였다

**① 기존 생각**
돌릴 때마다 모든 종목의 모든 공시를 처음부터 다시 내려받아 분석.

**② 바꾼 것**
- 직전에 본 것과 **같은 공시면 재분석·재저장을 건너뜀** (공시마다 붙는 고유 번호로 "새 것인지" 판별).
- 한 발 더: **무거운 문서를 내려받기 전에**, 가벼운 "목록"만 먼저 확인해 새 것이 아니면 **다운로드 자체를 안 함**.
- 뉴스도 같은 원리로 **이미 본 기사는 다시 AI에 안 보냄** (기사별 고유 링크로 판별, 프로그램을 껐다 켜도 유지).

**③ 좋아진 점**
- **불필요한 다운로드·AI 재호출이 없어 비용·시간 대폭 절감**.
- "변화가 있을 때만 움직인다"는 원칙이라 **종목 수를 2,000개로 늘려도 감당 가능한 구조**.

---

### (6) 종목을 "크기순으로 줄 세워" 중요한 것부터 돌린다

**① 기존 생각**
감시 종목 50개를 그냥 순서대로.

**② 바꾼 것**
- Yahoo에서 **시가총액을 받아** 종목을 큰 순서로 정렬 → **우선순위 = 시총 순위**.
- "상위 몇 개만 활성"으로 **작업 범위를 고정**(예: 상위 2,000개만 돌림).
- 스케줄러가 이 순서를 따르므로 **큰 종목(영향력 큰 정보)부터** 자동 처리.

**③ 좋아진 점**
- 50 → 200 → **2,000개**로 늘려도 **중요한 종목이 먼저** 처리됨.
- 데이터 소스도 **무료·키 불필요**(Yahoo)라 운영 비용 0.

---

### (7) "정보가 실제로 주가에 먹혔는지"를 스스로 채점한다 (핵심 철학)

**① 기존 생각**
신호를 내는 것까지만. 그 신호가 맞았는지는 확인 안 함.

**② 바꾼 것**
- 저장된 각 신호에 대해, **그 이후 +1·+3·+5 거래일 실제 수익률**을 자동으로 붙임.
- "호재라고 봤는데 진짜 올랐나?"를 대조해 **적중/실패**를 라벨링.
- 이걸 **이벤트 종류별·출처별로 집계한 성과 리포트**를 뽑음.
- 단, **여기서 가중치를 자동으로 바꾸진 않음** — 데이터가 적을 때 자동 튜닝하면 우연에 끼워맞추는
  **과적합** 위험이 커서, 지금은 **사람이 리포트를 보고 판단**하는 데까지만.

**③ 좋아진 점**
- "우리 정보 분석이 실제로 돈이 되는가"를 **숫자로 검증**하는 토대가 생김 (이 시스템의 진짜 엣지).
- 어떤 이벤트·출처가 잘 맞는지 쌓이면, 나중에 **근거 있는 가중치 조정**이 가능.

---

### (8) 욕심내지 않고, 지금 안 할 것을 "근거 있게" 뺐다

무조건 다 넣는 게 아니라, **왜 지금은 아닌지**를 분명히 하고 미룬 항목들:

| 뺀/미룬 것 | 이유 |
|---|---|
| 애널리스트 예상치 대비 "어닝 서프라이즈" | **예상치 데이터가 유료** → 지금은 전년 대비 증감만 사용 |
| 이벤트 종류를 19종으로 잘게 나누기 | 종류가 너무 많으면 **종목당 데이터가 부족**해 나중에 성과 검증이 무의미 → 6~8종으로 |
| 성과 기반 가중치 **자동** 튜닝 | 데이터가 적을 때 자동 조정하면 **과적합** 위험 → 우선 수동 기록·분석부터 (위 (7)) |

## 3. 역할 경계 (팀 협업 관점 — 중요)

이 정보분석 파트는 **다음까지만** 낸다:
> **구조화된 신호(event_type·sentiment·certainty·importance) + 보수적 1차 매매권한 제안**

**하지 않는 것 (= Strategist 에이전트 몫):**
- 실제 시장반응(가격·거래량) 기반 최종 판단
- 최종 매수/매도 권한 부여 및 사이징
- 여러 신호 종합·포트폴리오 판단

→ 인터페이스는 **`DisclosureSignal` / `NewsSignal` 스키마 + SQLite 저장**(`disclosure_signals`·`news_signals`).
   Strategist 담당자는 이 스키마/DB만 보면 됨. (백테스트용 수익률 컬럼도 같은 테이블에 있음)

## 4. 기술 스택

- **Python + Pydantic AI** (구조화 출력·검증·재시도의 핵심 엔진)
- **LLM: OpenAI gpt-4o-mini** (분석가 티어, 현재 보유 키 기준. `LLMClient`로 추상화돼 교체 쉬움)
- **데이터: 전부 무료/무키** — SEC(User-Agent만), Google News RSS, **Yahoo(시총·가격)**
- **저장: SQLite** (`data/quantinue.sqlite`) — 공시·뉴스 신호 + 유니버스 + 백테스트 수익률

## 5. 남은 작업 (대부분 Phase 2 정교화)

MVP 골격(수집→분석→저장→유니버스→스케줄러→백테스트)은 **한 바퀴 완성**. 남은 건:

1. **XBRL 어닝 서프라이즈** — 애널 컨센서스가 유료라 무료 대안 검토 중(보류).
2. **섹터 정보 채우기** — 종목별 섹터 태깅(Yahoo 추가 호출).
3. **이벤트 온톨로지 정교화** — 현재 `other`로 분류되는 게 많음 → 실제 패턴 보며 조정.
4. **표본이 쌓이면 실제 가중치 튜닝** — 백테스트 성과 근거로(과적합 안 되게 신중히).
5. **대규모 성능** — 2,000개 주기 실행 시 동시성/큐 분산.

## 6. 알려진 한계 (투명 공개)

- **시장반응·실제 매수권한 없음** → 의도된 것(Strategist 몫).
- **뉴스가 헤드라인 위주**(RSS 한계) → 깊은 분석엔 본문 수집 필요.
- **event_type = other 빈발** → 온톨로지가 실제 패턴에 덜 맞음. 데이터 보며 조정 예정.
- **Yahoo는 비공식 소스** → 대량 조회 시 간헐적 실패 가능(실패분은 건너뛰고 계속). 정확한 시총/잔액은 별도 확인.
- **백테스트 수익률은 5거래일 지나야 확정** → 최근 신호는 시간이 지나야 채워짐.
- **10-Q/K 위험문구 스캔은 원문 다운로드** → 전체 유니버스 대량 스캔 시 비용(현재는 새 보고서일 때만 받음).
