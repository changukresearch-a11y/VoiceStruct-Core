## 1. 무엇을 만들었나 (개발 현황)

| 영역 | 내용 | 상태 |
|---|---|---|
| **공시 8-K** | SEC 수집 → 8-K Item 라우터 → 하드리스크 룰 → LLM 이벤트 해석 | ✅ 실데이터 |
| **공시 10-Q/K** | XBRL로 YoY 수치를 **코드로** 계산 → LLM은 수치 해석만. **본문에서 going concern 등 위험문구 스캔** | ✅ 실데이터 |
| **공시 Form 4** | 내부자거래 XML 파싱 → **100% 코드 스코어링**(LLM 미사용) | ✅ 실데이터 |
| **뉴스** | Google News RSS 수집 → 출처 4단계 + 키워드 필터 → LLM(사실확인·신뢰도) | ✅ 실데이터 |
| **사전필터 · 하드룰 · 권한** | LLM 전 노이즈/출처 차단 · 위험문구 강제차단 · 6단계 권한 1차 제안 | ✅ |
| **Strategist 인터페이스** | 종목당 1개 `AnalysisBundle`로 압축해 전달 (§4) | ✅ 실물 검증 |

### 실행 명령어 (팀원이 바로 돌려볼 수 있게)
```bash
cd quantinue
pip install -r requirements.txt
cp .env.example .env      # SEC_USER_AGENT / OPENAI_API_KEY 채우기

# 공시 (단건)
python run_skeleton.py --ticker AAPL --form 8-K  --live --llm   # 이벤트
python run_skeleton.py --ticker AAPL --form 10-Q --live --llm   # 재무(+위험문구 스캔)
python run_skeleton.py --ticker AAPL --form 4    --live         # 내부자거래(LLM 미사용)

# 뉴스
python run_news.py --ticker AAPL --limit 4 --llm

# Strategist에 넘길 번들 (공시+뉴스 → 종목당 1개)
python run_bundle.py --ticker NVDA --form 8-K --news-limit 8 --llm
```

## 2. 기존 설계 대비 보완/수정한 점 (핵심)

> 처음 보는 팀원도 이해할 수 있게, 각 항목을 **① 기존엔 이랬다 → ② 이렇게 바꿨다 → ③ 그래서 뭐가 좋아졌다** 순서로 풀어 씀.
> 배경 설계는 `0701.html`(05장)과 고도화 초안. 아래는 그걸 실제로 만들면서 내린 결정들.

---

### (1) AI에게 다 맡기지 않고, "틀리기 어려운 구조"를 먼저 짰다

**① 기존 생각**
초안에서는 "성능 좋은 AI(LLM)한테 공시·뉴스를 통째로 주고 판단을 뽑자"는 그림이었음.

**② 바꾼 것**
AI를 **판단의 전부가 아니라 한 부품**으로 위치시킴. 판단의 뼈대(어떤 걸 걸러낼지·어떤 규칙으로 위험을 막을지·결과를 어떤 형식으로 낼지)는 **사람이 만든 고정 규칙**이 잡고, AI는 그 울타리 안에서 **해석만** 하게 함.
특히 정보 종류마다 AI를 쓰는 비중을 다르게 둠:
| 정보 종류 | 처리 방식 | 이유 |
|---|---|---|
| **내부자 거래(Form 4)** | **AI 안 씀. 100% 계산 규칙**<br>(임원이 사면 호재 / 팔면 약한 악재 / 스톡옵션·세금목적 매도는 중립, 직급 높을수록 가중치↑) | 숫자·거래유형이 명확해 AI가 지어낼 여지 자체를 없앰 |
| **실적 보고서(10-Q/K)** | **숫자는 계산기로, 해석만 AI** | 매출·이익 증감은 계산이 정확. AI는 "그래서 좋다/나쁘다"만 |
| **속보성 공시(8-K)** | 본문 문장만 AI가 해석 | 자유 서술이라 AI가 필요한 유일한 영역 |

**③ 좋아진 점**
AI가 사실을 **지어내는(할루시네이션) 사고를 구조적으로 차단**. 돈이 걸린 판단에서 "AI가 매출 숫자를 잘못 읽는" 최악의 실수가 원천적으로 안 나옴. AI 호출도 꼭 필요한 데만 써서 **비용·속도도 절감**.

---

### (2) AI에 넘기기 전에 "쓰레기 정보를 미리 걸러내는" 관문을 뒀다

**① 기존 생각**
들어온 뉴스·공시를 일단 AI한테 다 보내서 판단.

**② 바꾼 것**
AI에 보내기 **전에** 두 겹의 자동 필터를 통과시킴:
- **키워드 필터**: 노이즈성 단어가 들어간 기사는 버림.
- **출처 등급(4단계)**: 공식 발표·대형 언론은 신뢰, 출처 불명·루머성은 아예 차단.

그리고 여기서 실수하기 쉬운 함정을 하나 미리 막아둠 → **"좋은 신호가 나쁜 단어 때문에 억울하게 버려지는" 문제.** 예를 들어 호재성 공시에 우연히 "investigation(조사)" 같은 단어가 섞여 있어도 버리지 않도록, **"신뢰 키워드"가 "차단 키워드"보다 우선**하게 설계.

**③ 좋아진 점**
- 불필요한 AI 호출이 줄어 **비용·속도 개선**.
- 애초에 **믿을 만한 정보만 분석**에 들어가 결과 품질↑.
- "억울한 오기각"까지 막아 **진짜 기회를 놓치지 않음**.

---

### (3) "사도 되나?"를 애매하게 두지 않고, 6단계 신호등으로 통일했다

**① 기존 생각**
매매 허용 여부의 단계가 문서마다 제각각이고 기준이 흐릿했음.

**② 바꾼 것**
모든 판단 결과를 **딱 6단계 신호등**으로 통일:
```
전면 금지  >  매수 금지  >  위험 축소  >  관망만
                                          >  소액만 매수 가능  >  매수 가능
   (빨강) ────────────────────────────────────────────── (초록)
```
그리고 두 가지 안전장치:
- 여러 규칙이 서로 다른 등급을 내면 → **항상 가장 보수적인(안전한) 등급을 채택**.
- 아직 **시장이 실제로 반응했는지 확인 안 된 상태**면, 매수 신호라도 일단 **"관망"으로 한 단계 낮춤** (성급하게 사지 않게).
- 추가: **10-Q/K 재무보고서 본문에서 "계속기업 불확실성(going concern)"·"중대한 내부통제 결함" 같은 위험 문구가 보이면** 숫자가 좋아도 자동으로 **매수 금지**로 강등.

**③ 좋아진 점**
- 판단 결과가 **누가 봐도 똑같이 읽히는 공통 언어**가 됨(다음 단계 담당자가 바로 활용).
- 시스템이 **의심스러울 땐 무조건 안전한 쪽**으로 기울어 **큰 손실 위험을 줄임**.

---

### (4) 공시와 뉴스는 "판단은 따로, 출력 형식은 공유"로 만들었다

**① 기존 생각**
공시 파트와 뉴스 파트를 각각 별개로 개발.

**② 바꾼 것**
- **판단 로직은 분리** (공시와 뉴스는 성격이 다르니 각자의 분석기·규칙을 가짐).
- 대신 **결과를 담는 공통 형식**(정보를 표준 형식으로 정리 → 최종 등급 결정 → 정해진 신호 스키마로 출력)은 **하나로 공유**.

**③ 좋아진 점**
- **같은 코드를 두 번 만들지 않음** → 버그·유지보수 부담↓.
- 나중에 세 번째 정보원(예: SNS·실적 컨센서스)을 붙일 때 **공통 형식에 꽂기만** 하면 됨 → **확장이 쉬움**.

---

### (5) 욕심내지 않고, 지금 안 할 것을 "근거 있게" 뺐다

무조건 다 넣는 게 아니라, **왜 지금은 아닌지**를 분명히 하고 미룬 항목들:

| 뺀/미룬 것 | 이유 |
|---|---|
| 애널리스트 예상치 대비 "어닝 서프라이즈" | **예상치 데이터가 유료** → 지금은 전년 대비 증감만 사용 |
| 이벤트 종류를 19종으로 잘게 나누기 | 종류가 너무 많으면 **종목당 데이터가 부족**해 나중에 성과 검증이 무의미 → 6~8종으로 |

**③ 좋아진 점**
한정된 인원·기간에 **핵심(정보 거르기·출처 신뢰·안전한 등급 결정)에 집중**해 실제 동작하는 결과물을 냄.

## 3. 역할 경계 (팀 협업 관점 — 중요)

이 정보분석 파트는 **다음까지만** 낸다:
> **구조화된 신호(event_type·sentiment·certainty·importance) + 보수적 1차 매매권한 제안**

**하지 않는 것 (= 다른 담당 몫):**
- 실제 시장반응(가격·거래량) 기반 최종 판단, 매수/매도 권한 부여·사이징, 포트폴리오 종합 → **Strategist~게이트**
- 종목 명단(유니버스)·주기 실행(스케줄러)·저장·백테스트 → **팀장(파이프라인/인프라)**

→ 인터페이스는 **`DisclosureSignal` / `NewsSignal` 스키마 객체를 반환**하는 것.
   (R.md 기준 `agents/collector.py`가 입력을 받아 이 스키마를 리턴 → 뒷단 파이프라인이 받아 씀)

## 4. Strategist에 넘기는 형식 (이은미와 협의)

우리 공시·뉴스 신호를 **종목당 1개 `AnalysisBundle`** 로 압축해서 넘긴다 (날것 신호 다발 X):
- **LLM용** = 읽기 쉬운 압축 텍스트(`to_prompt()`), **코드용** = 타입 객체 (한 객체 두 얼굴).
- 하드리스크(상폐·going concern)는 **별도 플래그(override)** 로 박아 "추론으로 무시" 못 하게.
- 실행: `python run_bundle.py --ticker NVDA --form 8-K --news-limit 8 --llm`
- **형식·근거·집계규칙**: `인터페이스명세_정보분석→Strategist.md` · **실행법**: `run_bundle_사용법.md`

## 5. 기술 스택

- **Python + Pydantic AI** (구조화 출력·검증·재시도의 핵심 엔진)
- **LLM: OpenAI gpt-4o-mini** (분석가 티어. `core/llm.py`로 추상화돼 교체 쉬움)
- **데이터: 무료/무키** — SEC EDGAR(User-Agent만) · Google News RSS
- **출력: 정해진 신호 스키마 객체**(`DisclosureSignal`/`NewsSignal`) — 저장·DB는 파이프라인(팀장) 몫

## 6. 남은 일 (우리 파트 ①②)

1. **Strategist 인터페이스 확정** — 이은미와 집계 가중치·event_type 목록 등 최종 합의 (명세 §9).
2. **이벤트 온톨로지 정교화** — 현재 `other`로 분류되는 게 많음 → 실제 패턴 보며 조정.
3. **뉴스 본문 수집** — 지금은 헤드라인 위주(RSS) → 깊은 분석엔 본문 필요.
4. *(Phase 2)* **XBRL 어닝 서프라이즈** — 애널 컨센서스가 유료라 무료 대안 검토(보류).

---

## 7. 알려진 한계 (투명 공개)

- **시장반응·실제 매수권한 없음** → 의도된 것(다른 담당 몫).
- **뉴스가 헤드라인 위주**(RSS 한계) → 깊은 분석엔 본문 수집 필요.
- **event_type = other 빈발** → 온톨로지가 실제 패턴에 덜 맞음. 데이터 보며 조정 예정.
- **10-Q/K 위험문구 스캔은 원문 다운로드** → 새 보고서일 때만 받도록 하는 게 좋음(비용).
