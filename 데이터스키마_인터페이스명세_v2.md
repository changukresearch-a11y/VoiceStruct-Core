# 데이터 스키마 · 인터페이스 명세 v2 — 정보분석(공시·뉴스) → Strategist

> 작성: 정보분석 담당(정창욱) · 대상: Strategist 담당(이은미) 및 전 팀
> 갱신: 2026-07-02 팀 회의 피드백 반영 (**v1 폐기 → v2**)
> 구현: `app/common/analysis_bundle.py` (`DisclosureBundle` / `NewsBundle`)

---

## ⚡ TL;DR

1. **공시·뉴스는 완전 별개 2객체**로 넘어간다 — `DisclosureBundle` / `NewsBundle`. (v1의 "종목당 1개 합산 Bundle"은 폐기. 합치기는 Strategist가.)
2. **모든 점수는 0.0~1.0 소수점**으로 통일. 방향도 **`sentiment_score`(0~1: 0=악재·0.5=중립·1=호재)**로 점수화하고, `sentiment` 라벨을 파생 병기. `importance`는 방향 뺀 강도(별도 축).
3. **종목 식별 3필드** — `ticker` · `company_name`(공시가 SEC에서 자동) · `category`(스크리닝 에이전트가 전달).
4. **스크리닝(50개 선별)은 스크리닝 에이전트 몫** → 우리는 종목 리스트를 입력으로 받는다.

## 2. 공통 규약

### 2-1. 종목 식별 (두 Bundle 공통)

| 필드 | 타입 | 채우는 주체 | 설명 |
|---|---|---|---|
| `ticker` | `str` | 정보분석 | 종목 티커 (대문자). 예: `NVDA` |
| `company_name` | `str \| None` | **공시 collector (SEC 자동)** | 예: `NVIDIA CORP`. 뉴스만 있을 땐 스크리닝/공시에서 전달받아 채움 |
| `category` | `str \| None` | **스크리닝 에이전트** | 섹터/산업. 예: `반도체`. 우리는 필드만 두고 받아 표시 |
| `as_of` | `str` | 정보분석 | 분석 기준일 `YYYY-MM-DD` |

### 2-2. 점수 규약 (전부 `float` 0.0~1.0)

| 개념 | 규칙 |
|---|---|
| 정규화 | 내부 0~10 척도를 `/10` 하여 0~1로 (`_n10()`), 소수 2자리 반올림 |
| `sentiment_score` | **방향×강도를 0~1로** (`0.5 + 0.5 × signed`). 0=강악재·0.5=중립·1=강호재. **고도화(비대칭 신뢰 감쇠): 호재(+)는 `confidence`로 중립쪽 감쇠**(근거 약한 낙관 과신 방지), **악재(−)는 감쇠 안 함**(안전 우선·경고 유지) |
| `importance` | 강도·중요도만(부호 없음). 거래정지·상폐 계열이 1.0에 가까움 |
| `risk_score` | 하방·위험 강도 |
| `confidence` | 신뢰도. 공시=`certainty_level` 수치화 / 뉴스=`certainty × is_confirmed × source_trust` |
| `source_trust` | (뉴스 전용) 출처 신뢰도 (LLM 판단, 본문 톤+등급) |
| `grade_score` | (뉴스 전용) 출처 **정책등급** 점수. `ALLOW=1.0 · GRAY=0.6 · WATCH=0.3 · BLOCK=0.0` (신뢰가중 평균) |
| `confirmed_score` | 확정도. 공시=`1.0`(법적 의무 제출) / 뉴스=`확정건수 / 전체` |

> **불리언 → 0/1 정수:** `has_signal` · `hard_block`은 JSON에서 `1`/`0` 정수로 노출(의미 유지, 타입만 통일). `hard_block=1`은 여전히 매수 override.

### 2-3. 방향 — `sentiment_score` + `sentiment`

- **`sentiment_score`** `float` 0~1 — 방향을 점수화. `0.0`=강한 악재, `0.5`=중립, `1.0`=강한 호재. (방향×강도를 0~1 축에 인코딩)
- **`sentiment`** 라벨 — `positive` · `negative` · `neutral` · `mixed`. `sentiment_score`에서 파생 병기(가독성용). 호재+악재 혼재는 `mixed`.

> 세 필드 역할 분리: `sentiment_score`=방향(+강도 반영) · `importance`=순수 강도 · `sentiment`=방향 라벨.

### 2-4. `event_type` 통일 온톨로지 (11종, 공시·뉴스 공유)

`app/common/ontology.py` 단일 정의. 과세분화 금지(표본 부족).

```
earnings · guidance_change · ma · capital_raise · management_change ·
insider_trade · product_deal · analyst_rating · regulation_legal ·
delisting_halt · other
```

## 3. `DisclosureBundle` — 공시 결과 (단독)

```python
class DisclosureBundle(BaseModel):
    # 종목 식별
    ticker: str
    company_name: str | None = None
    category: str | None = None
    as_of: str
    created_at: str = ""            # 레코드 생성 시각(초 포함, DB 생성일 성격)
    has_signal: int = 0             # 0/1 (분석된 공시 신호 유무)
    # 공시 원문 식별 (전달용)
    filing_title: str = ""          # 공시 제목
    filing_no: str = ""             # 공시 번호(accession)
    filed_at: str = ""              # 업로드 일시(초 포함, acceptanceDateTime 우선)
    # 공시 신호 (점수 0~1)
    event_type: str = "other"
    sentiment: str = "neutral"      # 방향 라벨 (sentiment_score에서 파생)
    sentiment_score: float = 0.5    # 0~1: 0=강악재·0.5=중립·1=강호재
    importance: float = 0.0         # 0~1 (강도, 방향 뺀 별도 축)
    risk_score: float = 0.0         # 0~1
    confidence: float = 0.0         # 0~1
    confirmed_score: float = 0.0    # 0~1 (공시=1.0 도장 찍힌 사실)
    fact_check: str = ""            # 팩트체크 한 문장(코드 판정)
    # 안전장치
    hard_block: int = 0             # 0/1 (매수 override)
    hard_block_reason: str | None = None
    # 근거 · 전문 요약
    reason: str = ""                # 한 줄 근거
    verdict: str = ""               # ★ 검증 결과 한 문장(맨 앞 호재/악재/중립)
    summary: str = ""               # ★ 공시 전문 3~4줄 요약
    keywords: list[str] = []        # ★ 핵심 키워드 5개
    ref: str = ""                   # accession_no 또는 url
```

| 필드 | 타입/범위 | 의미 · 근거 |
|---|---|---|
| `has_signal` | bool | LLM 분석 신호가 실제로 있는지. 없으면 아래 신호 필드는 기본값 |
| **`filing_title`·`filing_no`·`filed_at`** | str | 공시 제목·번호(accession)·업로드 일시(**초 포함**, acceptanceDateTime 우선). 신호 유무와 무관하게 전달 |
| `event_type` | 11종 | 공시 이벤트 유형 |
| `sentiment_score` | 0~1 | 방향 점수. 0=강악재·0.5=중립·1=강호재 |
| `sentiment` | 라벨 | 방향 라벨(점수에서 파생 병기) |
| `importance` | 0~1 | 강도만. 1~5일 스윙 영향. 거래정지·상폐≈1.0, 대형 M&A·실적쇼크 0.8~0.9 |
| `risk_score` | 0~1 | 하방 위험. 회계·희석·상폐일수록 ↑ |
| `confidence` | 0~1 | `certainty_level`(High=0.9/Med=0.6/Low=0.3) |
| `confirmed_score` | 0~1 | 확정도. 공시는 법적 의무 제출 → **1.0** ("도장 찍힌 사실") |
| **`fact_check`** | str | 팩트체크 한 문장(코드 판정). 공시=`Verified — official SEC filing` |
| **`created_at`** | str | 레코드 생성 시각(**초 포함** ISO, DB 생성일 성격) |
| `hard_block` | 0/1 | 상폐·거래정지·going concern 등 → 매수 override. **sentiment에 녹이지 않고 별도 플래그** |
| `hard_block_reason` | str/None | 차단 사유(결정저널·리뷰용) |
| `reason` | str | 한 줄 근거(수치 포함 권장) |
| **`verdict`** | str | 검증 결과 한 문장. 맨 앞 **호재/악재/중립** 명시 + 근거 요지 |
| **`summary`** | str | 공시 전문 **3~4줄 요약**(원문 근거만, 환각 방지) |
| **`keywords`** | list | 핵심 **키워드 5개**(검색/태깅용) |
| `ref` | str | 감사추적·백테스트 replay용 accession/링크 |

```python
class NewsBundle(BaseModel):

    company_name: str | None = None

    published_at: str = ""          # 업로드 일시(초 포함)

    news_rumor: int = 0             # 루머/추측 건수
    # 대표 신호 (점수 0~1)
    event_type: str = "other"
    sentiment: str = "neutral"      # 방향 라벨 (sentiment_score에서 파생)
    sentiment_score: float = 0.5    # 0~1: 0=강악재·0.5=중립·1=강호재
    importance: float = 0.0         # 0~1 (강도, 신뢰가중)
    peak_importance: float = 0.0    # 0~1 (집계 전 최고 잠재)
    risk_score: float = 0.0         # 0~1
    confidence: float = 0.0         # 0~1
    source_trust: float = 0.0       # 0~1 (LLM 판단)
    grade_score: float = 0.0        # 0~1 (출처 정책등급: ALLOW1·GRAY.6·BLOCK0)
    confirmed_score: float = 0.0    # 0~1 (확정 비율 = 확정건수/전체)
    fact_check: str = ""            # 팩트체크 한 문장(Verified/Unverified…)
    # 안전장치
    hard_block: int = 0             # 0/1
    hard_block_reason: str | None = None
    # 근거 · 전문 요약 (대표 기사 기준)
    top_evidence: list[str] = []
    reason: str = ""
    verdict: str = ""               # ★ 검증 결과 한 문장(호재/악재 판단)
    summary: str = ""               # ★ 뉴스 전문 3~4줄 요약
    keywords: list[str] = []        # ★ 핵심 키워드 5개
    ref: str = ""
```

| 필드 | 타입/범위 | 의미 · 근거 |
|---|---|---|
| `news_count` / `news_confirmed` / `news_rumor` | int | 분석된 뉴스 건수와 확정·루머 분해. "확정 여럿=신뢰↑, 한 곳뿐=루머 의심" |
| **`news_title`·`source`·`published_at`** | str | **대표 기사**의 제목·출처 언론사·업로드 일시(**초 포함**, RSS pubDate) |
| `importance` | 0~1 | 신뢰가중 평균 강도 |
| `peak_importance` | 0~1 | 집계 전 최고 잠재 중요도. "잠재 큰데 저신뢰라 눌린" 경우 병기(`[최고imp..·저신뢰]`) |
| `source_trust` | 0~1 | (뉴스 전용) 출처 신뢰도(LLM 판단). 소셜·루머 방어 |
| `grade_score` | 0~1 | (뉴스 전용) 출처 **정책등급** 점수. ALLOW1·GRAY.6·BLOCK0 (신뢰가중, 소셜/미등록=GRAY) |
| `confirmed_score` | 0~1 | 확정 비율 = 확정건수 / 전체 |
| **`fact_check`** | str | 팩트체크 한 문장(코드 판정). 확정→`Verified …`, 루머→`Unverified (rumor) …` |
| **`created_at`** | str | 레코드 생성 시각(**초 포함** ISO, DB 생성일 성격) |
| `top_evidence` | list[str] | 대표 근거 헤드라인(나머지는 DB) |
| **`verdict`·`summary`·`keywords`** | str·str·list | **대표 기사**(신뢰가중 선정)의 호재/악재 판정·전문 3~4줄 요약·키워드 5개 |

## 5. 집계 규칙 — 뉴스 여러 건 → 대표 1개

| 항목 | 규칙 |
|---|---|
| 방향/강도 | 확정·`source_trust`·`certainty` **신뢰가중**. 단 강한 악재(신뢰가중 signed ≤ −0.7)면 그쪽으로 보수적 |
| `sentiment_score` | 위 방향×강도를 `_senti_score`로 0~1 매핑. **호재만 `confidence`로 중립쪽 감쇠, 악재는 유지**(비대칭) |
| `importance` | 신뢰가중 평균(방향 뺀 크기). **감쇠 없음** — 방향/강도 축 분리 |
| `confidence` | `certainty × is_confirmed × source_trust`의 평균 |
| **대표 선정** | `event_type`·`reason`·`top_evidence`는 **`importance × confidence`(신뢰가중) 최대** 기사로. 시끄러운 저신뢰 루머가 대표가 안 되게(확정 위주) |
| `hard_block` | 어느 뉴스든 BLOCK류면 true (안전장치는 하나만 발동해도) |

> **산출 고도화 근거:** ① 방향점수는 **낙관만 신뢰로 깎고 악재는 유지**해 과신·펌프엔 보수적이되 경고는 흐리지 않는다("틀려도 작게 잃기"). ② 대표는 **가장 시끄러운 것(importance)이 아니라 가장 믿을 만하며 중요한 것(importance×confidence)**을 뽑아, 미확정 루머가 헤드라인이 되는 걸 막는다.

## 6. `to_prompt()` 렌더 샘플 (Strategist가 보는 것)

> ⚙️ **출력 언어 = 영어.** 자유서술(`summary`·`verdict`·`reason`·`evidence_quotes`)과 `to_prompt()` 라벨은 전부 **영어**다(전략가 에이전트가 영어로 읽음, 사람 검수용 아님). enum·점수 필드는 공통. `category`는 스크리닝이 준 값 그대로.

```
       Fact-check: Verified — official SEC filing (legally binding fact).

       Reason: "Large acquisition announced (30% premium)"

```
```
[NVDA · NVIDIA CORP · Semiconductors] News as_of 2026-07-02

```

## 7. 전달 방식

- **정식 전달 계약 = JSON.** 각 Bundle은 Pydantic 모델 → `model_dump_json()`으로 직렬화해 Strategist에 **JSON으로 전달**한다. 모든 값이 숫자(0~1 float · 0/1 int · 카운트) 또는 짧은 라벨이라 파싱·검증이 안정적.
- **보조 = `to_prompt()` 텍스트** — 사람이 읽는 요약·LLM 프롬프트 삽입용(선택). 같은 객체의 두 표현.
- **부 경로 = DB 사본**(`disclosure_signals` / `news_signals`) — audit·백테스트·결정저널·replay용.

엔트리: `python run_bundle.py --ticker NVDA --category "반도체" --llm` → 공시·뉴스 번들 2개 출력.

## 8. 이은미와 합의할 열린 질문

1. **결합 위치** — 공시·뉴스를 Strategist가 어떻게 합칠지(가중치). 우리는 합산 안 함(v2).
2. **집계 가중치 계수** — 신뢰·최신성 초기값은 임의 → 백테스트로 튜닝(자동조정 아님·수동).
3. **`category` 값 체계** — 스크리닝 에이전트가 넘길 카테고리 라벨 표준(GICS 섹터? 자체 분류?) 합의 필요.
